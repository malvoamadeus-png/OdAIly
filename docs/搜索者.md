# 搜索者

## 职责

搜索者负责判断新任务是否与 Odaily 已发布快讯或当前流水线中的候选事件重复。

搜索者只判断“是否已经报道过同一事件”，不判断真假，不补全背景，不改写快讯。

## 进入搜索者的任务

- X 来源：先经过判断者。只要判断者没有输出 `route=discard`，任务就进入搜索者。
- 竞品快讯：先进入搜索者。明显重复时直接跳过，未重复时再进入判断者。

Odaily 自身快讯不作为待处理任务进入搜索者，只作为本方已发布语料参与查重。

搜索 worker claim 到任务后会先执行统一时效兜底。`tasks.published_at` 缺失或超过 `PROCESSING_FRESHNESS_WINDOW_SECONDS` 窗口时，任务进入 `expired`，不做 embedding、不调用 AI 复判。

## 查重对象

搜索者需要检查两类对象：

- Odaily 已发布快讯：初始窗口为过去 1 天，窗口必须可配置。
- in-flight 候选事件：已经进入流水线但尚未正式发布的任务。

in-flight 候选事件包括：

- 已通过判断者、等待搜索者的 X 任务。
- 已通过搜索者、等待判断者的竞品任务。
- 已通过搜索者、等待编写的任务。
- 正在编写或已推送后台但尚未正式发布的任务。

## 向量相似度策略

搜索者分两层：

- 第一层：向量相似度快速判断。
- 第二层：搜索者-AI 对中间相似度候选做快速同事件判定。

建议初始阈值：

- `X1 = 0.88`：高相似阈值。大于等于该值时，直接认为重复。
- `X2 = 0.78`：AI 复判阈值。大于等于该值且小于 `X1` 时，交给搜索者-AI 判断。
- 小于 `X2`：默认认为不重复。

这些阈值只是初始建议，不应视为最终业务规则。上线前应使用真实 Odaily/竞品/X 样本回测：

- 正样本：同一事件的不同媒体写法。
- 负样本：同一主体但不同动作、不同金额、不同日期、不同进展的事件。
- 目标：优先避免重复发布，同时不能过度拦截同一主体的新进展。

## 相似度比较文本

向量化文本应尽量稳定，不直接使用全部原始 HTML 或噪声字段。

建议格式：

```text
标题：{title}
正文：{clean_content}
```

其中 `clean_content` 应先删除媒体固定前缀，例如：

- `BlockBeats 消息，5 月 9 日，`
- `PANews 5月9日消息，`
- `Odaily 星球日报讯 `

发布时间、来源、来源链接、点赞数、分享文案、页面噪声不进入 embedding 文本。发布时间、来源和链接只作为 metadata 记录，供日志、追溯和必要时的 AI 复判参考。

## 搜索者-AI

模型：

- 初始使用 `gpt-5.4-mini`。
- 用途是快速二分类，不负责写作。

触发条件：

- 候选信息与某条 Odaily 快讯的相似度满足 `X2 <= similarity < X1`。
- 或候选信息与某个 in-flight 候选事件相似度满足该区间。

AI 输入：

- 新候选信息：标题、清洗后正文。
- 最相似的 Odaily 快讯或 in-flight 候选事件：标题、正文、相似度、内部 ID。

默认不把大量历史快讯全部交给 AI。向量搜索先缩小候选集，AI 只看少量最相似结果。

AI 输出：

```json
{
  "is_duplicate": true,
  "duplicate_target_type": "odaily_published | inflight_candidate | none",
  "duplicate_target_id": "string",
  "reason": "same_event | same_topic_different_event | update_of_existing_event | unrelated"
}
```

业务使用：

- `is_duplicate=true`：任务标记为重复，记录命中的目标和原因。
- `is_duplicate=false`：进入后续流程。

`reason` 用于日志和人工排查，不参与复杂业务分支。

## AI 判定标准

判定为同一事件：

- 同一主体。
- 同一核心动作。
- 同一关键结果。
- 时间窗口一致或明显指向同一事件。
- 不同媒体只是标题、措辞、信息顺序不同。

不判定为同一事件：

- 同一主体但不同动作。
- 同一项目的融资、上线、被盗、澄清、价格变化等不同事件。
- 同一事件的新进展，且包含实质性新增事实。
- 只是同一赛道、同一币种、同一人物，但新闻事实不同。

对于“同一事件的新进展”，初始阶段建议不自动丢弃，交给后续编写者或人工审核判断是否值得发布。

## 双重发布规避

X 信源和竞品快讯可能在短时间内进入队列，并且指向同一事件。热门事件还可能出现多个竞品和多个 X 来源同时进入。

处理规则：

- 如果新任务与已发布 Odaily 快讯重复，直接标记重复。
- 如果新任务与 in-flight 候选事件重复，默认不再进入后续编写，而是挂到已有候选事件上作为补充来源。
- 已有候选事件保留一个主任务，其他重复来源记录为 supporting sources。
- 主任务优先级建议：X 权威原始来源高于竞品转述；已进入更后阶段的任务高于刚进入队列的任务。

这样可以避免同一事件被同时写成多条快讯，同时保留多个来源链接供后续审核参考。

## Embedding 存储策略

第一阶段不建议把所有 embedding 都保存到 Supabase。

原因：

- 搜索者需要高频读取近 1 天 Odaily 快讯，本地缓存更快。
- 当前查重主窗口很短，本地轻量存储足够承载。
- Supabase 主要保存任务状态、原始信息和处理结果，避免把频繁更新的向量缓存也放进去增加复杂度。

建议：

- Odaily 近 1 天快讯 embedding 保存在本地。
- Odaily 文本语料同时保存到 Supabase `odaily_reference_items` 和本地 SQLite，Supabase 用于共享状态和追溯，本地 SQLite 用于高频读取和 embedding 缓存。
- in-flight 候选事件 embedding 保存在本地，并在任务状态结束后过期。
- 竞品快讯和 X 任务的 embedding 可不长期保存；进入任务后只保存用于当前查重的短期缓存。

本地存储优先选择 SQLite，路径为 `data/processed/searcher/searcher.sqlite`，用于保存文本、embedding、发布时间、来源、过期时间和命中记录。

后续如果需要多服务器部署，再考虑：

- Supabase/Postgres + pgvector。
- 独立向量库。
- 每个 worker 本地缓存 + 定期同步 Odaily 快讯文本。

## Embedding 生成延迟

新信息进入时需要快速 embedding。应避免每次来新任务才冷启动模型或加载大量历史。

建议策略：

- 使用常驻搜索者 worker，而不是一次性脚本。
- worker 启动时预加载近 1 天 Odaily 快讯文本和 embedding。
- worker 定时或收到通知后增量更新 Odaily 快讯 embedding。
- 新任务进入时，只对新任务生成一次 embedding，然后和本地缓存做向量检索。
- embedding API 调用应支持批量生成，启动预热和历史补齐时批量处理。
- 对同一 `source + source_item_id` 的文本计算内容 hash，hash 未变时复用 embedding。

如果使用本地开源 embedding 模型：

- 模型加载可能耗时，应由常驻 worker 在启动时加载一次。
- 不应在每条新信息到来时启动一个新进程加载模型。

如果使用阿里云百炼 DashScope embedding API：

- 不存在本地模型冷启动问题，但有网络延迟。
- 仍应使用本地缓存减少重复请求。
- 默认使用 OpenAI 兼容模式 endpoint `https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings`，环境变量 `DASHSCOPE_API_KEY` 提供 key，默认模型 `text-embedding-v4`。

## 需要记录的结果

搜索者每次处理应记录：

- 当前任务 ID。
- 使用的 embedding 模型。
- 查重窗口。
- 最高相似度。
- 命中的 Odaily 快讯或 in-flight 候选事件。
- 是否调用搜索者-AI。
- 搜索者-AI 的结构化输出。
- 最终结论：重复、不重复、搜索失败。

这些信息用于后续调阈值、排查误杀和回放问题案例。

## 待讨论项

- `X1`、`X2` 的最终阈值需要用真实样本回测。
- supporting sources 是否要在后台审核界面展示，第一版先仅保存在内部任务记录。
