# 巨鲸

## 职责

巨鲸模块是一条独立监控链路，不进入 `tasks` 写作发布流程。当前包含两块：

- `链上`：监控配置地址在启用 EVM 链上的转账和疑似 Swap。
- `Hyperliquid`：监控配置地址在 Hyperliquid 上的开仓、平仓成交。

两块都由控制台维护地址和标签，后端 worker 负责轮询、识别、去重和落库。正常巨鲸业务消息由信息流插件展示，不再发送 Telegram。

## 前端输入

控制台新增一级标签“巨鲸”，页面内包含两个分区：

- `链上`：输入 `address + label`，支持启用、停用、删除，查看各链最近状态和播报历史。
- `Hyperliquid`：输入 `address + label`，支持启用、停用、删除，查看最近轮询状态和播报历史。

前端只维护地址级配置，不提供链选择，也不手工选择 Hyperliquid 合约或币种。

为便于排查异常批量写入，地址配置表会额外记录 `created_by`、`updated_by`，从 Supabase 登录态中提取操作者邮箱。

## 数据表

链上监控使用：

- `whale_watch_addresses`
- `whale_watch_chain_states`
- `whale_watch_activities`

`whale_watch_addresses` 额外持久化：
- `created_by`
- `updated_by`

Hyperliquid 监控使用：

- `whale_watch_hyperliquid_addresses`
- `whale_watch_hyperliquid_states`
- `whale_watch_hyperliquid_activities`

`whale_watch_hyperliquid_addresses` 额外持久化：
- `created_by`
- `updated_by`

`whale_watch_hyperliquid_states` 还会持久化聚合提醒状态：
- `aggregate_window_entries`
- `aggregate_alert_active`

`whale_watch_hyperliquid_activities` 还会额外记录：
- `alert_kind`
- `aggregate_fill_count`

这些表都走 Supabase Auth + RLS。只有 `authenticated` 且存在于 `console_admins` 白名单中的用户可以读写地址配置，并读取状态与活动记录；`anon` 不授予权限。

## Worker 与命令

初始化数据库：

```powershell
python backend\src\main.py whale-watch-init-db
```

如果生产里的 Hyperliquid 相关表还是旧结构，重新执行一次 `whale-watch-init-db` 会自动补齐聚合提醒需要的状态列和活动列，并同步更新活动表约束以允许 `Aggregate` 方向。

链上 worker：

```powershell
python backend\src\main.py whale-watch-worker
python backend\src\main.py whale-watch-worker --once
```

链上地址排查与清理：

```powershell
python backend\src\main.py whale-watch-list-addresses --include-disabled
python backend\src\main.py whale-watch-list-addresses --created-since-hours 24
python backend\src\main.py whale-watch-delete-addresses --created-since-hours 24
python backend\src\main.py whale-watch-delete-addresses --ids 101 102 103 --execute
```

`whale-watch-delete-addresses` 默认是 dry-run，只有传 `--execute` 才会真的删除。

Hyperliquid worker：

```powershell
python backend\src\main.py whale-watch-hyperliquid-worker
python backend\src\main.py whale-watch-hyperliquid-worker --once
```

生产环境分别由 `deploy/odaily-whale-watch.service` 与 `deploy/odaily-whale-watch-hyperliquid.service` 托管，服务统一读取 `/opt/OdAIly/.env`。

## 链上监控来源与识别

链上监控按 `WHALE_WATCH_CHAIN_KEYS` 轮询启用链，第一版默认启用：

- `ethereum` -> `https://eth.blockscout.com`
- `base` -> `https://base.blockscout.com`

每轮按 `address + chain` 调用 Blockscout v2：

- `/api/v2/addresses/{address}/transactions`
- `/api/v2/addresses/{address}/token-transfers`
- `/api/v2/transactions/{tx_hash}`

首次看到某个 `address + chain` 组合且没有 `seeded_at` 时，只 seed 当前第一页视图，不播报历史交易。

为避免未知 bug 一次性灌入大量链上地址，`whale_watch_addresses` 在数据库侧额外限制总量上限为 `500` 条；超过上限时，新的链上地址插入会直接失败并提示先排查异常批量写入。已有地址的更新和启停不受影响。

活动识别规则：

- 只有单向净流入或净流出时，识别为普通转账。
- 同一笔交易内同时存在资产流出与另一资产流入，且命中常见 DEX / router / pool 信号时，识别为疑似 Swap。
- 识别结果按 `address_id + chain_key + tx_hash + activity_fingerprint` 去重。

第一版 Swap 识别是规则判定，不做完整 ABI 深度解码。

## Hyperliquid 监控来源与识别

Hyperliquid worker 直接调用官方 `https://api.hyperliquid.xyz/info`，当前使用：

- `userFills`

每轮开始时，worker 只读取一次 `whale_watch_hyperliquid_settings` 作为本轮运行时门槛；如果配置表还没有 `global` 行，会自动写入默认值后继续运行。

识别范围：

- `Open Long`
- `Open Short`
- `Close Long`
- `Close Short`

首次看到地址且没有 state 时，只记录当前 `last_seen_time`，不播报历史成交。

识别结果保存：

- 方向、币种、价格、数量、名义价值、已实现盈亏
- `fill_key`
- `fill_time` / `fill_time_ms`
- `telegram_text`
- `tx_url`

过滤规则：

- 单笔门槛：`notional_usd >= single_fill_min_notional_usd` 时立即播报。
- 聚合门槛：单笔未达到门槛的成交会进入一个向前滚动的时间窗口。
- 聚合窗口：每轮都会丢弃 `now - aggregate_window_seconds` 之前的旧成交。
- 当窗口内累计名义价值首次达到 `aggregate_min_notional_usd` 时，发送一条聚合播报。
- 只要窗口累计金额一直保持在聚合门槛之上，就不会因为后续每一笔小单继续重复播报；只有当窗口掉回门槛以下后，下一次重新跨线才会再次播报。

Hyperliquid 活动按 `address_id + fill_key` 去重。

Hyperliquid 提醒里的地址链接统一使用：

```text
https://hyperbot.network/trader/{address}
```

## 通知边界

两条链路不再发送正常巨鲸 Telegram 播报；活动仍会落库并写回 `telegram_result.skipped=true`，由信息流插件展示。

链上 topic 回落顺序：

```text
WHALE_TELEGRAM_MESSAGE_THREAD_ID
TELEGRAM_MESSAGE_THREAD_ID
```

Hyperliquid topic 回落顺序：

```text
WHALE_HYPERLIQUID_TELEGRAM_MESSAGE_THREAD_ID
WHALE_TELEGRAM_MESSAGE_THREAD_ID
TELEGRAM_MESSAGE_THREAD_ID
```

上述 topic 环境变量仅作历史兼容。若 worker 自身失败、心跳丢失或活动处理异常，由监督者发送 Telegram 错误告警。

## 配置

链上：

```text
WHALE_WATCH_INTERVAL_SECONDS=60
WHALE_WATCH_CHAIN_KEYS=ethereum,base
WHALE_WATCH_REQUEST_TIMEOUT_SECONDS=20
WHALE_WATCH_MAX_ATTEMPTS=3
WHALE_WATCH_BACKOFF_SECONDS=1
WHALE_TELEGRAM_MESSAGE_THREAD_ID=
```

Hyperliquid：

```text
WHALE_HYPERLIQUID_TELEGRAM_MESSAGE_THREAD_ID=
WHALE_HYPERLIQUID_INTERVAL_SECONDS=60
WHALE_HYPERLIQUID_SINGLE_FILL_MIN_NOTIONAL_USD=500000
WHALE_HYPERLIQUID_AGGREGATE_MIN_NOTIONAL_USD=1000000
WHALE_HYPERLIQUID_AGGREGATE_WINDOW_SECONDS=600
WHALE_HYPERLIQUID_REQUEST_TIMEOUT_SECONDS=20
WHALE_HYPERLIQUID_MAX_ATTEMPTS=3
WHALE_HYPERLIQUID_BACKOFF_SECONDS=1
```

PostgreSQL 连接阶段还会复用全局超时配置：

```text
POSTGRES_CONNECT_TIMEOUT_SECONDS=10
POSTGRES_IDLE_IN_TRANSACTION_SESSION_TIMEOUT_MS=60000
```

Hyperliquid worker 的仓储连接会额外固定为 `autocommit`，并把 `application_name` 标记为 `odaily-whale-watch-hyperliquid`。这样每条 SQL 都以独立短事务执行，避免在 Supabase / Supavisor 前留下长期 `idle in transaction` 会话，把整轮轮询卡死在数据库 socket 上。

两条巨鲸常驻循环会捕获单轮外围异常并继续下一轮，避免一次地址列表读取、链上 API 或 Hyperliquid API 波动导致 worker 进程退出。

公共 Telegram 环境变量：

```text
TELEGRAM_BOT_TOKEN=
TELEGRAM_CHAT_ID=
TELEGRAM_MESSAGE_THREAD_ID=
```
