# 监督者

## 职责

监督者负责检查 worker 心跳、队列积压、处理中卡住任务和失败窗口，并发送 Telegram 告警。

## 当前必需心跳

- `x_capture`
- `non_mainstream_media`
- `competitor_monitor`
- `jin10_monitor`
- `local_pipeline`
- `whale_watch`
- `whale_watch_hyperliquid`

旧的 `x_process_*` 分阶段 worker、`external_media_alert_*` 分阶段 worker 和 `external_media_alert_fetcher` 都不再属于生产必需心跳。

## 任务监控范围

监督者仍覆盖：

- `x`
- `blockbeats`
- `panews`
- `jinse`
- `jin10`
- `non_mainstream_media`
- `ai_source`
- `mainstream_media`
- `external_media_alert`
- `ai_source_alert`

其中 `mainstream_media` 仅用于发现历史任务残留或堵塞。

本地流水线切换后，监督者不再把 `judged / searched / deduped / written / publisher_pending` 等阶段状态理解为“应该被某个远端 worker 领取的队列积压”。这些状态仍会作为观察状态保留在 `tasks` 中，真正的推进由 `local_pipeline` 本地 SQLite 队列负责。

当前仍需要重点告警：

- 旧入口或收集器遗留的长期 `pending`
- 本地流水线心跳丢失
- 近期失败状态：`judge_failed / domain_failed / search_failed / write_failed / format_failed / publish_failed / publisher_failed / notify_failed`

## X 抓取健康判定

监督者判定 X 抓取是否健康时，优先检查 `pipeline_worker_heartbeats` 中 `x_capture` 的近期成功心跳。

只有近期没有成功心跳时，才回退检查 `x_capture_attempts` 是否存在近期 `success` 记录。该 fallback 使用存在性检查，不做 `count(*)` 全量计数；如果 attempts 查询本身超时或失败，监督者按 X 抓取不健康继续告警，但不让自身进程崩溃。

## 初始化边界

`pipeline-supervisor` 是常驻监控命令，启动时不自动执行 Supabase schema 初始化。监控表由相关 `*-init-db` 命令初始化；生产排障时不要通过反复重启监督者触发 DDL。
