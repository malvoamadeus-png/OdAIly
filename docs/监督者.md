# 监督者

## 职责

监督者是只报警的健康检查 worker，用于发现流水线停摆、积压、卡住和持续失败。

监督者不自动重启 systemd 服务，不修改任务状态，也不清理队列。

## 检查范围

监督者检查：

- `x_capture` 是否有近期成功抓取。
- `competitor_monitor` 是否有近期成功心跳。
- `x_process_judge`、`x_process_search`、`x_process_write`、`x_process_format_publish` 是否持续运行并有成功心跳。
- 可处理任务是否长时间积压。
- 处理中任务是否卡住或锁过期。
- 最近窗口内失败任务是否异常增多。

心跳同时区分：

- `last_seen_at`：worker 进程还活着。
- `last_success_at`：worker 最近成功完成过有意义的工作。

服务 active 但持续失败时，也应按无近期成功心跳报警。

报警落库时，监督者会把 metadata 中的 `datetime` 等非 JSON 原生对象转为字符串，避免报警记录写入 `jsonb` 时导致监督者自身崩溃。

## 积压与卡住

可处理积压示例：

- `pending`
- `judged`
- `searched`
- `deduped`
- `written`

处理中卡住示例：

- `judging`
- `deduping`
- `writing`
- `formatting`

具体状态名以当前处理 worker 和 schema 为准。

## 报警规则

报警通过 Telegram 发送。默认同一 `alert_key` 在冷却窗口内不重复刷屏。

如果配置 `TELEGRAM_MESSAGE_THREAD_ID`，监督者报警会发送到对应 Telegram topic；留空时发送到群组默认话题。可用 `python backend/src/main.py telegram-test --message-thread-id <id>` 先验证 bot 是否能向指定 topic 发消息。

报警内容应包含：

- 组件名。
- 失败或积压类型。
- 对象表或任务 ID。
- 最近错误摘要。
- 检查窗口和阈值。

## 运行命令

```bash
python backend/src/main.py pipeline-supervisor --once
python backend/src/main.py pipeline-supervisor
```

建议 systemd 服务名：

```text
odaily-pipeline-supervisor.service
```

## 相关数据

监督者依赖：

- `pipeline_worker_heartbeats`
- `pipeline_alerts`
- `tasks`
- `x_capture_attempts`
- `x_task_pipeline`

具体表结构以当前 schema 和 `backend/packages/pipeline_supervisor` 实现为准。
