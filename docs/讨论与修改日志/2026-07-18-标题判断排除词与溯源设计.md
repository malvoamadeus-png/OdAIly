# 2026-07-18 标题判断、排除词与溯源设计讨论

## 大 KOL 知识：Prompt 还是 RAG

当前不建议优先上完整 RAG。

Crypto 里需要硬编码辅助判断的名人、机构和账号数量并不大，且判断需求通常不是“检索长文本知识”，而是“这个主体是否足够知名，标题能不能作为发言主体”。这种需求更适合先做轻量知识表或短 Prompt 片段：

- 在配置或代码种子里维护少量 `known_subjects`，字段包括显示名、别名、主体级别、适用来源和标题策略。
- Prompt 只注入与当前任务命中的少量主体，而不是把完整名单全部塞进去。
- 对高频大 KOL 可先用短白名单覆盖，例如 Rune、Cobie、Vitalik、CZ 等。

这样比 RAG 简单、稳定、可复盘，也不会把 Prompt 拉得太长。真正适合 RAG 的场景是主体数量持续扩大、每个主体有较长背景、需要根据不同上下文检索不同证据时。现在这个问题更像“可控词典 + 少量标题策略”。

## 排除词设计

现有竞品排除词是标题 + 正文子串匹配，管理上偏扁平，后续应改成“规则组 + 适用路径”的设计。

建议数据模型按组管理，而不是一个词一个词选择适用范围：

- 规则组：例如 `Ripple商务稿`、`TRON生态`、`非Crypto AI产品`、`营销活动`。
- 适用路径：每组选择适用于 `Crypto信源全文`、`竞品`、`X`、`AI信源`、`外媒标题提醒` 等路径。
- 匹配词：每组维护多行词或短语，例如 `Ripple`、`XRP`、`RLUSD`。
- 匹配模式：默认大小写不敏感子串；后续可增加“全词匹配 / 正则 / 中文短语”。
- 动作：默认 `drop_before_pipeline`，即入库前或判断前截断；必要时支持 `manual_review_only`。
- 说明和例外：记录为什么排除、是否有例外场景，避免未来忘记上下文。

前端交互建议以“规则组卡片”为单位管理：

- 列表页展示规则组名、启用状态、适用路径标签、词数、最近更新时间。
- 详情里批量编辑词表，多选适用路径，不对每个词单独勾选路径。
- 提供按路径筛选、按词搜索和命中预览。
- 增加“测试文本”区域，输入标题/正文后显示会命中哪些规则组。

## 竞品判断者专用 Prompt

已确定：竞品来源不再继续沿用 X 判断者 Prompt，而是改为专用竞品判断者 Prompt。每条竞品仍只经过一次判断者，但按来源选择更合适的判断词。

目标边界：

- X 判断者：继续只丢弃情绪、喊单、闲聊。
- 竞品判断者：允许按语言规则直接丢弃非 Crypto AI、传统财经无关内容、营销活动、例行公司新闻等内容。
- Crypto信源全文判断者：继续按 Crypto信源全文逻辑判断。
- AI信源判断者：继续按 AI/半导体重要性规则判断。

竞品判断者输出建议：

```json
{
  "route": "regular|onchain|funding|discard",
  "discard_type": "none|pure_emotion|baseless_trading_call|daily_chatter|non_crypto_ai|non_crypto_tradfi|marketing_activity|routine_company_news"
}
```

这样阿里巴巴秒悟这类“非 Crypto AI 产品发布”可以在竞品判断者中直接丢弃，不需要依赖硬编码关键词，也不会等到发布者才挂后台。

## 溯源与结构化输出

希望溯源更丰富是合理的，但不建议让最终稿输出承担这些额外信息。最终稿仍应保持“一行标题 + 正文”，避免调试字段污染发布内容。

已确定：结构化溯源只保留标题策略相关字段，不保留事实抽取字段和风险字段。

写作模型可以一次输出结构化 JSON，但最终稿仍只取 JSON 里的 `title` 和 `content`，避免调试字段污染发布文本。

建议输出：

```json
{
  "title": "最终候选标题",
  "content": "最终候选正文",
  "title_strategy": "speaker_anchor|entity_front|action_first|result_front|amount_front|time_window_front|plain",
  "title_strategy_reason": "为什么这样组织标题",
  "matched_known_subjects": [
    {
      "name": "Rune",
      "type": "kol",
      "importance": "high",
      "matched_alias": "Rune"
    }
  ],
  "matched_title_rules": ["known_speaker_anchor"],
  "feature_mode_applied": true,
  "feature_mode_reason": "核心新闻价值来自 Rune 对 Base 社区信任的评价，需保留发言主体"
}
```

字段含义：

- `title_strategy`：标题组织策略。`plain` 表示普通事实标题，避免使用 `standard` 与“标准模式”混淆。
- `title_strategy_reason`：解释为什么选择该标题策略。
- `matched_known_subjects`：说明是否命中高重要主体或大 KOL 知识，用于解释为什么允许 `Rune：...` 这类标题。
- `matched_title_rules`：说明命中的标题策略规则。
- `feature_mode_applied`：模型是否实际采用特色标题策略，不等同于 Prompt 版本上的特色模式开关。
- `feature_mode_reason`：解释实际采用特色写法的原因。

不保留：

- `primary_subject`
- `primary_action`
- `primary_object`
- `key_amounts`
- `time_scope`
- `risk_notes`

原因：这些更偏事实抽取或风险提示，当前复盘需求主要关心“为什么标题这么写”，不关心完整事实结构化。
