# 发布者及上下游改进计划-第一阶段

## 背景

当前已经有初步的发布者方案，见 `docs/OdAIly代码与整体设计/控制台-发布者.md`。但现状问题不只在发布者本身，也在发布者上游的来源拆分、判断边界和链路分流上。当前比较突出的现状包括：

- 放行内容偏少，很多值得发的内容会被排除在外。
- 品类切分过粗，难以看清“具体少放了哪些内容”。
- 分类说明不够清晰，后续很难直接拿来改发布者 Prompt。
- 前台对这些潜在可发内容的可见性不足，不利于编辑挑选和复盘。
- AI信源链路与其他自动化写作链路当前共用一个发布者口径，边界不够清楚。
- 竞品来源里的 AI 相关内容目前仍会进入正常判断与写作链路，后续需要重新定义是否应在上游直接拦截。

## 第一阶段目标

先不直接改发布者 Prompt，也不立即改代码或前台展示逻辑，先把“我们现在到底有什么待发布内容”盘清楚，并把后续拟改的上游边界先留档。

本阶段只完成两件事：

1. 拉出过去两个月的上游内容样本。
2. 对这些样本做第一版品类切分，形成“我们有什么待发布”的基础资料。

## 本轮数据口径

本轮先采用当前可稳定提取的执行口径：

- 自动链路里最终进入 `ready_review` 的挂后台稿件。
- `odaily_reference_items` 中过去两个月的 Odaily 已发布镜像。

说明：

- 这里的“挂后台稿件”对应发布者阶段已经完成决策、但没有自动发布的样本。
- 这里的“Odaily 已发布镜像”用来反向观察：现在线上已经在发哪些类型，帮助后续判断哪些类型值得纳入发布者。
- 第一阶段资料保留两类内容池，不在导出时强制跨池去重。

## 第一阶段产物

- 计划留档文档：本文件。
- 资料摘要：`docs/讨论与修改日志/发布者及上下游改进计划/我们有什么待发布-过去两个月.md`
- 全量明细导出：`data/exports/publisher_improvement/我们有什么待发布-过去两个月.csv`
- 可重复运行脚本：`tools/export_publisher_improvement_candidates.py`

## 第一阶段切分原则

第一版先做“可快速复盘、可继续筛选”的启发式切分，不追求一次就成为最终发布者分类。

导出资料至少包含：

- `类型`
- `内容`

同时补充这些辅助字段，便于后续筛选：

- `领域`
- `内容池`
- `来源`
- `时间`
- `发布渠道`
- `挂后台原因`

## 已确认纳入计划、暂不改代码的方向

### 1. 竞品来源中的 AI 相关内容已改为上游直接丢弃

目标口径：

- 来自 `blockbeats / panews / jinse` 的竞品内容，如果判断为 AI 相关，则后续不再进入正常判断保留、写作和发布者链路。
- 预期改造位置在发布者上游，更靠近判断者边界，而不是靠发布者阶段兜底。

这样做的原因：

- 竞品链路的目标仍应以 Crypto 快讯为主。
- AI信源已经是独立来源，竞品里的 AI 内容继续混入同一链路，会放大样本噪声。
- 如果后续发布者只讨论非 AI信源主链路，这批内容应尽量在更早阶段被切掉。

### 2. 已拆成两个发布者口径

当前发布者配置已经拆成两套口径：

- `AI信源`
- `常规`

这里的 `常规` 当前承接自动化主发布链路里除 `tasks.source = 'ai_source'` 之外的来源，包括 X、外媒、竞品、金十等。

当前实现状态：

- `常规` 已启用并内置通过 / 排除规则。
- `AI信源` 前端显示整块，但置灰并标注 `暂未启用`；运行时默认挂后台。

### 3. 常规发布者的第一版放行范围

当前已经确认的第一版类型口径如下。

可放行候选：

- 链上大额 / 巨鲸 / 清算
- 持仓 / 增减持 / 储备
- 政策 / 监管 / 法案
  - 但只保留与 Crypto 明确相关的内容
- 产品发布 / 升级 / 上线
  - 但只保留 Crypto 相关内容
- 宏观数据 / 利率 / 央行
- 机构观点 / 人物表态
- 融资
- 安全 / 事故 / 停运
  - 目前只接受黑客、被盗、漏洞、安全事故、停机停运这类明确安全事件
- 项目 / 组织重大动向
  - 包括裁员、组织调整、关键高管变动、项目停运、业务关闭、产品停止维护、重大转向

明确排除：

- 地缘政治
- 市场行情 / 价格异动
- 激励 / 赠款 / 活动
  - 例如交易大赛、奖池、奖励活动
- 战略合作

列为 `other` 后直接忽视：

- IPO / 上市 / 发售
- 并购 / 重组 / 整合
- 主网 / 测试网 / 基础设施

### 4. `other` 的 prompt 约束

这里的“列为其他，直接忽视”有明确约束：

- 后续 prompt 不需要为这类内容单独写规则。
- 只有命中明确放行分类时才进入对应分类。
- 没有命中明确分类的内容，默认归入 `other`。
- 进入 `other` 的内容一律忽视，不进入放行判断。

### 5. 本阶段新增产物

围绕上述口径，当前已经新增一份过滤 AI 之后的中间资料：

- `docs/讨论与修改日志/发布者及上下游改进计划/非AI信源发布者-类型初筛总结.md`
- `data/exports/publisher_improvement/非AI信源发布者-过去两个月.csv`

这份资料用于下一轮继续做“待二次筛选”的子类收敛，不代表代码已实现这些规则。

### 6. 关于“项目 / 组织重大动向”

像 `Robinhood近期裁员，BitGo削减15%员工` 这类内容，更接近“项目 / 组织重大动向”。

当前结论：

- 已新增为独立通过规则 `项目 / 组织重大动向`。
- 不再复用旧分类 `major_project_progress` 的语义。
- 新发布者不要求模型输出分类名，只输出 `pass / reject`。
