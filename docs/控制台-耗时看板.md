# 控制台-耗时看板

## 职责

`耗时看板` 是控制台一级页面，用于查看主快讯制作链路各环节耗时。默认视图不区分信源，合并展示 `24h / 72h / 7d` 三个固定窗口的总览指标；`按流程拆分` 仅作为排查视图。

该页面只消费后端返回的聚合快照，不展示全量明细、不拉取正文、不读取 raw payload 或模型输出大字段。

## 数据来源

- 统计范围只覆盖主写作链路：`x`、`blockbeats`、`panews`、`jinse`、`non_mainstream_media`、`ai_source`、`jin10`。
- 后端计算时只读取 Supabase/Postgres 中的轻量字段：`tasks.id/source/status/created_at/metadata`，以及 `x_task_pipeline` 的阶段完成时间戳、`news_type`、`publisher_decision`。
- 标题提醒链路 `external_media_alert_pipeline` 不纳入 V1，因为当前没有等价的阶段完成时间戳。
- 前端不直接读取 `tasks`、`x_task_pipeline` 或 Supabase 备份表。

## 本地快照

`editor-plugin-api-server` 启动时挂载耗时快照线程：

- 本地 SQLite 路径：`data/runtime/pipeline_timing.sqlite`。
- 服务启动时如果本地快照缺失或最新快照超过 1 小时，会立即计算一次。
- 后续默认每 `3600` 秒刷新一次，可通过 `PIPELINE_TIMING_REFRESH_INTERVAL_SECONDS` 调整。
- 本地 SQLite 只保留最近 14 天快照，避免文件长期膨胀。
- 前端进入页面或点击顶部刷新按钮时，只读取本地最新快照，不触发实时重算。

## 控制台接口

接口复用控制台后端鉴权：

```text
POST /console/pipeline-timing/get
```

- 请求必须携带 Supabase Auth token。
- 后端继续使用 `console_admins` 白名单校验控制台管理员权限。
- 返回结构固定为：

```json
{
  "generated_at": "ISO time",
  "windows": [
    {
      "hours": 24,
      "overall": {},
      "by_stage": [],
      "by_flow": [],
      "status_breakdown": []
    }
  ]
}
```

## 指标口径

固定窗口为 `24`、`72`、`168` 小时，按 `tasks.created_at >= generated_at - window` 取样。

展示指标：

- 样本数：窗口内符合主写作链路来源范围的任务数。
- 完成数：能计算出 `created_at -> publish_completed_at` 非负总耗时的任务数。
- 完成率：完成数 / 样本数。
- 均值：阶段或总耗时的平均秒数。
- 中位数：阶段或总耗时的中位秒数。

阶段统一为：

- `judge`
- `search`
- `write`
- `format`
- `publisher_decision`
- `publish_finalize`
- `total`

缺失时间戳不参与对应阶段均值和中位数；相邻阶段出现负数差值时忽略该阶段样本，避免异常回填时间污染统计。

## 流程归一

X 与金十按判断优先链路计算：

```text
created_at -> judge -> search -> write -> format -> publisher_decision -> publish_completed
```

竞品、Crypto信源全文、AI信源全文按查重优先链路计算：

```text
created_at -> search -> judge -> write -> format -> publisher_decision -> publish_completed
```

按流程拆分的分组为：

- `X常规`
- `X-AI信源账号`
- `竞品`
- `Crypto信源全文`
- `AI信源全文`
- `金十`

## Supabase 归档

Supabase 备份表为 `pipeline_timing_snapshots`，只保存聚合 JSON、窗口小时数、生成时间和创建时间。

- 表结构由显式初始化命令创建：`python backend\src\main.py editor-plugin-init-db`。
- 长驻服务启动时不自动 DDL。
- 写入 Supabase 是 best-effort：失败只记录日志，不影响本地 SQLite 快照，也不影响前端读取。
- 该表不作为前端实时查询来源；浏览器只访问 `/console/pipeline-timing/get`。
