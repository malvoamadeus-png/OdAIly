# 完整程序架构

## 当前有效主链路

- `X`：`x_capture` 入库后进入 `x_processing(judge -> search -> write -> format_publish -> publish)`
- `竞品快讯`：直接进入 `x_processing(search -> write -> format_publish -> publish)`
- `外媒全文`：统一由 `non_mainstream_media` 收集，写入 `tasks.source = 'non_mainstream_media'`，进入 `x_processing(judge -> search -> write -> format_publish -> publish)`
- `外媒标题提醒`：统一由 `non_mainstream_media` 的 `alert_only` 站点写入 `tasks.source = 'external_media_alert'`，进入 `external_media_alert(domain_judge -> search -> notify)`
- `AI信源全文`：统一由 `non_mainstream_media` 中 `source_group = 'ai_source'` 的 `write_flow` 站点收集，写入 `tasks.source = 'ai_source'`，进入 `x_processing(search -> judge -> write -> format_publish -> publish)`
- `AI信源标题提醒`：统一由 `non_mainstream_media` 中 `source_group = 'ai_source'` 的 `alert_only` 站点写入 `tasks.source = 'ai_source_alert'`，进入 `external_media_alert(domain_judge -> search -> notify)`

## 外媒系统级契约

- 业务上外媒和 AI信源都保留两条模式：`write_flow` 与 `alert_only`
- 外媒 source 映射：
  - `write_flow` -> `tasks.source = 'non_mainstream_media'`
  - `alert_only` -> `tasks.source = 'external_media_alert'`
- AI信源 source 映射：
  - `write_flow` -> `tasks.source = 'ai_source'`
  - `alert_only` -> `tasks.source = 'ai_source_alert'`
- 站点通过 `non_mainstream_media_sources.source_group` 区分 `external_media` 与 `ai_source`
- `mainstream_media` 只保留历史兼容，不再作为新的生产入口
- `external_media_alert_fetcher` 已退役，不再是生产必需 worker

## 控制台访问契约

- 控制台仍是浏览器直连 Supabase 的 Vite 前端，不新增独立控制台后端。
- 控制台访问依赖 Supabase Auth 邮箱密码登录；公开 `anon key` 只用于初始化客户端，不授予控制台数据权限。
- 控制台业务表统一通过 `authenticated + console_admins 白名单` 的 RLS 规则放行。
- 后端 worker 继续通过 `SUPABASE_DB_URL / DATABASE_URL` 直连 Postgres，不依赖控制台登录态，也不受前端会话过期影响。

## 插件契约

- 插件是独立 Chrome 插件，不属于控制台；代码位于仓库根目录 `extension/`，当前以 unpacked extension 方式加载。
- 插件当前规划两个一级模块：`信息流` 与 `快讯生成`。
- `信息流` 消费的是当前 Telegram 通知渠道里的同源信息，不新增新的抓取或审核来源。
- `信息流` 当前聚合四类内容：`新快讯`、`审核者`、`此前消息`、`巨鲸`；其中 `新快讯 / 审核者` 位于高频区，`此前消息 / 巨鲸` 位于低频区。
- 信息流 RPC 按区域和类型预留可见席位，避免 `审核者` 长期挤掉 `新快讯`，或低频区单一类型长期挤掉另一类。
- 信息流只展示最近 `30 分钟` 内的内容，旧消息不再占用侧边栏空间；窗口内多条消息通过侧边栏主视图纵向滚动承载，不压缩高频区卡片。
- `新快讯` 在插件里区分 `已直发` 和 `挂后台`；这里的 `挂后台` 仅指发布状态，不等于审核者提醒。
- `审核者` 与 `此前消息` 两类卡片当前提供 `接受 / 拒绝` 反馈，并把反馈人和反馈时间写入数据库，用于值班时效统计与误报复盘。
- `审核者` 与 `此前消息` 的详情链接优先指向 Odaily 对应快讯页；反馈成功后卡片不再展示操作按钮。
- `快讯生成` 由 X 页面内容抽取触发：content script 在 `x.com / twitter.com` 帖文操作区注入 Odaily 按钮，点击后打开 side panel 并切到 `快讯生成`。
- `快讯生成` 的 `AI查重` 复用 Linux 服务器上的 `搜索者` 能力；插件只负责传入抽取正文并展示返回结果。
- `快讯生成` 的 `快速生成` 和 `生成快讯` 由编辑手动选择 `常规 / 融资 / 链上` 模板，复用 Linux 服务器上的 `编写者1 -> 编写者2` 能力；只返回生成结果，不直接进入判断者、发布者，也不接入发布后异步 `审核者`。
- `快速生成` 仅在文案生成调用中固定使用 `gpt-5.4-mini` + `low`；`生成快讯` 和自动化采集、编写、发布主链路继续使用现有 writer 模型与 reasoning 配置，默认 `gpt-5.5` + `medium`。
- `快讯生成` 当前通过独立轻服务命令 `editor-plugin-api-server` 对外提供 `POST /plugin/news-gen/search`、`POST /plugin/news-gen/generate` 与 `POST /plugin/news-gen/quick-generate`；两个生成接口都要求传入 `news_type = regular | funding | onchain`。
- 生产环境中，`快讯生成` 轻服务由 `deploy/odaily-editor-plugin-api.service` 托管，默认对外监听 `0.0.0.0:8765`，供插件内置地址直接访问。
- 插件鉴权独立于控制台权限，复用 `Supabase Auth`，但使用插件专用白名单 `editor_plugin_users`；`快讯生成` 与 `信息流` 共用同一套插件白名单。
- 插件当前不新增控制台后端；`信息流` 通过 Supabase RPC `editor_plugin_profile / editor_plugin_feed / editor_plugin_state / editor_plugin_mark_seen / editor_plugin_submit_feedback` 读写数据，`快讯生成` 通过 Linux 服务器的轻接口调用既有搜索/写作能力。
- `快讯生成` 的请求与结果会单独写入 `editor_plugin_generation_logs`，不写入正式 `tasks` 发布流水线。
- 当前插件安装包内置 `Supabase URL` 和公开可分发的 `publishable key`，值班编辑不需要手填连接参数。
- 当前插件安装包同时内置 `快讯生成` 服务地址；值班编辑不需要手填接口地址。
- 插件在线心跳仍未实现；当前告警仍以 Telegram 为主。
- 插件当前支持 `Side Panel` 打开状态下的新消息短音提醒。
- 模块细节见 `docs/信息流插件.md`、`docs/快讯生成插件.md`。

## 定时行情简报契约

- `gate-tradfi` 与 `us-market` 共用 `odaily-worker` 调度进程，但日历规则分开维护。
- `gate-tradfi` 只按北京时间工作日执行：
  - `morning`：北京时间上午。
  - `open`：按美东开盘时点触发，但仍要求该触发时刻对应的北京时间是周一到周五晚间。
- `us-market` 按美东交易日执行：
  - `premarket`：只保留手动执行能力，不进入自动调度。
  - `open`：美东开盘。
  - `close`：北京时间次日早晨播报上一美东交易日收盘，因此整条链路从北京时间周一晚间开始，到北京时间周六早晨结束。
- `us-market` 的自动调度必须同时跳过美东周末和 NYSE 休市节假日；`gate-tradfi` 不复用这套美东周末判定。

## 巨鲸监控契约

- `巨鲸` 是独立监控链路，不进入 `tasks` 写作发布流程。
- 控制台维护两组地址配置：`链上` EVM 地址与 `Hyperliquid` 地址；前端不负责轮询逻辑。
- `whale-watch-worker` 使用 Blockscout v2 按地址轮询启用链，默认 `ethereum,base`，不监听全量 RPC。
- `whale-watch-hyperliquid-worker` 轮询 Hyperliquid `userFills`，只识别开多、开空、平多、平空成交。
- 两个 worker 首次看到地址时都只 seed 当前视图，不播报历史活动。
- 链上活动按 `address_id + chain_key + tx_hash + activity_fingerprint` 去重后写入 `whale_watch_activities`；Hyperliquid 活动按 `address_id + fill_key` 去重后写入 `whale_watch_hyperliquid_activities`。
- Hyperliquid 支持两级提醒：单笔门槛和滚动聚合窗口门槛；默认是 `500000` 美元单笔门槛、`1000000` 美元聚合门槛、`600` 秒窗口，提醒链接统一指向 `https://hyperbot.network/trader/{address}`。
- 两条链路都复用 Telegram 播报，并向监督链路写入独立 heartbeat：`whale_watch` 与 `whale_watch_hyperliquid`。
- 模块细节见 `docs/巨鲸.md`。
