# OdAIly 完整程序架构

## 本文定位

本文是 OdAIly 的系统级契约和文档导航，只记录长期稳定的目标、模块边界、数据流和跨模块约定。

具体输入输出、Prompt、阈值、环境变量、运行命令和失败处理放在各模块细节文档中维护。修改代码时，如果改变的是整体流程或跨模块契约，应同步更新本文；如果改变的是单个模块行为，应优先更新对应细节文档。

## 总体目标

OdAIly 用于从多个信源自动收集原始信息，经过判断、去重、编写、格式化后，将生成的快讯以“不发布”状态推送至后台，供新闻值班人员审核。

系统要求各模块可以单独启动，便于后续按服务器、资源消耗和稳定性要求拆分部署。

本项目的业务主体是 Odaily。BlockBeats、PANews、金色财经等 Crypto 新闻媒体属于竞品信源；Odaily 自身快讯不是竞品信源，而是本方已发布语料，供搜索者和事件复盘使用。

## 基本要求

- 使用 Supabase/Postgres 存储原始信息、任务状态、处理结果和控制台配置。
- 每个 worker 模块都可以单独启用。
- 任务处理需要避免重复消费，同一条任务同一阶段只能被一个 worker 处理。
- 发布流水线只处理来源原发布时间仍在时效窗口内的任务，默认窗口为 10 分钟。
- 控制台配置变更后，服务器侧应准实时感知，不采用高频轮询。
- Prompt 控制台只管理 Prompt 内容，不管理模型、reasoning effort、超时等运行配置。
- 每个代码模块需要单独维护 README 或细节文档，用于记录运行方式、环境变量、表结构依赖和实现细节。

## 总体流程

```text
收集者层
  -> 任务/事件存储层 Supabase/Postgres
  -> AI 处理层
  -> 审核层 人工 + AI 辅助
  -> 发布层
  -> 反馈 & 风控层
```

不同来源可以有不同入口顺序：

- X 来源：收集者-X -> 判断者 -> 搜索者 -> 编写者1 -> 编写者2 -> 后台审核。
- 竞品来源：收集者-竞品 -> 搜索者 -> 判断者 -> 编写者1 -> 编写者2 -> 后台审核。
- 外媒全文写作链路：收集者-非主流媒体 -> 搜索者 -> 判断者 -> 编写者1 -> 编写者2 -> 后台审核。
- 外媒标题提醒链路：收集者-非主流媒体 -> 判断者-领域分类 -> 搜索者 -> 提醒者。
- 编写者3不插入主发布链路；以 Odaily 已发布快讯镜像作为统一输入，异步判断是否需要补充“此前消息”，可写结果只发送到 Telegram topic2，不写回网站或后台发布接口。进入编写者3前先执行共享排除词和正向触发过滤，命中排除词或属于非触发类型时不进入编写者3判断、检索或生成。
- Odaily 自身快讯：作为 reference 和事件复盘来源，不进入 `tasks` 编写队列。
- 审核者：以 Odaily 全量已发布快讯镜像作为统一输入，异步检查中文标点、语法、错别字和格式异常；只向 Telegram 发送疑似问题提醒，不自动改稿，不阻断发布。
- 事件复盘来源：竞品和 Odaily 快讯进入 `newsflash_*`，不按 10 分钟发布时效过滤。

## 模块划分

### 收集者层

收集者负责从外部系统获取数据，转换为统一结构，并写入任务表或事件复盘表。收集者不负责深度去重、品类判断、写作或推送后台。

当前与本轮已确认的收集者：

- 收集者-X：读取控制台配置的 X/Twitter 账号，抓取公开内容，写入 `tasks`，详见 `收集者-X.md`。
- 收集者-竞品：采集 BlockBeats、PANews、金色财经等竞品快讯，同时采集 Odaily 快讯作为 reference。金色财经使用官方 `api.coinmeta.info/live/list` 作为来源入口，详见 `收集者-竞品.md`。
- 收集者-非主流媒体：按站点适配非 API 抓取方案。当前同一收集者同时承载“全文写作链路”和“标题提醒链路”：前者写入 `tasks.source='non_mainstream_media'`，后者写入 `tasks.source='external_media_alert'`，详见 `收集者-非主流媒体.md`。

未来可增加其他收集者，例如更多非主流媒体、项目方账号、链上数据、交易所公告等。新增收集者需要在细节文档中说明来源、去噪规则、写入表、初始状态和后续处理入口。

### 调度与任务推进

处理层从 `tasks` 中筛选需要处理的数据，并按阶段推进任务状态。

基本规则：

- 固定信源可以跳过部分判断流程。
- X 来源先进入判断者；未被判断者排除时，再进入搜索者查重。
- 竞品快讯和外媒全文写作链路先进入搜索者；确认未与 Odaily 已发布快讯或 in-flight 候选事件重复后，再进入判断者。
- 外媒标题提醒链路先进入“判断者-领域分类”；只有 `crypto` 通过后才进入标题提醒专用搜索阶段。
- 搜索者的查重方案详见 `搜索者.md`。
- 任务推进需要使用行锁或租约机制，避免多个 worker 同时处理同一条任务。

### AI 处理层

AI 处理层按 worker 拆分，不使用一个大函数串完整个流程。

- 判断者：对候选内容做预筛和路由，具体输出结构详见 `判断者.md`。
- 判断者-领域分类：只服务外媒标题提醒链路，判断标题是不是 Crypto，详见 `判断者-领域分类.md`。
- 搜索者：判断新任务是否与 Odaily 已发布快讯、历史提醒或当前流水线候选事件重复，详见 `搜索者.md`。
- 编写者1：按来源和路由选择 Prompt，生成快讯初稿并记录 Prompt 版本；非主流媒体来源使用独立模板，不复用 X 的 `regular / onchain / funding` 模板，详见 `编写者1.md`。
- 编写者2：使用确定性规则标准化标题和正文，并推送后台，详见 `编写者2.md`。
- 提醒者：只服务外媒标题提醒链路，发送 Telegram，不调用写作和后台推送，详见 `提醒者.md`。
- 编写者3：在主发布链路之外，异步判断 Odaily 已发布快讯是否需要补充“此前消息”，并把最终拼接内容发送到 Telegram topic2；命中共享排除词或不满足正向触发条件时不处理，详见 `编写者3.md`。

### 审核层

主发布链路只负责把自动生成内容推送到后台，默认 `isPublish=false`，由新闻值班人员人工审核后发布。

审核者属于发布后的异步 AI 辅助审核能力，以 Odaily 全量已发布快讯镜像为输入，覆盖自动流水线内容和人工直接发布内容。审核者只检查中文文本质量问题，发现疑似问题时发送到审核者专用 Telegram topic，不自动修改已发布内容，不替代人工终审，详见 `审核者.md`。

### 发布层

发布层负责将审核通过的快讯正式发布。当前 OdAIly 自动流水线不直接发布，只推送待审核内容。

### 监督者

监督者是只报警的健康检查 worker，不自动重启服务、不修改任务状态、不清理队列。心跳、积压、失败和报警规则详见 `监督者.md`。

### 反馈 & 风控层

暂缓设计。后续可用于记录发布效果、错误案例、重复案例、敏感内容命中和人工改稿结果。

## 控制台能力

控制台负责维护人工可配置的数据，不承载 worker 内部运行逻辑。控制台总览详见 `控制台.md`。

当前与本轮已确认的控制台模块：

- X 抓取配置：账号、全局频率、单账号频率、并发数、抖动和启停，详见 `控制台-X抓取.md`。
- 非主流媒体抓取配置：抓取站点、全局频率和抖动。站点目录由后端维护，前端按“进入编写链路 / 标题提醒链路”分组展示，但不允许改模式，详见 `控制台-非主流媒体抓取.md`。
- Prompt 管理：模板内容、版本、发布和回滚，详见 `控制台-Prompt.md`。
- 竞品配置：竞品排除词，详见 `控制台-竞品配置.md`。
- 事件复盘：事件列表、收藏、备注和详情，详见 `控制台-事件复盘.md`。

控制台配置变更应通过数据库通知机制让服务器侧 worker 感知，不要求重启服务。

## 跨模块技术约定

### 任务状态推进

任务从 `tasks` 表进入处理流程，状态按阶段推进。当前 X 处理阶段示例：

```text
pending -> judging -> judged -> deduping -> deduped -> writing -> written -> formatting -> ready_review
```

判断者判定跳过时，任务不再进入搜索者、编写者和推送流程。搜索者确认重复时，任务进入重复状态，并在需要时挂到已有候选事件的 supporting sources。

in-flight 候选事件只覆盖仍可继续推进的主任务；主任务进入 `discarded`、`expired` 或任一失败终态后，必须同步释放对应 candidate，避免阻塞后续同事件任务。

不同来源的阶段顺序可以不同：

- X 来源：先判断，再搜索。
- 竞品来源：`pending -> deduping -> searched -> judging -> deduped -> writing -> written -> formatting -> ready_review`
- 外媒全文写作链路：`pending -> deduping -> searched -> judging -> deduped -> writing -> written -> formatting -> ready_review`
- 外媒标题提醒链路：`pending -> classifying -> classified -> deduping -> deduped -> notifying -> notified`
- 外媒全文写作链路判断通过后，不再细分成 `regular / onchain / funding`，统一进入 `non_mainstream_media_writer`。

状态名、失败状态和落库字段以处理阶段文档及当前 schema 为准。

### 发布时效边界

发布流水线有两道时效门：

- 收集者入口：X 和竞品任务写入 `tasks` 前，按来源原发布时间过滤过期内容。
- 处理兜底：处理 worker claim 到任务后、进入判断、搜索、写作或格式化推送前，再检查 `tasks.published_at`。

默认时效窗口由 `PROCESSING_FRESHNESS_WINDOW_SECONDS=600` 控制。判断基准是来源原发布时间 `published_at/created_at`，不是 `tasks.created_at`。来源发布时间缺失或超过窗口时，任务进入终态 `expired`，不调用 AI、不推送后台、不发送 Telegram。

该规则只约束发布流水线。竞品和 Odaily 快讯进入 `newsflash_*` 事件复盘表时不做 10 分钟过滤，用于补齐服务崩溃期间的事件空白。

### 配置变更通知

控制台配置变更不采用高频轮询。

当前 X 抓取配置采用 Postgres `LISTEN/NOTIFY` 思路：控制台写入 Supabase/Postgres 表，数据库触发通知，worker 监听通知后重新加载配置。

Prompt 模板发布也复用类似机制：

- 控制台保存草稿时不通知 worker。
- 控制台发布 Prompt 版本时通知 worker。
- worker 收到通知后刷新本地 Prompt 缓存。
- worker 启动时也必须主动加载当前生效配置，不能只依赖通知。

### Prompt 版本追溯

编写者1每次生成快讯时，需要记录实际使用的 Prompt 模板和版本。

这样后续如果出现质量问题，可以追溯：

- 哪条任务生成了该快讯。
- 使用了哪个 Prompt 模板。
- 使用了哪个 Prompt 版本。
- 使用了哪个 worker 模块处理。

### 来源链接暴露

X 快讯可以把原始 X 链接作为 `sourceUrl` 推送后台。

竞品和非主流媒体来源链接只用于内部排查和 Telegram 内部提醒。推送后台时 `source_url` 默认保持为空，避免后台和公开链路直接暴露外部来源。

### Telegram 通知

当编写者2完成格式化并成功推送后台后，可以由 Telegram bot 同步向指定群组发送通知。

Telegram 发送层支持可选 `message_thread_id`。配置 `TELEGRAM_MESSAGE_THREAD_ID` 后，现有编写者2内部提醒和监督者报警会发送到该群组 topic；未配置时仍发送到群组默认话题。需要临时验证 topic 时，可使用 `telegram-test --message-thread-id` 指定目标 topic。

Telegram 通知只用于提醒值班人员，不代替后台审核和发布流程。全文写作链路的通知失败不应阻断快讯保存到后台，但需要记录失败日志。不同来源可以使用不同提醒前缀，例如 X 来源使用“X 平台有新快讯”，外媒全文写作链路使用“非主流外媒有新快讯”，外媒标题提醒链路使用“外媒标题提醒：来源｜标题”。

审核者使用独立的 `AUDITOR_TELEGRAM_MESSAGE_THREAD_ID`，必须指向名称为“审核者”的专用 topic，不复用通用 `TELEGRAM_MESSAGE_THREAD_ID`。

### 文档维护

本文只描述系统级契约和跨模块边界。

细节文档负责维护模块输入输出、Prompt、阈值、命令、环境变量、表结构依赖和失败策略。新增或修改 worker、控制台模块、Prompt 或 schema 时，应同步检查 `AGENTS.md` 中的文档维护规则。
