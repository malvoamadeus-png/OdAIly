## 总体目标

OdAIly 用于从多个信源自动采集原始信息，经过判断、去重、编写、格式化后，将生成的快讯以“不发布”状态推送至后台，供新闻值班人员审核。

系统要求各模块可以单独启动，便于后续按服务器、资源消耗、稳定性要求拆分部署。

本项目的业务主体是 Odaily。BlockBeats、PANews、金色财经等专注 Crypto 领域的新闻媒体属于竞品信源；Odaily 自身快讯不是竞品信源，而是搜索者用于判断是否已发布过同一事件的本方历史语料。

## 基本要求

- 利用 Supabase/Postgres 储存原始信息、任务状态和处理结果。
- 每个 worker 模块都可以单独启用。
- 任务处理需要避免重复消费，同一条任务同一阶段只能被一个 worker 处理。
- 控制台配置变更后，服务器侧应准实时感知，不采用高频轮询。
- Prompt 控制台只管理 Prompt 内容，不管理模型、reasoning effort、超时等运行配置。
- 每个代码模块需要单独维护 README，用于记录运行方式、环境变量、表结构依赖和实现细节。

## 总体流程

[信源采集层 A] ->
[任务/事件存储层 Supabase/Postgres] ->
[AI 处理层 B] ->
[审核层 人工 + AI] ->
[发布层] ->
[反馈 & 风控层]

## 模块功能划分

### 1. 信息采集程序 A

负责从外部系统获取数据，转换为统一结构，并写入 `tasks` 表。

输入来源：

- X/Twitter 账号列表：多账号情况下，要求零散、并发、带一定抖动地发起抓取任务。
- 媒体：RSS / API。
- 竞品：首批包含 BlockBeats、PANews、金色财经等 Crypto 新闻媒体，用于发现竞品已经发布但 Odaily 尚未发布的快讯线索。
- Odaily 自身快讯：作为本方已发布语料，不进入竞品发布队列，主要供搜索者做近 1 天语义查重。
- 项目方账号。
- 链上数据：交易、地址、OI 等。

行为：

- 抓取外部数据。
- 转换为统一任务结构。
- 写入 `tasks` 表。
- 新任务写入后，应能唤醒后续处理 worker。

### 2. 调度逻辑

调度逻辑可以在 B 层实现，负责从 `tasks` 中筛选需要处理的数据，并推进任务状态。

基本规则：

- 固定信源可以跳过部分判断流程。
- X 来源先进入判断者流程。
- X 来源未被判断者排除时，进入搜索者查重；确认不重复后进入后续快讯编写和发送环节。
- 竞品快讯先进入搜索者，与近 1 天 Odaily 快讯做相似度查重；确认不重复后再进入判断者。
- 外媒、链上等来源后续按各自规则进入对应流程。
- 搜索者的查重方案详见 `搜索者.md`；竞品采集范围和入库规范详见 `竞品监控.md`。

任务推进需要使用行锁或租约机制，避免多个 worker 同时处理同一条任务。

### 3. 信息处理程序 B

B 层按 worker 拆分，不使用一个大函数串完全部流程。

#### 判断者

判断者负责对 X 来源内容做第一道轻量判断：先判断是否具备写成新闻快讯的基本价值，再把可处理内容路由到对应快讯类型。

它不是事实查证者，也不做深度去重；事实补充、去重和写作仍由后续搜索者、编写者承担。

输入：

- `tasks` 表中的 X 来源原始内容。

输出：

```json
{
  "route": "regular | onchain | funding | discard",
  "discard_type": "none | pure_emotion | baseless_trading_call | daily_chatter | non_crypto_ai"
}
```

说明：

- `route` 是判断者的核心输出：
  - `regular`：进入常规快讯模板。
  - `onchain`：进入链上快讯模板。
  - `funding`：进入融资快讯模板。
  - `discard`：不进入搜索者和编写者流程，任务标记为跳过。
- `discard_type` 只在 `route=discard` 时有业务意义；进入后续流程时固定为 `none`。
- `regular`：常规快讯。
- `onchain`：链上快讯。
- `funding`：融资快讯。
- 判断者不输出 confidence、template_key、reason、entities 等解释性字段。
- `route` 与 Prompt 模板的映射由代码固定维护。
- 模型采用 `gpt-5.4-mini`，用于快速完成简单判断任务。

预筛选丢弃类型：

- `pure_emotion`：纯情绪表达，没有可报道事实。例如“你应该购买更多的 BTC”“ETH 要起飞了”“太牛了”。
- `baseless_trading_call`：无逻辑的纯粹喊单、价格口号或无事实支撑的涨跌判断。例如“买入 SOL”“BTC 很快到 20 万美元”。
- `daily_chatter`：寒暄、日常状态、meme、节日祝福、无新闻事实的社区闲聊。
- `non_crypto_ai`：AI 泛科技内容，且不属于当前自动快讯流水线需要处理的 Crypto 新闻。

判断者 Prompt 要求：

- Prompt 必须明确告诉模型：任务分两步，先判断是否属于四类可丢弃内容；不属于丢弃内容时，再判断 `regular`、`onchain`、`funding`。
- Prompt 必须写清四类丢弃内容的定义和例子，特别是“无逻辑的纯粹喊单”与“有事实依据的市场事件”之间的区别。
- Prompt 必须强调：不要因为内容有主观语气、营销语气或表达不规范就直接丢弃；只要有明确事实，就进入后续新闻流水线。
- Prompt 必须要求只输出结构化 JSON，不输出解释文本。
- Prompt 内容和版本由 Prompt 管理模块维护；模型、超时、重试等运行参数仍由服务端配置维护。

边界规则：

- 只要内容包含明确主体、明确动作和可报道结果，默认不要因为语气主观而丢弃；后续由编写者改写为新闻表达。
- 如果内容本身是引用、转发或截图说明，但原始任务中能看到完整事件信息，可以继续处理；只有捕获内容无法还原事件时才丢弃。
- 判断者只做“是否值得进入新闻流水线”的预筛选，不负责判断真假、补全背景或判断是否已经重复。
- 当前阶段，AI 泛科技内容由判断者识别并丢弃，不进入编写发布流程；未来如果 Odaily 需要发布部分 AI 相关新闻，再通过配置启用对应品类、搜索语料和编写模板。

#### 搜索者

搜索者负责判断通过判断者预筛选的新信息是否与过去已处理或已发布内容重复。

搜索者当前承担两类任务：

- X 来源：判断者未排除后，搜索者检查其是否与近 1 天 Odaily 快讯或当前流水线中的候选事件重复。
- 竞品快讯：先搜索近 1 天 Odaily 快讯，明显重复则直接跳过；未重复时再进入判断者。

搜索者使用向量相似度作为第一层快速判断，并在中间相似度区间调用搜索者-AI 做“是否同一事件”的快速判断。初始方案、阈值建议、AI 输入输出、双重发布规避和本地 embedding 缓存策略详见 `搜索者.md`。

通过搜索者后，任务进入编写队列。

#### 编写者1

编写者1负责根据任务来源和判断者结果调用对应 Prompt，生成快讯初稿。

规则：

- X 来源根据判断者输出的 `route` 选择常规、链上、融资三类模板之一；`discard` 任务不进入编写队列。
- 外媒来源调用外媒模板。
- 链上来源调用链上模板，发言人为 Odaily。
- 编写者1使用 AI 模型生成内容。
- 实际使用的 Prompt 版本需要记录到任务处理结果中，便于回溯。

#### 编写者2

编写者2负责对编写者1生成的内容进行标准化、格式化处理。

规则：

- 不调用 AI 模型。
- 使用正则和确定性规则处理格式。
- 固定前缀为“Odaily星球日报讯”，即“Odaily星球日报讯（空一格）正文……”。
- 其余格式化方案详见 `编写者2.md`。

编写者2处理完成后，将快讯以 `isPublish=false` 状态推送至后台。

#### 编写者3

补充此前消息，……

#### Telegram 通知

由于生成的快讯默认以“不发布”状态保存在后台，新闻值班人员可能无法第一时间看到新内容，因此需要增加 Telegram group 通知。

规则：

- 当编写者2完成格式化，并成功将快讯推送到后台后，由 Telegram bot 同步向指定群组发送通知。
- 通知内容格式：

```text
{信源}有新快讯：{标题}
{链接}
```

- 信源显示名：`x` 显示为 `X平台`，`blockbeats` 显示为 `律动`，`panews` 显示为 `PANews`，`jinse` 显示为 `金色财经`。
- 如果链接为空，只发送标题行，不输出空链接或占位文本。
- X 链接使用原始 X 链接；竞品 Telegram 内部通知可以展示竞品原始链接。
- 推送后台时竞品 `source_url` 仍保持为空，避免后台和公开链路暴露竞品来源。
- Telegram 通知只用于提醒值班人员，不代替后台审核和发布流程。
- 如果 Telegram 通知失败，不应阻断快讯保存到后台，但需要记录失败日志，便于排查。

### 4. 审核层

审核者细节暂缓设计。

当前要求：

- 编写完成后的快讯进入后台，默认 `isPublish=false`。
- 新闻值班人员在后台进行人工审核。
- 后续可增加 AI 辅助审核、风险提示、重复提示等能力。

### 5. 发布层

发布层负责将审核通过的快讯正式发布。

当前系统只负责把生成内容推送到后台，不直接发布。

### 6. 监督者

监督者是只报警的健康检查 worker，不自动重启任何服务。

职责：

- 监控 `x_capture`、`competitor_monitor`、`x_process_judge/x_process_search/x_process_write/x_process_format_publish` 等 worker 的心跳。
- 心跳同时区分 `last_seen_at` 和 `last_success_at`：服务还在运行但持续失败时，也要按无近期成功心跳报警。
- 检查可处理任务是否长时间积压，例如 `pending`、`judged`、`searched`、`deduped`、`written`。
- 检查处理中任务是否卡住或锁过期，例如 `judging`、`deduping`、`writing`、`formatting`。
- 检查最近窗口内 `judge_failed`、`search_failed`、`write_failed`、`format_failed`、`publish_failed` 是否异常增多。
- 检查 X 抓取最近是否没有成功 attempt，以及竞品监控是否长时间没有成功心跳。
- 报警通过 Telegram 发送，默认同一 `alert_key` 30 分钟内不重复刷屏。

监督者不负责：

- 不自动 `restart` systemd 服务。
- 不修改任务状态。
- 不清理队列。

运行方式：

```bash
python backend/src/main.py pipeline-supervisor --once
python backend/src/main.py pipeline-supervisor
```

建议 systemd 服务名：`odaily-pipeline-supervisor.service`。

### 7. 反馈 & 风控层

## 控制台能力

### 功能1：配置 X 抓取对象

参考 `控制台-X抓取.md`。

控制台可以配置：

- 抓取账号。
- 全局抓取频率。
- 单账号抓取频率。
- 并发数。
- 抖动时间。
- 启用/停用状态。

当前 X 配置变更应通过数据库通知机制让服务器侧 worker 感知，不要求重启服务。

### 功能2：配置竞品过滤词表

参考 `竞品监控.md`。

控制台可以配置：

- 竞品过滤词。
- 启用或停用某个过滤词。
- 删除过滤词。

过滤词只作用于竞品 `blockbeats`、`panews`、`jinse` 的标题和正文匹配，不作用于 X，也不作用于 Odaily reference。

### 功能3：配置 Prompt 模板

控制台需要支持 Prompt 编制能力。

范围：

- 管理 Prompt 内容。
- 管理 Prompt 版本。
- 发布某个版本为当前生效版本。
- 保留历史版本，便于回滚和追溯。

不包含：

- 不管理模型选择。
- 不管理 reasoning effort。
- 不管理超时、重试等运行参数。
- 不管理输出 schema。

Prompt 发布后，服务器侧 worker 应准实时知道有新版本生效，并在后续任务中采用新模板。

## 跨模块技术约定

### 任务状态推进

任务从 `tasks` 表进入处理流程。

状态推进建议按阶段拆分，例如：

```text
pending -> judging -> judged -> deduping -> deduped -> writing -> formatting -> ready_review
```

判断者输出 `discard` 时，任务应直接进入跳过状态，不再进入搜索者、编写者和推送流程。重复、失败、跳过等状态后续按实际模块补充。

不同来源的阶段顺序可以不同：

- X 来源：`pending -> judging -> judged -> deduping -> deduped -> writing -> formatting -> ready_review`。
- 竞品快讯：`pending -> deduping -> searched -> judging -> deduped -> writing -> formatting -> ready_review`。

搜索者需要同时检查已发布 Odaily 快讯和当前流水线中的 in-flight 候选事件，避免 X 信源、多个竞品在短时间内报道同一事件时触发多次发布。
搜索者确认重复时，任务进入 `duplicate`，并在需要时挂到已有候选事件的 supporting sources。

### 配置变更通知

控制台配置变更不采用高频轮询。

当前 X 抓取配置已经采用 Postgres `LISTEN/NOTIFY` 思路：控制台写入 Supabase/Postgres 表，数据库触发通知，worker 监听通知后重新加载配置。

Prompt 模板发布也应复用类似机制：

- 控制台保存草稿时不通知 worker。
- 控制台发布 Prompt 版本时通知 worker。
- worker 收到通知后刷新本地 Prompt 缓存。
- worker 启动时也必须主动加载当前生效配置，不能只依赖通知。

### Prompt 版本追溯

编写者1每次生成快讯时，需要记录实际使用的 Prompt 版本。

这样后续如果出现质量问题，可以追溯：

- 哪条任务生成了该快讯。
- 使用了哪个 Prompt 模板。
- 使用了哪个 Prompt 版本。
- 使用了哪个 worker 模块处理。
