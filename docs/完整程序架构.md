# OdAIly 完整程序架构

## 文档定位

本文只记录系统级契约、模块边界和文档索引，不重复单模块实现细节。
模块输入输出、Prompt、命令、环境变量和失败处理，以对应细节文档和实际代码为准。

## 系统目标

OdAIly 从多类外部来源持续采集候选内容，经过判断、去重、写作、格式化后，将结果推送到 Odaily 后台待人工审核。

当前有两类与本轮改动强相关的工作流：

- 发布主链路：X、竞品快讯、非主流媒体全文进入 `tasks`，由搜索者/判断者/编写者推进。
- 长期参考与事件复盘链路：Odaily 快讯和竞品快讯进入 `odaily_reference_items`、`newsflash_*`，供搜索和事件聚合复用。

## 存储分层

当前系统明确采用“双层存储”：

- Supabase/Postgres 是持久化、共享、控制台可见的权威业务存储。
- 本地 SQLite 是高频读写工作集，用于降低 Supabase egress。

本轮约定的本地运行时文件：

- 搜索工作集：`data/processed/searcher/searcher.sqlite`
- 竞品事件工作集：`data/processed/competitor_monitor/competitor_monitor.sqlite`

恢复原则：

- 本地库存在且可用时，优先走本地高频读路径。
- 本地库缺失或损坏时，worker 从 Supabase 拉取最近窗口重建，再恢复到 local-first。
- Supabase 的表名、前端读取方式和公开数据契约保持不变。

## 主流程

### 收集者层

- 收集者-X：抓取 X/Twitter 账号内容，写入 `tasks`。
- 收集者-竞品：抓取 BlockBeats、PANews、金色财经与 Odaily 快讯；竞品写入 `tasks`，Odaily 写入 `odaily_reference_items`；两者都进入 `newsflash_*` 事件复盘链路。
- 收集者-非主流媒体：按站点适配抓取，分别进入全文写作链路或标题提醒链路。

### AI 处理层

- 判断者：决定是否丢弃，以及进入哪类后续链路。
- 搜索者：做 Odaily 已发内容与 in-flight 候选事件去重。
- 编写者 1/2：生成草稿并格式化推送后台。
- 审核者：发布后异步巡检中文质量。

### 监督层

- 监督者只告警，不重启服务、不改任务状态。

## 本轮关键契约

### 搜索 local-first

搜索相关 worker 共用 `searcher.sqlite` 作为高频工作集。

本地缓存承载两类内容：

- 文档镜像：`odaily_reference_items`、活跃候选事件、外媒提醒历史
- embedding 缓存：对应文档和查询文本的向量结果

稳定态行为：

- `x_processing` 搜索阶段优先从本地读取 Odaily 参考语料和活跃候选事件。
- `external_media_alert` 优先从本地读取 Odaily 参考语料和已通知历史。
- 只有本地未预热时才回源 Supabase，并把最近窗口回填到本地。

### 竞品事件聚合 local-first

竞品事件聚合不再每轮从 Supabase 反复拉“最近事件源全文”。

当前契约是：

- `newsflash_items`、`newsflash_events`、`newsflash_event_sources` 仍由 Supabase 承担持久化和控制台读取。
- 竞品 worker 会把最近窗口的 `newsflash_items` 与 `event_sources` 镜像到本地 `competitor_monitor.sqlite`。
- 事件聚合读取最近语料时，优先从本地镜像取数。
- 本地 recent window 尚未预热时，只回源一次最近窗口并写入本地。
- 单条 item 当前归属若本地缺失，可按 item_id 回源补齐，避免重建整窗。

这意味着：热读路径已经脱离 Supabase，但远端仍是权威写入端。

### 心跳降频

所有主要 worker 共享同一条心跳策略：

- 稳定态最多每 60 秒写一次 `pipeline_worker_heartbeats`
- 状态切换时立即写
- 失败时立即写

监督者的超时阈值没有同步下调，仍高于新心跳频率。

### `x_capture_attempts` 采样

`x_capture_attempts` 仍保留，用于控制台观测和排障，但不再每次空轮询都写库。

写入规则：

- 非 success 一律写
- `new_count > 0` 或 `saved_count > 0` 一律写
- “success 且什么都没发生”的轮询，按账号最多每 10 分钟采样一条
- 如果元数据指纹发生实质变化，即使仍是空轮询也会写

### 监督者健康判定补充

由于 `x_capture_attempts` 被采样，监督者对 X 收集健康的判定改为：

- 最近成功 attempt 存在，或
- 最近成功 heartbeat 存在

二者满足其一即可认为收集链路仍在健康工作，避免采样后误报。
