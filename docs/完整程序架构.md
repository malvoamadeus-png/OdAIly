# 完整程序架构

## 当前有效主链路

- `X`：`x_capture` 入库后提交本地流水线 `write_flow` job；标记为 `AI信源` 的 X 账号走 `judge_ai -> search -> write -> format_publish -> publish`，其他 X 走 `judge_crypto -> search -> write -> format_publish -> publish`
- `竞品快讯`：`competitor_monitor` 入库后提交本地流水线 `write_flow` job，执行 `search -> judge_crypto -> write -> format_publish -> publish`
- `Crypto信源全文`：统一由 `non_mainstream_media` 收集，写入 `tasks.source = 'non_mainstream_media'` 后提交本地流水线 `write_flow` job，执行 `search -> judge_crypto -> write -> format_publish -> publish`
- `Telegram-first Crypto信源全文发现`：`non_mainstream_media_telegram` 独立轮询 `PhoenixNewsEN`，只处理 `CoinDesk / Cointelegraph / Decrypt`，命中原站链接并抓到正文后，同样写入 `tasks.source = 'non_mainstream_media'` 并提交本地流水线 `write_flow` job
- `Crypto信源标题提醒`：统一由 `non_mainstream_media` 的 `alert_only` 站点写入 `tasks.source = 'external_media_alert'` 后提交本地流水线 `alert_only` job，执行 `domain_judge -> search -> notify`
- `混合信源全文`：统一由 `non_mainstream_media` 中 `source_group = 'mixed_source'` 的 `write_flow` 站点收集，先在收集器内分流为 `crypto / ai / discard`，再分别写入现有 Crypto信源或 AI信源链路
- `混合信源标题提醒`：统一由 `non_mainstream_media` 中 `source_group = 'mixed_source'` 的 `alert_only` 站点收集，先在收集器内分流为 `crypto / ai / discard`，再分别写入现有 Crypto信源或 AI信源标题提醒链路
- `AI信源全文`：统一由 `non_mainstream_media` 中 `source_group = 'ai_source'` 的 `write_flow` 站点收集，写入 `tasks.source = 'ai_source'` 后提交本地流水线 `write_flow` job，执行 `search -> judge_ai -> write -> format_publish -> publish`
- `AI信源标题提醒`：统一由 `non_mainstream_media` 中 `source_group = 'ai_source'` 的 `alert_only` 站点写入 `tasks.source = 'ai_source_alert'` 后提交本地流水线 `alert_only` job，执行 `domain_judge -> search -> notify`
- `金十`：`jin10_monitor` 轮询金十快讯接口，写入 `tasks.source = 'jin10'` 后提交本地流水线 `write_flow` job，执行 `judge_jin10 -> search -> write -> format_publish -> publish`
- Supabase/Postgres 只作为原始记录、阶段结果、最终结果和失败原因的档案库；生产主链路不再依赖 `tasks.status + locked_by/locked_until + 多个 worker claim` 做阶段交接。
- 搜索者查重使用的 Odaily 已发布快讯历史以 Odaily 官方快讯 API 为真实来源；`odaily_reference_items` 是本地同步后写入 Supabase 的档案备份，只在 Odaily API 不可用时作为兜底读取源。
- 本地 job 唯一键统一为 `source + source_item_id`，重复来源内容不会重复进入本地队列。
- 所有生产链路写入 `tasks.content`、`odaily_reference_items.content`、`newsflash_items.content`、插件 `post_text` 以及编写阶段正文时，统一保留有效段落内容，但会移除空白段落；多段正文只使用单个 `\n` 作为段间分隔，不保留空行。
- 为满足事件复盘和阶段耗时分析需求，主写作链路需要把各阶段完成时间持久化到 `x_task_pipeline`。当前系统级阶段时间戳包括：`judge_completed_at`、`search_completed_at`、`write_completed_at`、`format_completed_at`、`publisher_decided_at`、`publish_completed_at`。复盘时应优先使用这些显式时间戳，而不是仅靠 `tasks.updated_at` 或结果表创建时间反推。
- `耗时看板` 复用这些阶段时间戳做聚合分析，不引入新的任务状态或 worker 交接边界；V1 只覆盖主写作链路，不纳入标题提醒链路 `external_media_alert_pipeline`。

## 本地优先目标架构

- 长期目标是插件信息流本地化、主生产链路本地优先、Supabase 降级为异步档案库。
- 插件信息流目标上从 Linux 本地 feed store 读取最近消息、分区状态和反馈状态，不再在刷新时实时触发 `editor_plugin_feed` 聚合查询。
- 采集、处理、审核、巨鲸、Writer3 等生产事件目标上先写 Linux 本地库或本地队列，再由异步 syncer 归档到 Supabase。
- Supabase 短时不可用、session pooler 满或远端 SQL 变慢，不应阻断插件信息流展示和主生产链路推进；它主要承担控制台查询、历史复盘和备份归档。
- 当前代码已完成插件轻服务请求路径本地化，并已让新快讯、Crypto信源标题提醒、审核者、Writer3 此前消息、链上巨鲸和 Hyperliquid 巨鲸在产出时直接写 Linux 本地 feed store；后台 syncer 默认只做反馈归档，Supabase feed 回填需显式开启。目标架构和迁移边界见 `docs/本地优先与Supabase档案库架构.md`。

## Crypto信源与混合信源系统级契约

- 业务上 Crypto信源、AI信源、混合信源都保留两条模式：`write_flow` 与 `alert_only`
- Crypto信源站点还带只读发现模式 `discovery_mode`：
  - `direct`
  - `telegram_primary_direct_fallback`
- Crypto信源 source 映射：
  - `write_flow` -> `tasks.source = 'non_mainstream_media'`
  - `alert_only` -> `tasks.source = 'external_media_alert'`
- AI信源 source 映射：
  - `write_flow` -> `tasks.source = 'ai_source'`
  - `alert_only` -> `tasks.source = 'ai_source_alert'`
- 混合信源 source 映射：
  - 收集阶段先分流，不新增独立 `tasks.source`
- 站点通过 `non_mainstream_media_sources.source_group` 区分 `external_media / ai_source / mixed_source`
- `mainstream_media` 只保留历史兼容，不再作为新的生产入口
- 写作模板层面，Crypto信源全文使用 `mainstream_media_writer`，AI信源全文使用独立 `ai_source_writer`；标记为 `AI信源` 的 X 账号仍按 X 常规写作使用 `x_regular_writer`。
- `external_media_alert_fetcher` 已退役，不再是生产必需 worker

## 运行面

- 本地流水线：
  - `local_pipeline`
  - HTTP 入口默认 `127.0.0.1:8776`
  - 本地 SQLite 队列位于 `data/runtime/local_pipeline.sqlite`
  - 服务启动时会把锁定超过 `30` 分钟的本地 `running` job 回收到可重试状态，避免历史僵尸 job 永久卡住队列表现
- 常驻 worker 的单轮外围异常应被捕获并进入下一轮，避免一次配置读取、任务领取、外部 API 或数据库短暂波动导致进程永久退出；真正的异常由 heartbeat、失败状态和监督者 Telegram 告警暴露
- 常驻收集：
  - `non_mainstream_media`
  - `non_mainstream_media_telegram`
  - `x_capture`
  - `competitor_monitor`
  - `jin10_monitor`
- `non_mainstream_media_telegram` 只负责 Telegram-first 发现入口，不替代原 `non_mainstream_media` 站点直抓 worker；两者并行运行。
- 旧 `x-process-worker` 和 `external-media-alert-worker` 命令保留为紧急回滚入口；生产 systemd 不再启动这些分阶段 worker。
- 收集器提交本地流水线失败时，不把来源 item 标记为已看，下一轮继续重试。
- 切换前未完成旧任务由 `local-pipeline-skip-legacy --execute` 标记为 `legacy_skipped`，不导入本地队列，也不继续补跑。
- 旧分阶段 worker 的数据库 `LISTEN` 开关默认关闭；旧命令只作为紧急回滚入口保留。

## 控制台访问契约

- 控制台主体仍是浏览器直连 Supabase 的 Vite 前端；发布者规则、耗时看板和 BlockBeats key 状态例外，走服务器接口读写 Linux 本地配置或快照。
- 控制台访问依赖 Supabase Auth 邮箱密码登录；公开 `anon key` 只用于初始化客户端，不授予控制台数据权限。
- 控制台业务表统一通过 `authenticated + console_admins 白名单` 的 RLS 规则放行。
- 发布者规则配置由 `editor-plugin-api-server` 暴露 `/console/publisher-rules/get` 与 `/console/publisher-rules/save`，但权限使用 `console_admins` 白名单，不使用插件白名单。
- 发布者规则运行时读取服务器本地 `data/config/publisher_rules.json`，Supabase `publisher_rule_config` 只保存当前快照，不作为 worker 轮询源。
- 发布者控制台前端优先读取服务器接口；当接口不可用时，前端回退展示 Supabase `publisher_rule_config` 当前快照，并提示加载告警，避免页面静默显示空规则。
- `耗时看板` 由 `editor-plugin-api-server` 暴露 `/console/pipeline-timing/get`，同样使用 Supabase Auth token 与 `console_admins` 白名单校验；前端只读取服务器本地 `data/runtime/pipeline_timing.sqlite` 中的聚合快照，Supabase `pipeline_timing_snapshots` 只做聚合归档备份。
- `竞品配置` 页的 BlockBeats API Key 由 `editor-plugin-api-server` 暴露 `/console/blockbeats-key/get` 与 `/console/blockbeats-key/save`，同样使用 Supabase Auth token 与 `console_admins` 白名单校验；key 与额度状态只保存到服务器本地 `data/config/blockbeats_key.json`，不写入 Supabase。
- 发布者自动发布是否开启，仅由规则文件中的 `regular.enabled` / `ai_source.enabled` 决定；`tasks.source = 'ai_source'` 和 `tasks.source = 'x'` 且 `tasks.metadata.x_account_is_ai_source = true` 都使用 `ai_source` 发布者规则块。旧 `publisher_settings.enabled`、`publisher_channels.enabled` 和北京时间窗不再参与运行时判定。
- 后端 worker 继续通过 `SUPABASE_DB_URL / DATABASE_URL` 直连 Postgres，不依赖控制台登录态，也不受前端会话过期影响。

## X 发言人命名契约

- `x_capture_accounts` 除抓取账号本身的配置外，还承载账号级 `写作名` 配置和插件展示用 `AI信源` 标记。
- `写作名` 的作用仅限于 X 发稿口径标准化，不改动原始抓取到的 `author_display_name`、`author_username` 和正文。
- `is_ai_source` 不改变 `tasks.source = 'x'`，但会让自动化主链路进入 `judge_ai`；保留后仍作为 X 常规快讯写作，发布阶段按 `AI信源` 发布者规则块处理。当前 `AI信源` 发布者关闭，因此这类任务不会调用发布者模型，会直接挂后台。
- 标记账号的新快讯在 Chrome 插件信息流中进入 AI区。
- 自动化 X 主链路与插件 `快讯生成` 共用同一套 X 发言人命名规则，不各自维护映射。
- 进入 `编写者1` 前，X 来源作者名按以下顺序计算 `effective_author_name`：
  - 优先使用账号配置里的 `写作名`
  - 没有 `写作名` 时回退到原始 `author_display_name`
  - 仍为空时回退到 `author_username`
- 常规模板里“{发言人}在X平台发文表示”中的 `{发言人}`，以及其他依赖 X 发言人名称的写作场景，统一使用 `effective_author_name`。

## 插件契约

- 插件是独立 Chrome 插件，不属于控制台；代码位于仓库根目录 `extension/`，当前以 unpacked extension 方式加载。
- 插件当前规划两个一级模块：`信息流` 与 `快讯生成`。
- `信息流` 消费现有业务上游结果，不新增新的抓取或审核来源；正常业务消息以插件展示为主，Telegram 只保留系统级错误、宕机、积压和程序异常告警。
- `信息流` 当前聚合五类内容：`新快讯`、`Crypto信源标题提醒`、`审核者`、`此前消息`、`巨鲸`，并按 `高频区 / AI区 / 低频区` 三个展示分区渲染。
- 高频区展示普通 `新快讯 / Crypto信源标题提醒 / 审核者`；AI区展示 `tasks.source = 'ai_source'` 和标记为 `AI信源` 的 X 账号新快讯；低频区展示 `此前消息 / 巨鲸`。
- 信息流接口返回字段不变，`lane` 当前取值为 `high / ai / low`。
- 信息流读模型按区域和类型预留可见席位，避免 `审核者` 长期挤掉 `新快讯`，或低频区单一类型长期挤掉另一类。
- 信息流当前只展示最近 `2 小时` 内的内容，旧消息不再占用侧边栏空间；窗口内多条消息通过各自分区内部滚动承载，不改变三个区的展示高度。
- `新快讯` 在插件里区分 `已直发` 和 `挂后台`；这里的 `挂后台` 仅指发布状态，不等于审核者提醒。
- `审核者` 与 `此前消息` 两类卡片当前提供 `接受 / 拒绝` 反馈，并把反馈人和反馈时间写入数据库，用于值班时效统计与误报复盘。
- `审核者` 与 `此前消息` 的详情链接优先指向 Odaily 对应快讯页；反馈成功后卡片不再展示操作按钮。
- 生产文本 LLM 调用统一经过本机 `odaily-litellm.service`，后端服务访问 `ODAILY_LLM_BASE_URL`。业务代码使用 `odaily-gpt-writer`、`odaily-deepseek-fast`、`odaily-deepseek-review`、`odaily-deepseek-auditor` 四个模型别名；真实 OpenAI 中转和 DeepSeek 官方配置只在 LiteLLM 配置与服务器 `.env` 中维护。DashScope embedding 不经过 LiteLLM。
- `快讯生成` 由 X 页面内容抽取触发：content script 在 `x.com / twitter.com` 帖文操作区注入 Odaily 按钮，点击后打开 side panel 并切到 `快讯生成`。
- `快讯生成` 的 `AI查重` 复用 Linux 服务器上的 `搜索者` 能力；插件只负责传入抽取正文并展示返回结果。
- `快讯生成` 的 `快速生成` 和 `生成快讯` 由编辑手动选择 `常规 / 融资 / 链上` 模板，复用 Linux 服务器上的 `编写者1 -> 编写者2` 能力；只返回生成结果，不直接进入判断者、发布者，也不接入发布后异步 `审核者`。
- 插件轻服务中 `AI查重` 与 `快速生成` / `生成快讯` 使用不同模型别名：查重读取 `SEARCH_AI_REVIEW_*` 并走 `odaily-deepseek-review`，生成读取 writer 配置并走 `odaily-deepseek-fast` 或 `odaily-gpt-writer`。
- `快讯生成` 的 X 作者命名不直接使用插件页面抽取到的显示名；服务端会按 `x_capture_accounts` 中与 `author_handle` 对应的 `写作名 -> author_display_name -> author_handle` 顺序计算 `effective_author_name`，再交给 `编写者1`。
- `快速生成` 默认使用 `odaily-deepseek-fast` + `low`；`生成快讯` 使用现有 writer 模型别名，默认 `odaily-gpt-writer`，但在插件轻服务内固定使用 `low` reasoning。
- 插件通过独立轻服务命令 `editor-plugin-api-server` 对外提供统一接口：`/plugin/auth/*` 负责插件登录态，`/plugin/feed/*` 负责信息流读写，`/plugin/news-gen/*` 负责快讯生成。
- `快讯生成` 的 `POST /plugin/news-gen/search`、`POST /plugin/news-gen/generate` 与 `POST /plugin/news-gen/quick-generate` 都要求插件登录态；两个生成接口都要求传入 `news_type = regular | funding | onchain`。
- 生产环境中，插件轻服务由 `deploy/odaily-editor-plugin-api.service` 托管，默认只监听 `127.0.0.1:8765`；公网入口由 `deploy/odaily-editor-plugin-api.Caddyfile` 提供 HTTPS 反向代理。
- 插件鉴权独立于控制台权限，复用 Supabase Auth 里的邮箱密码，但不依赖浏览器直连 Supabase Auth token 接口；轻服务通过 Postgres 校验 Supabase Auth 用户密码，再签发插件专用短 session。
- 插件轻服务承接邮箱密码登录时，生产地址必须使用 HTTPS；浏览器侧会拒绝向非 localhost 的 HTTP 插件服务提交密码。
- 插件使用专用白名单 `editor_plugin_users`；`快讯生成` 与 `信息流` 共用同一套插件白名单。
- `信息流` 当前实现由插件轻服务读写 Linux 本地 SQLite store；核心业务 worker 产出消息时直接写本地 feed，后台 syncer 默认只把本地 `接受 / 拒绝` 反馈异步写入 `editor_plugin_submit_feedback`。`editor_plugin_feed` 回填需显式开启，浏览器不直接调用 Supabase RPC，插件刷新请求路径也不再同步调用 `editor_plugin_feed`。
- `快讯生成` 的请求与结果会单独写入 `editor_plugin_generation_logs`，不写入正式 `tasks` 发布流水线。
- 当前插件安装包内置 HTTPS 插件轻服务地址；值班编辑不需要手填接口地址。
- 插件在线心跳仍未实现；系统错误、积压、心跳和程序异常告警仍以 Telegram 为主，审核者、此前消息、巨鲸和新快讯不发 Telegram。
- 插件当前支持 `Side Panel` 打开状态下的新消息短音提醒；新消息标题前 2 分钟使用红色显示，之后恢复普通颜色。
- 模块细节见 `docs/信息流插件.md`、`docs/快讯生成插件.md`。

## Supabase 维护契约

- Supabase schema / RPC / 索引初始化只由显式 `*-init-db` 命令承担；长驻 worker 和插件轻服务启动时不自动执行建表、建索引或 `CREATE OR REPLACE FUNCTION`。
- 即使是保留用于紧急回滚的旧 `x-process-worker` 命令，也不在启动路径自动执行远端 `init_schema()`；需要补 schema 时先显式运行对应 `*-init-db` 命令。
- 后端各 Postgres 仓储连接统一附带 `POSTGRES_CONNECT_TIMEOUT_SECONDS`、`POSTGRES_IDLE_IN_TRANSACTION_SESSION_TIMEOUT_MS` 和固定 `application_name`，用于限制连接卡死、抑制长期 `idle in transaction`，并让 `pg_stat_activity` 能直接定位来源服务。
- 部署新 schema 时，先在本地提交并推送代码，再在服务器执行对应 `*-init-db`，确认完成后重启相关服务；不要通过反复重启长驻服务触发迁移。
- Supabase 运行日志和已完成任务大字段清理由 `maintenance-cleanup` 命令承载。
- 命令默认 dry-run，只输出将删除或清理的行数；必须显式传入 `--execute` 才会修改数据库。
- 默认策略：运行日志类保留 `7` 天，插件 `接受 / 拒绝` 反馈保留 `90` 天，已完成任务的大字段在 `7` 天安全窗口后清理。
- 插件“已看”记录不作为业务留存目标；当前只保留反馈记录用于复盘。
- 细节和当前调研结论见 `docs/Supabase优化调研.md`。

## 定时行情简报契约

- `gate-tradfi` 与 `us-market` 共用 `odaily-worker` 调度进程，但日历规则分开维护。
- `gate-tradfi` 只按北京时间工作日执行：
  - `morning`：北京时间上午。
  - `open`：按美东开盘时点触发，但仍要求该触发时刻对应的北京时间是周一到周五晚间。
- `us-market` 按美东交易日执行：
  - `premarket`：只保留手动执行能力，不进入自动调度。
  - `open`：美东开盘。
  - `close`：北京时间次日早晨播报上一美东交易日收盘，因此整条链路从北京时间周一晚间开始，到北京时间周六早晨结束。
- `us-market` 的自动调度必须同时跳过美东周末和 NYSE 休市节假日；`gate-tradfi` 不复用这套美东周末判定。

## 巨鲸监控契约

- `巨鲸` 是独立监控链路，不进入 `tasks` 写作发布流程。
- 控制台维护两组地址配置：`链上` EVM 地址与 `Hyperliquid` 地址；前端不负责轮询逻辑。
- `whale-watch-worker` 使用 Blockscout v2 按地址轮询启用链，默认 `ethereum,base`，不监听全量 RPC。
- `whale-watch-hyperliquid-worker` 轮询 Hyperliquid `userFills`，只识别开多、开空、平多、平空成交。
- 两个 worker 首次看到地址时都只 seed 当前视图，不播报历史活动。
- 链上活动按 `address_id + chain_key + tx_hash + activity_fingerprint` 去重后写入 `whale_watch_activities`；Hyperliquid 活动按 `address_id + fill_key` 去重后写入 `whale_watch_hyperliquid_activities`。
- Hyperliquid 支持两级提醒：单笔门槛和滚动聚合窗口门槛；默认是 `500000` 美元单笔门槛、`1000000` 美元聚合门槛、`600` 秒窗口，提醒链接统一指向 `https://hyperbot.network/trader/{address}`。
- 两条链路正常活动都写入数据库并进入信息流插件，不再发送 Telegram 播报；同时向监督链路写入独立 heartbeat：`whale_watch` 与 `whale_watch_hyperliquid`。
- 模块细节见 `docs/巨鲸.md`。
