# 审核者

## 职责

审核者负责对 Odaily 已发布快讯做中文文本质量检查，发现疑似语法、错别字、表达不顺或明确格式异常时，记录结果供信息流插件和后续复盘使用。

审核者的目标是补足人工发稿场景中的低成本错误检查。当前新闻系统仍有大量快讯由人工直接发布，这些内容不会经过收集者、判断者、搜索者、编写者1和编写者2，因此审核者必须以 Odaily 全量已发布快讯镜像为输入，而不是只处理自动流水线生成的快讯。

审核者不负责：

- 自动修改已发布快讯。
- 自动撤回、下架或重新发布快讯。
- 判断新闻价值或是否应发布。
- 替代人工终审。
- 检查事实真伪、行情数据真实性或来源可靠性。

## 链路位置

审核者位于发布后的异步质量检查链路，不插入主发布链路。

```text
Odaily 已发布快讯同步到 odaily_reference_items
  -> 审核者领取未审核快讯
  -> GPT-5.4-mini 做中文质量检查
  -> 无问题则记录通过
  -> 有问题则记录结果，不发送 Telegram
```

审核者失败或超时时，不影响原快讯发布状态，也不影响判断者、搜索者、编写者或编写者3。系统级异常由监督者通过 Telegram 告警。

## 输入范围

输入来自 Odaily 已发布快讯镜像，至少包含：

- 来源侧 ID。
- 标题。
- 正文。
- 发布时间。
- 原始链接或后台可定位 ID。
- 原始 payload 中可用于排查的关键字段。

其中正文应保留 Odaily 原始段落换行；如果来源 HTML 已按段落或列表拆分，审核者应把这些换行视为正文结构的一部分，而不是先压平成单行再检查。

输入范围包括：

- 自动流水线生成并由人工审核后发布的快讯。
- 人工直接在后台新增或编辑后发布的快讯。
- 其他最终出现在 Odaily 已发布快讯接口中的快讯。

审核者不应从 `tasks` 表领取任务，因为 `tasks` 只覆盖自动流水线候选内容，不能覆盖人工直接发布内容。

## 检查重点

第一版只检查中文文本质量问题，重点包括：

- 语法错误：主谓宾残缺、语序明显不通、重复粘贴、断句异常。
- 错别字或同音误用：仅在上下文非常明确时提示，避免过度猜测。
- 明确格式异常：例如重复粘贴导致的结构损坏，但不把纯标点样式问题视为有效报警项。

第一版不检查或只低优先级提示：

- 标点符号问题本身，包括连续标点、英文标点混用、句末缺少必要标点、括号或引号不配对、半角/全角标点切换。
- 中英文之间、数字与中文单位之间、数字与币种之间、英文缩写与括号之间、中文与括号之间的空格风格。
- 千分位逗号、钱包地址展示、省略号、半角/全角括号、数量表达是否统一。
- 已存在的分行、段落换行或标签式列表换行；不要仅因上一行末尾没有句号、下一行以“交易开放时间：”“杠杆倍数：”等标签开头就提示补句号。
- 正文尾部固定标语，例如“在定价之前，看见变化”；该标语不进入语序或润色检查。
- 可省略的结构助词“的”，例如“14.25%消费税”“应 Shielded Labs 请求”这类表达不提示补“的”。
- 新闻标题中的常见省略式表达，例如“一男子”“一女子”“一地址”“一新创建地址”等标题体量词省略，不按语法残缺处理。
- 链上、钱包、地址、交易所资金流转语境中的动作词误判；对“提出 / 转出 / 转入 / 提取 / 存入”等表述，若上下文可解释为 transfer、withdraw、deposit 等链上动作，不提示改成“买入 / 购入 / 加仓”。
- 主观风格偏好。
- 可改可不改的润色建议。
- 需要外部事实核验才能判断的问题。
- 需要理解复杂行情或链上数据才能确认的问题。
- 依赖外部世界当前事实才能判断的人名、机构、职位、地名、项目名、代币名等实体替换；只有同一条快讯内部已经直接出现建议替换后的实体，且能引用该原文片段证明前后矛盾时才允许提示。
- 依赖外部背景知识、历史事件印象或常识时间线去猜测日期、年份是否写错的问题；只有同一条快讯内部存在可直接引用的日期矛盾或明确时间关系冲突时才允许提示。

## AI 模型

当前默认使用 GPT-5.5，`reasoning effort = medium`。

模型配置应属于 worker 运行配置，不放进 Prompt 控制台。建议通过环境变量或服务配置维护：

- `AUDITOR_MODEL`：默认 `gpt-5.5`。
- `AUDITOR_REASONING_EFFORT`：默认 `medium`。
- `AUDITOR_OPENAI_BASE_URL`：审核者独立兼容接口 base URL；为空时沿用 `X_PROCESS_OPENAI_BASE_URL` / `OPENAI_BASE_URL`。
- `AUDITOR_OPENAI_API_STYLE`：审核者独立接口风格；为空时沿用 `X_PROCESS_OPENAI_API_STYLE`。
- `AUDITOR_OPENAI_API_KEY`：审核者独立 API key；未设置且审核者 base URL 或模型名包含 `deepseek` 时，会读取 `DEEPSEEK_API_KEY` / `DEEPSEEK_API` / `DeepSeek_API`。
- `AUDITOR_OMIT_REASONING_EFFORT`：兼容不接收 reasoning 参数的接口，例如 DeepSeek 非思考模式。
- `AUDITOR_CHAT_RESPONSE_FORMAT_MODE`：`chat_completions` 下的 JSON 输出模式，默认 `json_schema`；DeepSeek 非思考测试使用 `json_object`。
- `AUDITOR_APPEND_JSON_SCHEMA_TO_PROMPT`：当接口只支持 JSON object 时，把 JSON Schema 约束追加到 prompt；审核者切 DeepSeek 时必须开启，否则容易因缺少 `severity/type/location` 等字段被解析器过滤。
- `AUDITOR_MAX_ITEMS_PER_RUN`：每轮最多检查条数。
- `AUDITOR_LOOKBACK_MINUTES`：启动或恢复时向前补扫窗口。
- `AUDITOR_TELEGRAM_MESSAGE_THREAD_ID`：历史兼容配置；正常审核者问题不再发送 Telegram。

如果实际 OpenAI 兼容服务对模型名、超时、JSON 输出或 reasoning effort 有差异，应在运行配置中处理，不把这些差异写入 Prompt 模板。

审核者问题只记录到状态表和信息流插件。Telegram 只用于监督者发现审核者 worker 停止、心跳丢失或程序异常这类系统级问题。

## Prompt 输出要求

审核者 Prompt 应要求模型输出结构化结果，便于代码稳定判断是否报警。

建议输出字段：

```json
{
  "has_issue": true,
  "severity": "low | medium | high",
  "issues": [
    {
      "type": "punctuation | grammar | typo | format | other",
      "location": "title | content",
      "original": "原文片段",
      "suggested": "建议改法",
      "reason": "一句话说明",
      "evidence": "仅数据类指正时填写的同文依据片段；否则为空字符串"
    }
  ],
  "summary": "面向值班人员的一句话摘要"
}
```

输出约束：

- `has_issue=false` 时不得输出润色建议。
- 只报告明确错误或高置信异常，不输出泛泛的“可优化表达”。
- `original` 必须是标题或正文中真实存在的短片段。
- `suggested` 只给最小必要改法，不整段重写。
- 如果 issue 涉及数字、金额、日期、比例、数量级等数据类指正，`reason` 必须直接写清“为什么错”，不能只写“疑似有误”。
- 如果 issue 涉及数字、金额、日期、比例、数量级等数据类指正，`evidence` 必须引用同一条标题或正文中真实存在、且能直接支持该指正的另一处片段；没有这类同文依据时，必须输出 `has_issue=false`。
- 如果 issue.suggested 改动了日期或年份，`evidence` 还必须包含同一条标题或正文中的另一处完整日期，或能直接证明时间关系冲突的原文片段；否则必须输出 `has_issue=false`。
- 如果 issue.suggested 改动了人名、机构、职位、地名、项目名、代币名等实体，`evidence` 必须直接引用同一条标题或正文里已经出现建议实体的原文片段；否则必须输出 `has_issue=false`。
- 代码会在解析模型输出后丢弃纯标点问题、固定尾部标语、补“的”、标题体量词补全、链上 transfer 动作误判、无同文证据的实体替换等已知误报；过滤后没有有效 issue 时按通过记录。
- 不输出模型推理过程。

## 通知边界

只有 `has_issue=true` 且至少存在一条有效 issue 时记录为 `flagged`，不发送 Telegram。

提醒内容应包含：

- 严重级别。
- 快讯标题。
- 发布时间。
- 后台或原文定位链接。
- 问题类型。
- 原文片段。
- 建议改法。
- 简短原因。
- 数据类指正的同文依据片段。

`telegram_result` 固定为 `skipped=true`，表示内容质量类提醒已转交信息流插件。审核者自身失败、超时、心跳丢失等系统级问题由监督者发现后再发 Telegram。

需要防止刷屏：

- 同一条快讯同一审核版本只提醒一次。
- 同一快讯重新发布或正文变化后，可以重新审核。
- 低严重级别问题可合并为一条消息发送。

## 信息流插件反馈

当前独立的 `信息流插件` 会消费审核者结果，并在插件里给值班编辑提供：

- `接受`
- `拒绝`

这两个动作只表示编辑是否认可审核者指出的问题成立，用于：

- 统计值班响应速度。
- 复盘审核者误报。

插件反馈本身不代表正文已修改，也不代表快讯已撤回或重发。当前生产以插件展示和数据库留痕为准。

## 状态与幂等

建议为审核者维护独立状态表，避免重复审核和重复提醒。

建议状态：

- `pending`：待审核。
- `processing`：处理中。
- `passed`：未发现明确问题。
- `flagged`：发现问题并已记录。
- `failed`：审核失败，可重试。
- `skipped`：输入不完整、过旧或不满足处理条件。

幂等键建议使用：

```text
source + source_item_id + content_hash + prompt_version
```

其中 `content_hash` 覆盖标题和正文。这样同一快讯内容未变化时不会重复报警；人工修正后如果 Odaily 镜像内容发生变化，可以重新审核。

## 运行边界

审核者应支持：

- `--once`：运行一轮后退出，便于测试和 cron。
- 常驻 worker：按间隔领取新发布快讯。
- 启动补扫：按 `AUDITOR_LOOKBACK_MINUTES` 补扫近期已发布快讯，避免短暂停机漏审。
- 常驻循环会捕获单轮外围异常并继续下一轮，避免一次领取任务、数据库连接或外部服务波动导致审核者进程退出。

运行命令：

```powershell
python backend\src\main.py auditor-init-db
python backend\src\main.py auditor-worker --once
python backend\src\main.py auditor-worker
```

建议 systemd 服务名：

```text
odaily-auditor.service
```

审核者不使用发布流水线的 10 分钟时效门。它检查的是已发布内容质量，可以允许较长补扫窗口，但应限制最大补扫数量，避免恢复后集中调用模型和集中报警。

## 与其他模块关系

- 收集者-竞品：负责同步 Odaily 已发布快讯镜像；审核者复用该镜像输入，但不参与竞品任务入库。
- 编写者2：负责自动流水线生成内容的确定性格式化；审核者覆盖发布后的全量快讯，包括人工直接发布内容。
- 编写者3：同样以 Odaily 已发布快讯镜像为入口，但目标是补充“此前消息”；审核者只做中文质量检查。
- 监督者：检查审核者心跳和失败率，但不替审核者修改状态；只有审核者 worker 异常这类系统级问题才发 Telegram。

## 需要注意的问题

- 误报成本：审核者应偏向“明确错误才记录”，否则信息流会很快失去参考价值。
- 标点边界：纯标点、括号样式、半角/全角切换不进入审核者报警范围，避免低价值刷屏。
- 数据类指正：审核者不能靠常识臆测改数字；只有同一条快讯内部存在可引用的矛盾或校验依据时才允许报警。
- 实体替换边界：审核者不能根据外部知识把人名、机构、职位、地名、项目名、代币名改成“更常见”的写法；只有同文已直接出现建议实体且能证明前后矛盾时才允许报警。
- 日期边界：审核者不能靠外部背景知识或事件记忆猜年份；改日期或年份时，必须能引用同文里的另一处完整日期或明确时间冲突依据。
- 全量覆盖：必须从 Odaily 已发布快讯镜像读取，不能只从自动流水线读取。
- 内容变化：需要用标题和正文 hash 识别人工修正后的重新审核。
- 链接定位：Telegram 必须给值班人员可定位到原快讯的链接或 ID，否则提醒不可执行。
- 费用和限速：全量发布后审核会持续调用模型，需要每轮上限、补扫窗口和失败退避。
- 术语保护：加密行业项目名、交易所名、代币名、英文缩写和链名不应被模型随意中文化或改写。
- 重复提醒：同一问题不要反复刷屏；低优先级问题应合并。
- 职责边界：审核者只提示文本问题，不判断事实真假，不自动改稿。

## 待定项

- 审核者状态表名和字段。
- 审核者 Prompt 模板是否纳入 Prompt 版本追溯表，还是先由代码种子文件维护。
- 是否在控制台增加审核者问题列表和人工处理状态。
- 是否需要对高严重级别问题额外通知值班负责人。
