# 快讯生成插件

> 本文描述 Chrome 插件里第二个一级模块 `快讯生成` 的当前实现。它与 `信息流` 共用同一个侧边栏插件、同一套登录态，但服务的是另一条操作链路：从 X 页面抽取正文，调用服务端能力做查重和快讯生成，再把结果回送到侧边栏。

## 定位

`快讯生成` 是值班编辑在浏览 X 时的就地生成工具。

它不替代控制台，也不替代现有自动流水线；它只是把“看到一条 X 内容后，快速查重并生成快讯”的高频动作，搬到浏览器侧边栏里完成。

第一版目标：

- 在 `x.com / twitter.com` 页面就地注入 Odaily 按钮。
- 点击后抽取当前帖文正文，并自动跳转到插件一级模块 `快讯生成`。
- 侧边栏上半区展示抽取到的原文，下半区展示查重结果或生成结果。
- 提供三个固定动作：`AI查重`、`快速生成`、`生成快讯`。
- 生成结果支持 `复制`，复制后自动切回一级模块 `信息流`。

第一版非目标：

- 不直接发布快讯。
- 不替代控制台编辑器。
- 不在插件里直接改 Prompt。
- 不在插件里承接整条 `judge -> search -> write -> format_publish -> publish` 自动流水线。

## 一级导航位置

插件顶部一级导航改为两个并列入口：

- `信息流`
- `快讯生成`

其中：

- `信息流` 继续负责通知查看、反馈和留痕。
- `快讯生成` 负责 X 内容抽取、查重和快讯草稿生成。

当用户在 X 页面点击注入按钮时，侧边栏应自动切到 `快讯生成`。

## 参考实现来源

当前仓库 `Reference/dist/` 下已有另一个 Chrome 插件成品，可直接借鉴其“X 页面注入按钮 -> 打开 side panel -> 把抽取结果塞到侧边栏”的机制。

已确认的参考点：

- [content-script.js](/mnt/d/Coding/OdAIly/Reference/dist/content-script.js:1)
  - 监听 X/Twitter 页面 DOM 变化。
  - 在 `article[data-testid="tweet"]` 操作区插入自定义按钮。
  - 点击后抽取正文并通过 `chrome.runtime.sendMessage` 发给后台。
- [background.js](/mnt/d/Coding/OdAIly/Reference/dist/background.js:1)
  - 收到消息后写入 `chrome.storage.local`。
  - 直接打开 `sidePanel`。

第一版建议沿用这个思路，但把数据结构、消息类型和侧边栏页面切换逻辑统一到当前 `extension/` 插件内。

## X 页面注入按钮

### 按钮样式

参考用户提供的 F12 信息，第一版直接复用同款视觉：

- 元素：`img.custom-button`
- 图标：`https://oss.odaily.top/image/logo-primary.svg`
- 宽高：`18.75px`
- 外边距：`auto 10px`
- 圆角：`50%`
- 鼠标：`pointer`

按钮插入位置：

- X 帖文卡片 `article[data-testid="tweet"]`
- 底部操作按钮所在 `div[role="group"]`

要求：

- 每条帖文只插入一次，不重复挂按钮。
- DOM 变化后自动补插，适配 X 时间线懒加载。
- 后续如果 X DOM 结构改动，以“帖文操作区追加 Odaily 图标”为唯一交互目标，不要求死守某个 class 名。

### 点击后的动作

用户点击按钮后，插件执行四步：

1. 抽取当前 X 帖文的正文与基础元数据。
2. 写入本地暂存状态。
3. 打开侧边栏。
4. 自动切换到一级模块 `快讯生成`。

建议抽取字段：

- `source_type = x_post`
- `platform = x`
- `author_display_name`
- `author_handle`
- `post_text`
- `post_url`
- `posted_at`
- `post_id`

第一版最低要求是必须拿到 `post_text`；其余字段拿得到就一并传给后端，方便后续提示词或定位。

当前实现就是按这组字段发给服务端轻接口。

作者名称标准化约定：

- 插件前端继续上传页面抽取到的 `author_display_name` 和 `author_handle`
- 服务端不直接把页面抽取到的作者显示名作为最终发稿名
- 生成前会按 `author_handle` 去匹配 `x_capture_accounts` 中的 `写作名`
- 最终作者名 `effective_author_name` 计算顺序为：`写作名 -> author_display_name -> author_handle`

## 侧边栏界面

`快讯生成` 一级模块采用上下两段，不再额外做复杂工作台。

### 上半区

上半区固定展示：

- 抽取到的原文
- 来源信息（作者、链接、时间，能拿到就展示）
- 三个固定按钮：
  - `AI查重`
  - `快速生成`
  - `生成快讯`

交互要求：

- 原文默认完整展示，必要时允许滚动，不做只展示一两行的省略。
- 用户在 `快讯生成` 页面内再次点击 X 页面按钮时，上半区内容立即替换成最新抽取内容。
- 在没有抽取内容前，按钮禁用或弱化显示。
- `快速生成` 与 `生成快讯` 是悬浮菜单入口，本身不直接触发请求；鼠标悬停或键盘聚焦后，在按钮下方竖排展示 `常规 / 融资 / 链上` 三个选项，只有选项按钮会触发生成。
- 悬浮菜单在父按钮与选项之间保留可 hover 的缓冲区，避免鼠标下移时菜单过早消失。

### 下半区

下半区是结果区，按最后一次操作展示结果：

- 点击 `AI查重`：显示搜索者返回结果。
- 点击 `快速生成` 或 `生成快讯` 下方的类型选项：显示编写者1、编写者2最终产出的快讯。

第一版不强求把“查重结果”和“生成结果”同时并排展示，按单结果区切换即可；但 `快速生成` 和 `生成快讯` 的结果优先级更高，生成成功后应占据下半区主视图。

## AI查重

### 业务边界

`AI查重` 调用的就是现有 `搜索者` 能力。

这里不新增一套新的查重逻辑，不重新造 Prompt，也不在插件前端做相似度判断。插件只负责：

- 把抽取到的 X 原文发给 Linux 服务器。
- 让服务端调用现有 `搜索者` 流程。
- 把服务端返回的查重结果展示在侧边栏里。

### 服务端约定

当前实现已新增一个面向插件的轻接口：

- `POST /plugin/news-gen/search`

请求体建议包含：

- 当前插件用户身份信息
- 来源类型与来源链接
- X 原文正文
- 作者和时间等补充字段

服务端内部职责：

- 复用现有 `搜索者` 的查重能力和输出口径。
- 不要求把这次操作正式写成一条待发布任务。
- 返回适合插件直接展示的结果摘要。

返回结果至少应包含：

- 是否判定为重复/同事件
- 相关历史候选列表
- 每条候选的标题、时间、链接
- 搜索者给出的简短结论或说明

### 前端展示

查重结果在下半区展示，建议拆成：

- 顶部结论区：`疑似重复` / `未发现明显重复`
- 候选列表区：展示 Odaily 历史项
- 说明区：展示搜索者摘要

第一版只做查看，不需要“接受/拒绝”这类反馈按钮。

## 生成快讯

### 业务边界

用户点击 `快速生成` 或 `生成快讯` 下方的 `常规 / 融资 / 链上` 选项后，服务端只走：

- `编写者1`
- `编写者2`

这里不强制再跑一次搜索者，也不调用判断者，不直接进入发布者，不接入发布后异步 `审核者`。

也就是说，`快讯生成` 更接近“拿原文快速产出一版 Odaily 格式快讯”，不是“直接发稿”。

### 服务端约定

当前实现已新增第二个面向插件的轻接口：

- `POST /plugin/news-gen/generate`

并新增快速生成接口：

- `POST /plugin/news-gen/quick-generate`

请求体建议包含：

- 当前插件用户身份信息
- 来源类型与来源链接
- X 原文正文
- 作者、时间、post id 等补充字段
- `news_type`：必填，取值为 `regular`、`funding` 或 `onchain`，分别对应 `常规`、`融资`、`链上`

服务端内部职责：

- 使用插件前端传入的 `news_type` 直接选择常规、融资或链上写作模板。
- 先基于 `author_handle` 解析 `effective_author_name`，规则为 `写作名 -> author_display_name -> author_handle`。
- 复用现有 `编写者1` Prompt 与生成逻辑。
- 继续复用 `编写者2` 的格式化逻辑，产出最终定稿。
- 返回最终 `title / content`，供插件展示和复制。

模型边界：

- `生成快讯` 继续使用现有 `X_PROCESS_WRITER_MODEL` 与 `X_PROCESS_WRITER_REASONING_EFFORT` 配置，默认是 `gpt-5.5` + `medium`。
- `快速生成` 只在文案生成调用中固定使用 `gpt-5.4-mini` + `low`。
- 自动化采集、编写、发布主链路不受 `快速生成` 影响。

当前实现不把这次生成结果写入 `tasks` 主表，也不推进到 `publisher_pending`；而是单独写入 `editor_plugin_generation_logs` 做轻量留痕。

### 前端展示

下半区生成结果建议固定为：

- 标题
- 最终快讯正文
- `复制` 按钮

如果生成失败，则在结果区显示失败原因，不要吞错误。

## 复制与回跳

生成快讯结果区提供一个固定 `复制` 按钮。

点击后执行三件事：

1. 把最终快讯正文复制到剪贴板。
2. 记录一次本地成功复制状态，避免用户不确定是否复制成功。
3. 侧边栏一级导航自动切回 `信息流`。

这里的“切回信息流”是工作流设计的一部分：编辑复制完稿子后，默认回到值班主视图，继续盯通知。

第一版不要求复制标题；如果后续编辑反馈需要标题一并带走，再增加“复制标题+正文”模式。

## 前端状态流

建议新增一组插件内消息与本地状态：

- `NEWS_GEN_FILL_SOURCE`
  - 由 content script 发给 background
  - 表示已抽取一条 X 正文
- `activeTopTab = news_gen`
  - 由 background 或 sidepanel 设置
  - 控制侧边栏一级导航切换到 `快讯生成`
- `newsGenDraft`
  - 存在 `chrome.storage.local`
  - 保存最近一次抽取的原文和来源元数据
- `newsGenResult`
- 仅保存最近一次 `AI查重`、`快速生成` 或 `生成快讯` 的结果

这样即使 side panel 已经打开、或中途被关闭重开，也能恢复最近一次抽取内容。

当前实现实际使用：

- content script -> background 消息类型：`ODAILY_NEWS_GEN_CAPTURED`
- 本地存储键：`activeTopTab`、`newsGenDraft`、`newsGenResult`

## 服务端边界

`快讯生成` 不是浏览器本地 AI 功能。

服务端执行边界明确为：

- 内容抽取：浏览器插件完成
- 查重与写作：Linux 服务器完成
- 结果展示与复制：浏览器插件完成

服务端应尽量复用现有模块，而不是为插件单独再写一套搜索和写作 Prompt：

- `AI查重` -> 复用 `搜索者`
- `快速生成` -> 由编辑手动选择 `常规 / 融资 / 链上`，复用 `编写者1`、`编写者2`，文案生成模型固定为 `gpt-5.4-mini` + `low`
- `生成快讯` -> 由编辑手动选择 `常规 / 融资 / 链上`，复用 `编写者1`、`编写者2`

当前服务端入口命令：

```bash
./tools/dev python backend/src/main.py editor-plugin-api-server --host 127.0.0.1 --port 8765
```

生产环境约定：

- 插件安装包内置的 `快讯生成` 服务地址为 `http://47.76.243.147:8765`。
- Linux 服务器通过 `odaily-editor-plugin-api.service` 运行该接口，并绑定 `0.0.0.0:8765` 供插件直接访问。

当前实现会在服务端启动时：

- 校验 Supabase JWT
- 校验 `editor_plugin_users` 白名单
- 初始化插件表结构
- 初始化 `x_processing` schema 并补齐 Prompt 模板
- 连接 PostgreSQL 时默认使用 `POSTGRES_CONNECT_TIMEOUT_SECONDS=10`，避免服务启动时无上限卡住

前端请求失败处理：

- `AI查重`、`快速生成` 和 `生成快讯` 共用统一结果区错误提示。
- 插件端为这三个请求加了本地超时控制；超时后直接提示 `请求超时，请稍后重试`，避免侧边栏长时间无响应。
- JWT 过期、白名单拒绝、参数缺失、服务端异常等情况，都按统一错误结构回传到结果区。

## 鉴权

第一版建议复用当前插件已有登录态，不新增独立账号体系。

推荐方式：

- 插件继续使用现有 `Supabase Auth` 登录态。
- 调用 Linux 服务器接口时，带上当前用户会话标识或可校验令牌。
- 服务器只放行插件白名单用户，不因为拿到了插件安装包就默认可用。

原则上，`快讯生成` 与 `信息流` 共用同一套插件白名单。

## 与信息流模块的关系

- `信息流` 是值班常驻主视图。
- `快讯生成` 是从 X 页面触发的就地生产工具。
- 两者共用同一个 side panel 和登录态，但不是同一条数据链路。

边界上：

- `信息流` 读的是 Telegram 同源通知与反馈状态。
- `快讯生成` 读的是 X 页面抽取内容，以及 Linux 服务器返回的查重/生成结果。
