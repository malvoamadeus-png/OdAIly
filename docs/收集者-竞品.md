# 收集者-竞品

## 职责

收集者-竞品负责：

- 抓取 BlockBeats、PANews、金色财经快讯
- 抓取 Odaily 快讯作为参考语料
- 将竞品快讯写入 `tasks`
- 为新竞品快讯提交本地流水线 `write_flow` job
- 将 Odaily 参考语料写入 `odaily_reference_items`
- 将竞品与 Odaily 快讯写入 `newsflash_*` 事件复盘链路

当前实现仍使用 `competitor_monitor` 包和 `competitor-monitor-worker` 命令名，文档统一称“收集者-竞品”。

## 存储边界

### Supabase / Postgres

仍然是权威业务存储，负责：

- `tasks`
- `odaily_reference_items`
- `newsflash_items`
- `newsflash_events`
- `newsflash_event_sources`
- 控制台可见的事件摘要、收藏、备注

### 本地 SQLite

本轮新增竞品运行时库：

- `data/processed/competitor_monitor/competitor_monitor.sqlite`

当前本地库承载：

- 最近窗口的 `newsflash_items` 镜像
- 已知 `event_sources` 归属镜像
- recent window 预热标记

本地库缺失或损坏时：

- worker 会从 Supabase 拉取最近窗口重建
- 重建完成后恢复到 local-first 模式

## 处理流程

### 1. 抓取与清洗

- 拉取竞品快讯与 Odaily 快讯
- 做标题/正文清洗
- 对竞品快讯应用排除词
- 金色财经只在 API 返回外部原文链接时才保存 `source_url`，优先取 `link`，其次取外部 `jump_url`
- 如果金色财经只返回站内 `jinse` 页面，或没有外部原文链接，则 `source_url` 留空，避免把竞品站内页透出到我方后台

竞品排除词命中后：

- 不进入 `tasks`
- Odaily 参考语料不使用这套排除词，仍会进入 `odaily_reference_items`
- 不进入 `newsflash_*`

### 2. 写入远端业务表并提交本地流水线

- 竞品快讯：按时效规则决定是否写入 `tasks`
- Odaily 快讯：写入 `odaily_reference_items`
- 竞品和 Odaily：都写入 `newsflash_items`
- Odaily 快讯只做标题/正文清洗，不走 `competitor_filter_keywords` 排除；这样我方已发布快讯不会因为 `突破`、`跌破`、`爆仓` 等宽词被误杀，导致搜索者查重缺参考语料。
- Odaily 参考语料的正文清洗只应去掉 UI 噪音、媒体前缀和固定栏目包装，不应删除会影响语义判断的正文词语，例如 `利好`、`利空` 等内容词。
- Odaily 参考语料若原始 HTML 已用 `<p>`、`<br>` 等标签分段，写入 `odaily_reference_items.content` 时必须保留这些语义段落边界，但统一压成“单换行分段”；不能把相邻段落直接压成一行，也不能保留空白段落或双空行，否则会把 `交易对` 与下一行 `交易开放时间：` 粘连，或在后台正文里留下无效空行。
- 新竞品快讯入库后提交 `write_flow` job

竞品快讯进入本地流水线后的执行顺序为：

```text
search -> judge_crypto -> write -> format_publish -> publish
```

也就是说，竞品先查重，再判断，再生成和发布。

当前补充约束：

- 来自 `blockbeats / panews / jinse` 的竞品内容，如果在判断阶段被识别为 `AI 泛科技 / 半导体` 且缺少明确 Crypto 语境，会直接按 `non_crypto_ai` 丢弃。
- 这类内容会在 `judge_crypto` 阶段结束，不进入写作、发布者和前台发布链路。

本地流水线 job 唯一键是 `source + source_item_id`。入队失败时，本轮会记录失败并等待下轮重试，不把这条竞品快讯视为已经成功交给发布链路。

### 3. 回填本地工作集

远端写入完成后，worker 会把事件聚合需要的最近语料镜像到本地 SQLite。

额外约定：

- Odaily 参考语料保存成功后，也会同步镜像到共享 `searcher.sqlite`
- 这样搜索者不需要再为这批热语料反复回源 Supabase

### 4. 事件聚合

事件聚合当前是“本地最近窗口镜像优先”：

- `list_recent_event_sources(...)` 默认从本地 SQLite 读取
- 如果本地 recent window 尚未建立，先从 Supabase 拉一次最近窗口，再落本地
- 针对当前 item 的既有归属，如果本地缺失，可按 item_id 回源补齐，不会整窗重拉

聚合完成后：

- 事件归属和事件摘要仍写回 Supabase
- 本地镜像同步更新归属结果

事件聚合里的 AI 复核只用于辅助判断相似但不确定的两条快讯是否属于同一事件。如果 AI 网关返回空响应、非 JSON 或临时异常，本轮会把对应候选对视为“不合并”，继续完成其余事件归属。

如果事件聚合外围仍出现临时异常，收集者-竞品会记录日志，并在 heartbeat metadata 的 `event_error` 中保留错误；但这类事件复盘辅助链路失败不会让整轮抓取失败，也不会阻断成功心跳。

常驻 worker 会捕获单轮外围异常并在下一轮继续运行，避免一次 Supabase、事件聚合或本地流水线提交波动导致收集者-竞品进程退出。

## 与搜索者的关系

收集者-竞品维护两类会被搜索者复用的数据：

- `odaily_reference_items`：保存 Odaily 已发布快讯的 Supabase 档案备份；`x_processing` 和 `external_media_alert` 查重实时刷新优先调用 Odaily 官方快讯 API，仅在 API 不可用时回退读取该表
- `newsflash_*`：供控制台事件复盘展示

当前没有改动 Supabase 表结构或前端读取接口。

## 时效规则

竞品写入 `tasks` 时仍受发布链路时效窗限制：

- 默认由 `PROCESSING_FRESHNESS_WINDOW_SECONDS=1200` 控制
- 超过窗口的竞品快讯不进入 AI 发布主链路

但事件复盘链路不使用这个 10 分钟窗口裁切：

- 旧快讯仍可进入 `newsflash_*`
- 用于服务重启后的最近窗口补齐和事件连续性维护

## 心跳

收集者-竞品会写 `pipeline_worker_heartbeats`，但已改为降频策略：

- 稳定态最多 60 秒一条
- 状态变更立即写
- 失败立即写

python backend\src\main.py competitor-init-db
