# 收集者-竞品

## 职责

收集者-竞品负责持续采集竞品快讯和 Odaily 自身快讯，完成基础清洗、排除词过滤、任务入库和事件复盘入库。

竞品快讯用于发现 Odaily 可能尚未发布的快讯线索。Odaily 自身快讯不是竞品来源，只作为本方 reference 语料，用于搜索者查重和事件复盘。

当前代码实现仍使用 `competitor_monitor` 包和 `competitor-monitor-worker` 命令，本轮只统一文档口径，不重命名代码。

## 来源范围

竞品来源：

- BlockBeats。
- PANews。
- 金色财经：使用官方 `https://api.coinmeta.info/live/list`，默认 `limit=50`，读取 `list[].lives[]`。

本方 reference 来源：

- Odaily 快讯接口。

后续可增加其他 Crypto 新闻媒体。新增来源需要明确来源侧 ID、发布时间、标题、正文、原始链接和原始 payload 的获取方式。

金色财经字段契约：

- `source_item_id` 使用 `lives[].id`。
- `published_at` 使用 `lives[].created_at` Unix 秒时间戳。
- 标题从 `lives[].content` 开头的 `【...】` 提取。
- 正文使用去掉标题和“金色财经报道”等固定前缀后的 `lives[].content`。
- `source_url` 使用 `https://www.jinse2.com/lives/{id}.html`。

## 入库职责

收集者-竞品负责：

- 调用竞品 API 或页面抓取逻辑。
- 调用 Odaily 快讯接口获取本方 reference。
- 提取标题、正文、发布时间、来源链接、来源侧 ID 和原始 payload。
- 清洗媒体固定前缀和页面噪声。
- 确保同一 `source + source_item_id` 不重复写入。
- 未命中排除词且未过期的竞品快讯写入 `tasks`。
- 未命中排除词的竞品和 Odaily 快讯写入 `newsflash_*` 事件复盘表。

收集者-竞品不负责：

- 判断是否已被 Odaily 报道。
- 判断是否值得发布。
- 判断快讯类型。
- 生成快讯文本。
- 推送后台。

这些职责分别由搜索者、判断者、编写者和发布层承担。

## 排除词

收集者-竞品在写入正式编写流程和事件复盘流程前，读取控制台维护的 `competitor_filter_keywords` 启用词表。数据库表名沿用旧名，控制台和文档统一称为“排除词”。

默认启用词：

- `跌破`
- `突破`
- `爆仓`
- `Bitget`

匹配规则：

- 匹配来源：`odaily`、`blockbeats`、`panews`、`jinse`。
- 不匹配 X 来源。
- 匹配范围是清洗后的标题 + 正文。
- 中文按原文子串匹配；英文大小写不敏感。

命中后的行为：

- 不写入 `tasks`。
- 不写入 `odaily_reference_items`。
- 不进入 `newsflash_items/newsflash_events/newsflash_event_sources`。
- 不进入搜索者、判断者、编写者和推送流程。
- 只在服务日志中记录来源、来源侧 ID、命中词和标题。

## 流程位置

```text
竞品/Odaily 快讯采集
  -> 排除词命中则跳过并写服务日志
  -> 未命中排除词时写入事件复盘表
  -> 未命中排除词且 10 分钟内的竞品写入 tasks
  -> 搜索者判断是否重复
  -> 未重复时进入判断者
  -> 判断者决定是否进入后续快讯流水线
```

Odaily 自身快讯只进入 reference 和事件复盘，不进入 `tasks` 编写队列。

## 发布流水线时效过滤

收集者-竞品拆开两条写入路径：

- `newsflash_items/newsflash_events/newsflash_event_sources`：不做 10 分钟时效过滤，允许服务重启后补齐事件页空白。
- `tasks`：写入前按来源原发布时间过滤，过期竞品快讯不进入 AI 处理、后台推送或 Telegram。

时效规则：

- 默认窗口由 `PROCESSING_FRESHNESS_WINDOW_SECONDS=600` 控制。
- 判断基准使用竞品来源原始发布时间 `published_at/created_at`，不使用任务入库时间。
- 原发布时间缺失或解析失败时，不写入 `tasks`。
- Odaily reference 保留用于查重和事件复盘，是否过期不影响事件页归纳。
- 过滤日志记录来源、来源侧 ID、原发布时间、延迟秒数和动作 `skip_tasks_keep_newsflash`。

## 事件复盘入库

收集者-竞品同时维护长期事件复盘数据：

- `newsflash_items`：每条原始快讯。
- `newsflash_events`：事件聚合记录。
- `newsflash_event_sources`：事件与来源快讯的关系。
- `newsflash_event_favorites`：控制台收藏。
- `newsflash_event_notes`、`newsflash_item_notes`：控制台备注。

事件归属依赖 embedding 和相似度判断，embedding 保存在服务器本地 SQLite 缓存，不写入 Supabase。Supabase 只保存业务数据、事件关系、收藏和备注。

事件归属有运行边界：

- 每轮事件归属会输出阶段日志，例如 upsert、查询已有归属、embedding、匹配、写入和更新摘要。
- 已经存在 `newsflash_event_sources` 归属关系的 item 再次被抓到时，只刷新相关事件摘要，不重复做 embedding 和归属判断。
- `COMPETITOR_EVENT_ASSIGNMENT_TIMEOUT_SECONDS` 控制单轮事件归属硬超时，默认 240 秒。
- 事件归属超时或失败时，本轮仍继续按时效规则写入 `tasks` 和 Odaily reference，并通过 worker 心跳记录 `event_aggregator` 失败。

事件归属规则和控制台展示详见 `控制台-事件复盘.md`。

## 竞品任务字段

竞品任务写入 `tasks` 时，至少保留：

- `source`：竞品来源，例如 `blockbeats`、`panews`、`jinse`。
- `source_item_id`：竞品来源侧唯一 ID。
- `source_url`：原始链接。
- `title`：清洗后的标题。
- `content`：清洗后的正文。
- `published_at`：竞品发布时间。
- `raw_payload`：原始响应中的关键字段。
- `status`：初始状态。

具体表结构以 `tasks` 表和当前 schema 为准。

## 运行命令

初始化竞品、搜索和事件复盘相关表：

```powershell
python backend\src\main.py competitor-init-db
```

运行一次收集：

```powershell
python backend\src\main.py competitor-monitor-worker --once
```

常驻运行：

```powershell
python backend\src\main.py competitor-monitor-worker
```
