# 控制台

## 职责

控制台负责维护人工可配置的数据和复盘视图，不承载 worker 内部运行逻辑。

当前控制台主体是 Vite 前端，绝大多数模块直接连接 Supabase。发布者规则配置、耗时看板和 BlockBeats key 状态例外：前端通过 HTTPS 控制台接口访问服务器本地配置或快照。

## 访问控制

- 控制台使用 Supabase Auth 的邮箱密码登录。
- 前端仍使用 `VITE_SUPABASE_URL` 和 `VITE_SUPABASE_ANON_KEY` 初始化客户端，但控制台业务表不再对 `anon` 开放。
- 只有 `authenticated` 会话且邮箱存在于 `console_admins` 白名单中的用户，才能读取或修改控制台数据。
- `tasks` 是多来源共用档案表，控制台白名单用户需要能读取所有来源；前端页面再按模块自行筛选来源，不在 RLS 层限制为 `x`。
- 控制台不开放前端自助注册；管理员账号由 Supabase Auth 后台创建，再通过本地 CLI 加入白名单。

## 模块索引

- `信源管理`：控制台一级标签，内含 `X`、`Crypto信源`、`AI信源`、`混合信源`、`律动key` 五个子页；`X` 子页细节见 `控制台-X抓取.md`，站点子页细节见 `控制台-非主流媒体抓取.md`，`律动key` 子页细节见 `控制台-律动key.md`。
- `信息周期监控`：控制台一级标签，按最近任务查看 X、Crypto信源、AI信源、历史媒体源、标题提醒、竞品和金十的入库与处理状态；任务到达发布者后展示发布者写入的业务判断理由。
- `耗时看板`：控制台一级标签，默认合并展示主快讯制作链路在 `24h / 72h / 7d` 窗口内的耗时聚合；页面通过控制台后端读取 Linux 本地 SQLite 快照，不由浏览器实时查询 Supabase。细节见 `控制台-耗时看板.md`。
- `控制台-Prompt.md`：Prompt 模板、版本、删除、发布/回滚和特色模式开关。
- `控制台-金十.md`：金十链路总开关、轮询频率、接口配置、最近运行状态和最近入库任务。
- `控制台-流程展示.md`：纯展示页，总览主工作流、节点职责、分流关系、搜索者查重口径、发布后审核者位置和模型 / 思考等级，不写入配置、不推进任务。
- `控制台-竞品配置.md`：竞品排除词维护。
- `控制台-事件复盘.md`：事件列表、收藏、备注和详情。
- `控制台-发布者.md`：发布者规则编辑页，维护 `常规` / `AI信源` 两块规则和块级启停；旧总开关、旧渠道开关和旧北京时间窗不再参与发布者运行时判定。

后续如果控制台新增较大的功能模块，应继续按子文档拆分，避免单个控制台文档过大。

## 数据边界

控制台可以写入配置类和复盘类表：

- `x_capture_settings`
- `x_capture_accounts`
- `tasks`
- `x_task_pipeline`
- `non_mainstream_media_settings`
- `non_mainstream_media_sources`
- `prompt_templates`
- `prompt_template_versions`
- `publisher_settings`
- `publisher_channels`
- `jin10_settings`
- `jin10_seen_items`
- `competitor_filter_keywords`
- `newsflash_event_favorites`
- `newsflash_event_notes`
- `newsflash_item_notes`
- `pipeline_timing_snapshots`

控制台不直接推进任务状态，不替代 worker 的锁、重试和失败处理。

其中 `x_capture_accounts` 除抓取频率、启停等账号规则外，还维护 X 写作链路使用的账号级 `写作名`。该字段由自动化 X 主链路与插件 `快讯生成` 共用。

`信息周期监控` 只读 `tasks` 与 `x_task_pipeline`，默认按每个来源各自读取最近有限条任务，再在前端合并排序；这样 X 高频入库不会挤掉 Crypto信源、AI信源、历史媒体源、标题提醒、竞品或金十记录，也不分页扫描全表。展示和搜索优先使用流水线最终标题 / 正文，缺失时回退到入库原始标题 / 正文。发布者说明优先来自 `publisher_output.reason`，这是发布者模型或保守挂后台分支写给编辑看的业务理由。

`耗时看板` 的前端不直接读取 `tasks`、`x_task_pipeline` 或 `pipeline_timing_snapshots`。`editor-plugin-api-server` 每小时用必要轻量字段计算聚合结果，写入 `data/runtime/pipeline_timing.sqlite`；Supabase 表 `pipeline_timing_snapshots` 只保存聚合 JSON 作为备份留档，归档失败只记录日志，不影响本地看板读取。

`信源管理 / 律动key` 子页的 BlockBeats API Key 与额度状态不进入 Supabase。前端通过 `editor-plugin-api-server` 的 `/console/blockbeats-key/get` 和 `/console/blockbeats-key/save` 读写 Linux 本地 `data/config/blockbeats_key.json`；路径可由 `BLOCKBEATS_KEY_CONFIG_PATH` 覆盖。接口继续复用控制台 Supabase Auth 会话与 `console_admins` 白名单。

## 配置生效

配置变更的基本要求：

- worker 启动时主动加载当前配置。
- 控制台发布或保存配置后，由数据库通知 worker 刷新本地缓存。
- worker 不依赖高频轮询发现配置变化。

不同模块的通知机制见对应控制台子文档。

## 控制台读取策略

- 控制台页面默认按当前页首次进入时加载该页所需数据，不在后台统一预加载所有模块。
- 顶部刷新按钮只刷新当前页。
- 默认不做前端自动轮询；如果后续某个模块需要更高时效，应在对应子文档单独说明刷新频率与原因。
- 这条约束用于控制 Supabase 读取流量，避免控制台在无人关注时持续刷库。
- `耗时看板` 遵循同一策略：进入页面或点击刷新时只读取本地 API 当前快照，不触发实时重算；快照由后端小时级线程维护。

## 本地开发

```powershell
cd frontend
npm install
npm run dev
```

前端需要：

```text
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
VITE_CONSOLE_API_BASE_URL=
```

生产环境的 `VITE_CONSOLE_API_BASE_URL` 应指向发布者控制接口的 HTTPS 地址，例如：

```text
https://47.76.243.147.sslip.io
```

首次启用控制台登录时，还需要：

1. 在 Supabase Dashboard 的 Authentication 中创建管理员用户，邮箱与白名单保持一致。
2. 在 Supabase Dashboard 中把 Site URL / Redirect URL 指向控制台正式域名，例如 `https://od-a-ily.vercel.app`。
3. 在本地用数据库直连权限执行：

```powershell
python backend\src\main.py console-grant-admin --email your-admin@example.com
```

常用管理员白名单命令：

```powershell
python backend\src\main.py console-list-admins
python backend\src\main.py console-revoke-admin --email your-admin@example.com
```

## 巨鲸控制台

- 控制台新增一级标签 `巨鲸`，页面内包含 `链上` 与 `Hyperliquid`。
- `链上` 负责维护 EVM 地址和自定义标签，支持启用、停用、删除，并展示 worker 写回的链状态与最近播报。
- `Hyperliquid` 负责维护 Hyperliquid 地址和自定义标签，支持启用、停用、删除，并展示最近轮询状态与播报历史。
- `Hyperliquid` 还提供三个人工可修改参数：`单次门槛`、`聚合门槛`、`聚合时长`。
- 新增控制台数据表：`whale_watch_addresses`、`whale_watch_chain_states`、`whale_watch_activities`、`whale_watch_hyperliquid_settings`、`whale_watch_hyperliquid_addresses`、`whale_watch_hyperliquid_states`、`whale_watch_hyperliquid_activities`。
- 前端只写地址配置和 Hyperliquid 阈值配置；链选择、轮询、活动识别、滚动聚合窗口计算和 Telegram 发送分别由 `whale-watch-worker` 与 `whale-watch-hyperliquid-worker` 负责。
- 巨鲸地址配置会记录 `created_by`、`updated_by` 作为审计字段；链上地址表在数据库侧额外限制总量上限 `500`，超过后会拒绝新地址插入，避免异常批量写入继续扩大。
- 模块细节见 `docs/巨鲸.md`。
