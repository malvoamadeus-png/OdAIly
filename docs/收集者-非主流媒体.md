# 收集者-非主流媒体

## 说明

当前实现包名仍为 `non_mainstream_media`，但本文只维护外媒业务细节。AI信源也暂时复用该实现包，业务细节单独见 `收集者-AI信源.md`。

它负责：

- 维护站点注册表
- 发现新文章或新标题
- 按 `discovery_mode` 决定走站点直抓还是 Telegram 优先发现
- 按 `source_group` 区分外媒与 AI信源；本文只展开 `external_media`
- 按 `pipeline_mode` 决定进入编写链路还是标题提醒链路
- 完成站内去重与 `tasks` 入库
- 入库成功后提交本地流水线 job

worker 启动后会加载当前配置，之后不保留专门的数据库监听连接；运行中按固定窗口轮询重载配置，默认最多 5 分钟生效一次，不需要重启服务。

## 站点与模式

### 外媒 `write_flow`

- `a16z_crypto_posts`
- `forbes_digital_assets`
- `hk01_virtual_assets`
- `coindesk`
- `cointelegraph`
- `decrypt`
- `tether_news`

这些站点会抓标题、来源链接、正文、作者等完整文章信息，并写入：

- `tasks.source = 'non_mainstream_media'`
- `tasks.status = 'pending'`

随后提交本地流水线 `write_flow` job，由 `local_pipeline` 执行 `search -> judge_crypto -> write -> format_publish -> publish`。

## 发现模式

- `discovery_mode = direct`
  - 由现有 `non_mainstream_media` worker 直接访问站点列表页、RSS 或 sitemap 做发现
- `discovery_mode = telegram_primary_direct_fallback`
  - 由新增 `non_mainstream_media_telegram` worker 先轮询 `https://t.me/PhoenixNewsEN`
  - 当前只用于：
    - `coindesk`
    - `cointelegraph`
    - `decrypt`
  - Telegram 命中时会直接提取原站详情链接，立即尝试抓正文并入队 `write_flow`
  - 同时原有 `non_mainstream_media` 站点直抓发现仍继续运行，作为漏检 / 晚到兜底，不会因为 Telegram worker 存在而停掉

## Telegram-first worker

- 组件名固定为 `non_mainstream_media_telegram`
- 独立命令：`backend/src/main.py telegram-discovery-worker`
- 独立 systemd：`deploy/odaily-telegram-discovery.service`
- 独立环境变量：
  - `TELEGRAM_DISCOVERY_API_ID`
  - `TELEGRAM_DISCOVERY_API_HASH`
  - `TELEGRAM_DISCOVERY_SESSION_PATH`
  - `TELEGRAM_DISCOVERY_PROXY`
  - `TELEGRAM_DISCOVERY_CHANNEL`
  - `TELEGRAM_DISCOVERY_POLL_INTERVAL_SECONDS`
  - `TELEGRAM_DISCOVERY_RETRY_DELAY_SECONDS`
  - `TELEGRAM_DISCOVERY_RETRY_MAX_ATTEMPTS`

### Telegram-first 处理语义

- 只处理：
  - `pipeline_mode = write_flow`
  - `discovery_mode = telegram_primary_direct_fallback`
  - `site_key in {coindesk, cointelegraph, decrypt}`
- 从 Telegram 消息的这些位置提取原站详情链接：
  - 明文 URL
  - 隐藏链接实体 `text_url`
  - URL 实体
  - web preview / media URL
- 提取不到原站 URL 时，不建重试，不 mark seen，直接放过，交给原站点直抓入口兜底
- Telegram 命中后正文抓取流程：
  - 立即尝试 1 次
  - 若失败且属于可重试错误，则按 `120 秒` 间隔再重试
  - 最大补抓 `3` 次
  - 总抓取次数上限为 `4` 次
- 可重试错误固定包括：
  - `404`
  - `429`
  - `5xx`
  - timeout / connection error
  - 正文 body 为空
- 重试队列只保存在 worker 内存中；worker 重启后待重试任务会丢失，这是当前有意保留的产品选择
- 去重键使用 `site_key + canonical/detail_url`
- Telegram 抓取失败时不 `mark_seen`
- Telegram 抓取成功并成功完成 `save_task(...)` / 本地流水线提交后，才 `mark_seen`

### Tether News 特例

- `tether_news` 的新文章发现优先使用公开 `WordPress REST API`：`https://tether.io/wp-json/wp/v2/posts?categories=3...`
- 单篇正文也优先通过 `wp-json/wp/v2/posts?slug=...&_embed=wp:term` 获取，不依赖 `/news/` 页面 DOM
- 如果生产 Linux 服务器访问 `tether.io` 原站遇到 Cloudflare `403 challenge`，收集器会自动 fallback 到 `https://r.jina.ai/http://...` 读取同一份官方列表 / 正文页面
- fallback 顺序是：官方 `wp-json` -> Jina 代理的 `wp-json` -> Jina 代理的 `/news/` 列表页；正文则是官方 `wp-json` 失败后切到 Jina 代理正文页
- 这样可以稳定拿到：
  - 标题
  - 正文 HTML
  - 发布时间
  - 分类 / 标签
  - 规范化文章链接
- 该站点接入 `write_flow`，和其他外媒全文一样直接生成 `tasks.source = 'non_mainstream_media'` 的待编写任务

### Cointelegraph 特例

- 新文章发现优先使用官方 `Google News sitemap`：`https://cointelegraph.com/sitemap/google-news.xml`
- 该源直接提供：
  - 标题
  - 详情链接
  - `news:publication_date`
- `https://cointelegraph.com/rss` 继续保留为 fallback，只在 sitemap 抓取失败或返回空列表时启用
- 发现层拿到的发布时间会透传到详情解析和 `tasks.published_at`
- 即使正文页缺少标准 `article:published_time`，也会优先沿用发现层时间，避免任务因缺少 `published_at` 被 freshness gate 直接过期

### Telegram 提前命中与正文可取性测试

- 仓库新增独立测试脚本 `tools/telegram_writeflow_probe.py`，用于验证 `write_flow` 站点在“Telegram 频道先提示、原发现链路稍后才看到”的情况下，是否已经可以直接抓到正文。
- 当前测试目标只覆盖：
  - `coindesk`
  - `cointelegraph`
  - `decrypt`
- 脚本不接入生产 `non_mainstream_media` worker，只复用现有 `fetch_article(...)` 解析器做单次探针。
- 测试逻辑是：
  - 从 `https://t.me/PhoenixNewsEN` 读取频道消息
  - 识别类似 `CoinDesk:` / `Cointelegraph:` / `Decrypt:` 的前缀
  - 尝试从同一条 Telegram 消息里提取目标站详情链接
  - 把提取出的链接直接组装成 `DiscoveredPage(detail_url=...)`
  - 调用现有正文抓取器验证是否能拿到 `title / content / published_at / canonical_url`
- 这项测试验证的是“正文抓取器是否支持 Telegram-first 直取详情页”，而当前生产实现已经把这条能力正式用于 `CoinDesk / Cointelegraph / Decrypt`。
- 推荐先跑历史回扫：
  - Windows PowerShell：`py tools\telegram_writeflow_probe.py --mode backscan --backscan-limit 200`
- 如果回扫里出现 `probe_success`，说明至少对该条频道消息，正文闭环已经可以直接跑通。

### 外媒 `alert_only`

- `ft_crypto`
- `wsj_business`
- `wsj_economy`
- `wsj_finance`
- `fortune_crypto`
- `the_block`

这些站点只保留标题/摘要级材料，并写入：

- `tasks.source = 'external_media_alert'`
- `tasks.status = 'pending'`

随后提交本地流水线 `alert_only` job，由 `local_pipeline` 执行 `domain_judge -> search -> notify`。

## 本地流水线交接

- `tasks` 继续保存原始标题、正文/摘要、来源链接、metadata 和最终状态。
- `write_flow` 结果继续写入 `x_task_pipeline`。
- `alert_only` 结果继续写入 `external_media_alert_pipeline`。
- 本地流水线 job 唯一键是 `source + source_item_id`，重复文章或重复标题不会重复排队。
- 收集器只在本地流水线入队成功后 mark seen。
- 如果入队失败，本轮记录错误，不 mark seen，下一轮继续重试。

## The Block 特例

- The Block 继续固定停留在 `alert_only`，只做标题提醒，不进入正文编写链路
- 新的标题发现优先级是：
  - 官方 `rss.xml`：`https://www.theblock.co/rss.xml`
  - `r.jina.ai` 代理后的官方 `rss.xml`
  - `Google News RSS` fallback：`https://news.google.com/rss/search?q=site:theblock.co&hl=en-US&gl=US&ceid=US:en`
- 这样做的原因是：Google News 对 The Block 存在明显收录延迟，可能导致标题提醒晚于原文发布时间数小时
- 服务器侧直连 `https://www.theblock.co/rss.xml` 仍可能遇到 Cloudflare / 403，但通过 `r.jina.ai` 代理读取官方 RSS 仍可稳定拿到：
  - 标题
  - 官方详情链接
  - `pubDate`
- The Block 正文抓取仍不可用，因此即使发现源切回官方 RSS，也只保留标题提醒用途
