# 判断者

## 职责

判断者指本地流水线复用的 `x_processing` 判断阶段。生产不再常驻启动分阶段 `x-process-worker` 服务，而是由 `local_pipeline` 在同一进程内按来源串联调用。

- `judge_crypto`：判断者-Crypto。
- `judge_ai`：判断者-AI。
- `judge_jin10`：判断者-金十。

旧 `judge` stage 仍保留为兼容入口，不作为生产常驻服务。

默认模型配置：

- `X_PROCESS_JUDGE_MODEL`：生产默认 `odaily-deepseek-review`，由本机 LiteLLM 路由到 DeepSeek 官方模型。
- `X_PROCESS_JUDGE_REASONING_EFFORT`：默认 `low`

判断者是固定使用 DeepSeek 的三个环节之一。生产环境中后端访问 `ODAILY_LLM_BASE_URL`，模型名只使用 LiteLLM 业务别名；真实 DeepSeek 模型名只在 LiteLLM 配置里维护。判断者支持独立覆盖模型接口，不影响编写者、发布者和搜索 AI 复核配置：

- `X_PROCESS_JUDGE_OPENAI_BASE_URL`：仅判断者使用的兼容接口 base URL；为空时沿用 `X_PROCESS_OPENAI_BASE_URL` / `ODAILY_LLM_BASE_URL`。
- `X_PROCESS_JUDGE_OPENAI_API_STYLE`：仅判断者使用的接口风格；为空时沿用 `X_PROCESS_OPENAI_API_STYLE`。
- `X_PROCESS_JUDGE_OPENAI_API_KEY`：仅判断者使用的 API key；未设置且判断者 base URL 或模型名包含 `deepseek` 时，会读取 `DEEPSEEK_API_KEY` / `DEEPSEEK_API` / `DeepSeek_API`。
- `X_PROCESS_JUDGE_OMIT_REASONING_EFFORT`：兼容不接收 reasoning 参数的接口，例如 DeepSeek 非思考模式。
- `X_PROCESS_JUDGE_CHAT_RESPONSE_FORMAT_MODE`：`chat_completions` 下的 JSON 输出模式，默认 `json_schema`；DeepSeek 非思考测试使用 `json_object`。
- `X_PROCESS_JUDGE_APPEND_JSON_SCHEMA_TO_PROMPT`：当接口只支持 JSON object 时，把 JSON Schema 约束追加到 prompt，降低缺字段风险。

运行交接：

- 新内容由收集器提交到本地 SQLite 队列，`local_pipeline` 读取本地 job 后直接调用判断、查重、编写、发布阶段。
- Supabase 的 `tasks` 和 `x_task_pipeline` 只保存输入、阶段结果、最终状态和失败原因，不再作为判断者与下一阶段之间的任务领取队列。
- 判断阶段成功结束时，`x_task_pipeline.judge_completed_at` 必须写入完成时间，用于后续阶段耗时复盘。

判断者-Crypto 负责处理：

- 未标记为 AI信源的 `tasks.source = 'x'`
- `tasks.source = 'non_mainstream_media'`

竞品来源 `tasks.source in ('blockbeats', 'panews', 'jinse')` 仍由本地流水线的 `judge_crypto` 节点执行，但在节点内按来源选择独立的竞品 Prompt 与 JSON Schema；不会复用 X 判断词，也不会增加第二次模型调用。

判断者-AI 负责处理：

- 当前账号配置 `x_capture_accounts.is_ai_source = true` 的 `tasks.source = 'x'`
- `tasks.source = 'ai_source'`

判断者-金十负责处理：

- `tasks.source = 'jin10'`
- Prompt key 固定为 `jin10_judge`
- 控制台显示名为“判断者-金十”

## Crypto信源全文判断

统一 Crypto信源全文现在全部通过 `tasks.source = 'non_mainstream_media'` 进入判断者-Crypto。

判断结果仍只允许：

- `route = 'non_mainstream_media'`
- `route = 'discard'`

这样可以保持现有流水线兼容，但后续写作模板使用 `mainstream_media_writer`。

## AI信源全文判断

AI信源全文通过 `tasks.source = 'ai_source'` 进入判断者-AI，当前站点包括 TheElec CHINA、ETNews Electronics、ETNews SW、ZDNet Korea Semiconductor、CTEE Semiconductor。

判断者-AI 会调用判断模型，不再确定性直通。

保留规则：

- `tasks.source = 'ai_source'` 保留后写入 `news_type = 'ai_source'`。
- X AI信源账号保留后统一写入 `news_type = 'regular'`，继续使用 X 常规写作模板。
- `news_type = 'ai_source'` 的 AI信源全文写作使用 `mainstream_media_writer`；标记为 AI信源的 X 账号仍使用 `x_regular_writer`。

丢弃规则：

- 丢弃统一写入 `discard_type = 'non_crypto_ai'`，不额外保存细分原因。
- 重要性不清、只是泛泛产业动态或营销表达时默认丢弃。
- 纯态度表达、吐槽、嘲讽、站队、情绪化评价或笼统指责默认丢弃；尤其是只说“某媒体/某机构又在散播 FUD”“叙事荒谬”“又在带节奏”这类内容，但没有新增事实、明确判断依据、可归纳的产业结论或对重要事件的实质性补充时，按 `non_crypto_ai` 丢弃。

直接屏蔽：

- 非 AI 相关的新品发布、消费电子、家电新品发布类。
- 人事、组织调整、招聘、活动、ESG 类。
- 终端品牌营销、渠道活动、促销、赞助类。
- 泛 AI、软件、机器人、非半导体科技类。
- 地缘、宏观、名人互动、行业花边类。

可以发布：

- 财报、营收、股价、评级类。
- 重大产能扩张、建厂、资本开支类。
- 关键技术突破、国产替代、先进工艺、材料、设备类。
- 量产、头部客户导入、关键供应链变化类。
- 产业政策、补贴、出口管制、监管类。
- 并购、融资、IPO、重大资本运作类。

补充边界：

- 如果内容带有强烈立场或批评语气，不因“语气冲”本身丢弃。
- 只有当它缺少新增事实、明确判断依据、可归纳的产业结论或对重要事件的实质性补充时，才作为低信息密度内容丢弃。

## 金十判断

金十快讯数量和品类都很多，因此金十判断者不使用固定代码分类，而是读取控制台维护的自然语言 Prompt 作为主题白名单。

运行规则：

- 输入为金十快讯标题、正文、发布时间和 metadata。
- Prompt 由 `prompt_templates.template_key = 'jin10_judge'` 管理。
- 模型输出只允许 `decision = 'publish'` 或 `decision = 'discard'`。
- `publish` 固定写入 `x_task_pipeline.news_type = 'regular'`，继续进入 `search -> write -> format_publish -> publish`。
- `discard` 固定写入 `discard_type = 'off_topic'`，任务结束为 `discarded`。
- 非 JSON、缺字段或输出了其他 decision 时，不做兜底放行，任务进入 `judge_failed`。

Prompt 可以直接写自然语言主题，例如“美伊冲突、霍尔木兹海峡通行、和平协议拟定与美伊谈判”。如果快讯表达的是“建立机制确保霍尔木兹通道保持开放”，模型应能按“美伊谈判 / 霍尔木兹海峡通行”理解为命中主题并放行。

## 来源专用判断词与丢弃规则

每条任务只调用一次判断模型。判断节点根据来源选择 Prompt 和 JSON Schema：

- X：只允许按 `pure_emotion`、`baseless_trading_call`、`daily_chatter` 丢弃。
- 竞品：允许按 `pure_emotion`、`baseless_trading_call`、`daily_chatter`、`non_crypto_ai`、`non_crypto_tradfi`、`marketing_activity`、`routine_company_news` 丢弃。
- Crypto信源全文：使用 Crypto信源专用判断词。
- AI信源与 X-AI 账号：使用 AI信源专用判断词。
- 金十：使用控制台维护的主题白名单 Prompt。

竞品判断按完整语言语义识别核心主体、动作与结果是否和 Crypto 有实质关联。零散出现区块链、代币、AI 或公司名不能单独构成放行理由；阿里巴巴企业 AI 平台这类非 Crypto AI 产品发布应输出 `non_crypto_ai`。代码不再维护 `COMPETITOR_NON_CRYPTO_AI_PATTERNS` 等硬编码预丢弃词表。

判断模型空响应、非 JSON、缺字段或结构解析失败时，任务进入现有重试 / `judge_failed` 路径，不再按关键词强行选择 `regular / onchain / funding` 放行。

每次成功判断会在 `x_task_pipeline.judge_output` 中保存代码生成的 `rule_set` 与 `rule_version`，用于确认任务实际使用的来源规则。

Chrome 插件 `快讯生成` 的文本生成接口不经过判断者；插件由编辑在前端手动选择 `常规 / 融资 / 链上` 模板。

## X 作者名输入

对 `tasks.source = 'x'` 的判断输入，判断者会优先使用统一作者名 `effective_author_name` 作为 prompt 里的作者字段。

- 优先使用当前 `x_capture_accounts` 里按账号维护的 `写作名`
- 没有 `写作名` 时回退到任务元数据里的原始 `author_display_name`
- 仍为空时回退到任务元数据里的 `author_username`

这样判断者与编写者1在 X 发言人名称上保持一致，不会出现前后口径不一致。

## 输出

- X 保留：`tasks.status = 'judged'`，`x_task_pipeline.news_type in ('regular', 'onchain', 'funding')`
- 竞品 / Crypto信源 / AI信源保留：`tasks.status = 'deduped'`
- Crypto信源保留：`x_task_pipeline.news_type = 'non_mainstream_media'`
- AI信源全文保留：`x_task_pipeline.news_type = 'ai_source'`
- X AI信源账号保留：`x_task_pipeline.news_type = 'regular'`
- 金十保留：`x_task_pipeline.news_type = 'regular'`
- 丢弃：`tasks.status = 'discarded'`
