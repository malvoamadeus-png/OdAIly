# 搜索者

## 搜索阶段

搜索者现在由本地流水线 `local_pipeline` 串联调用，不再作为独立常驻数据库 worker 领取 `tasks`。

### 自动快讯搜索

处理：

- `x`
- `blockbeats` / `panews` / `jinse`
- `non_mainstream_media`
- `ai_source`

统一 Crypto信源全文和 AI信源全文现在也走这条搜索链路，因此 CoinDesk、Cointelegraph、Decrypt、TheElec CHINA、ETNews 会与 X/竞品一起做：

- Odaily 历史查重
- 运行中 candidate 去重

### 标题提醒搜索

处理：

- `external_media_alert`
- `ai_source_alert`
- 历史兼容 `mainstream_media`

它只服务标题提醒链路，不负责新的 Crypto信源或 AI信源全文写作任务。

## 运行交接

- 收集器先写入 `tasks` 作为档案，再提交本地 SQLite job。
- `local_pipeline` 直接按来源调用搜索阶段，搜索结果仍写回 `x_task_pipeline.search_result` 或 `external_media_alert_pipeline.search_result`。
- `tasks.status` 继续保留 `deduped / searched / duplicate / search_failed` 等观察状态，但不再用于多个常驻 worker 之间抢任务。
- 主写作链路中，搜索阶段成功结束时必须写入 `x_task_pipeline.search_completed_at`，供事件复盘拆分“入库到搜索完成”耗时。

## 历史集合

- `odaily_reference_items`
- 本地 search cache
- `external_media_alert_history`

`external_media_alert_history` 现在主要承载 Crypto信源标题提醒和 AI信源标题提醒历史；历史 `mainstream_media` 仍可共用这份窗口。

运行中 candidate 变化频繁，搜索阶段每次都会从数据库刷新 active candidate 集合，并同步写入本地 search cache，避免连续来源之间因为缓存滞后漏判重复。

Odaily 历史集合以 Odaily 官方快讯 API 为真实来源。搜索阶段每次查重会按窗口调用 Odaily API 拉取最新已发布快讯，并同步刷新本地 search cache；只有 Odaily API 拉取失败时，才回退读取 Supabase `odaily_reference_items` 备份或本地 cache。Supabase 在这条链路中只是档案和兜底来源，不作为判断“最新 Odaily 已发布快讯”的主源。

## 查重流程

`x_processing` 搜索阶段当前只保留三段式查重：

1. 精确重复：
   - 来源链接规范化后完全相同。
   - 标题规范化后完全相同。
   - 运行中 candidate 可按必要场景比较内部 `doc_id`。
2. 向量相似度：
   - 对当前任务、Odaily 历史和运行中 candidate 计算 embedding 相似度。
   - 相似度达到 `SEARCH_DUPLICATE_THRESHOLD` 时直接判为疑似重复。
3. AI 复核：
   - 相似度未达到直接判重阈值，但达到 `SEARCH_AI_REVIEW_THRESHOLD` 时，调用搜索者 AI 复核。
   - AI 复核只判断当前内容与命中候选是否为同一事件、旧闻重复或新进展。

当前默认 `SEARCH_AI_REVIEW_THRESHOLD = 0.65`，用于把更多“核心事实相同，但新增补充指标、主体别名或数字表达不同”的候选送入 AI 复核。

搜索者 AI 复核使用独立模型配置，不复用判断者模型配置：

- `SEARCH_AI_REVIEW_MODEL`：生产默认 `odaily-deepseek-review`，由本机 LiteLLM 路由到 DeepSeek 官方模型。
- `SEARCH_AI_REVIEW_REASONING_EFFORT`：默认 `low`。
- `SEARCH_AI_REVIEW_OPENAI_BASE_URL` / `SEARCH_AI_REVIEW_OPENAI_API_STYLE` / `SEARCH_AI_REVIEW_OPENAI_API_KEY`：仅搜索 AI 复核使用的兼容接口配置；未设置时沿用 `X_PROCESS_OPENAI_*` / `ODAILY_LLM_*`。
- `SEARCH_AI_REVIEW_OMIT_REASONING_EFFORT`：兼容不接收 reasoning 参数的接口，例如 DeepSeek 非思考模式。
- `SEARCH_AI_REVIEW_CHAT_RESPONSE_FORMAT_MODE`：`chat_completions` 下的 JSON 输出模式，默认 `json_schema`；DeepSeek 非思考模式应改为 `json_object`。
- `SEARCH_AI_REVIEW_APPEND_JSON_SCHEMA_TO_PROMPT`：当接口只稳定支持 JSON object 时，把 JSON Schema 约束追加到 prompt；搜索复核切 DeepSeek 非思考模式时应开启。

生产中搜索 AI 复核固定走 DeepSeek；Chrome 插件 `AI查重` 复用这一配置。写作、发布和插件生成继续走 GPT 模型别名。

当搜索阶段最终没有判成重复时，`x_task_pipeline.search_result` 现在也会保留本轮最佳候选的目标类型、目标 ID、标题、相似度，以及 AI 复核的原始输出（如果本轮做过 AI 复核），便于后续排查“为什么没判重”。

当前不再使用“共享词 / 词面规则”直接判重。也就是说，除精确重复外，所有非精确候选都必须先经过向量相似度筛选；只有进入阈值区间的候选才可能交给 AI 复核。

向量相似度计算仍保持现有语义：对当前任务向量与候选文档向量计算 cosine similarity，并选择最高相似度候选。实现上优先使用 NumPy 批量计算，以减少 Python 循环开销；如果运行环境没有安装 NumPy，会自动回退到原有逐条计算逻辑，不改变查重结果和失败边界。

`快讯生成` 插件的 `AI查重` 轻接口复用同一套查重边界：

- 精确重复
- 向量相似度
- 必要时 AI 复核

插件不会额外启用独立的共享词规则或前端判断。
