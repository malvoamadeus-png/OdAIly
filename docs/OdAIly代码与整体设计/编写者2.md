# 编写者2

## 职责

编写者2对应 `x_processing` 的 `format_publish` 阶段，负责：

- 规范标题和正文格式
- 产出最终定稿 `final_title` / `final_content`
- 把任务状态推进到 `publisher_pending`
- 写入 `x_task_pipeline.format_completed_at` 作为定稿完成时间，供复盘拆分编写者2耗时
- 最终正文保留有效段落内容，但移除空白段落；若存在多段，只使用单个 `\n` 作为段间分隔，不保留空行
- 最终正文会补中英、中数之间缺失的空格，例如 `Altman表示` -> `Altman 表示`、`1000美元` -> `1000 美元`
- 定稿阶段会兜底拦截 Markdown 链接、`http(s)` 链接、`原文链接`/`来源链接`提示、正文首行的“标题：”“标题为：”“正文为：”等字段标签、重复标题或以完整标题作为首句，以及“我会先”“我将先”“以下是”“下面是”等模型过程说明；命中时进入 `format_failed`，不推进到发布者。
- 标题和正文中的 `@Jason60704294` 或 `Jason60704294` 会确定性替换为 `“先定10个大目标”`；替换在空格规范化后再次校正，固定短语内部不会被改成 `先定 10 个大目标`。`CXMT` 或 `ChangXin Memory Technologies` 也会确定性替换为 `长鑫存储`。

编写者2不再负责：

- 调用发布接口
- 判断是否自动发布
- 发送统一编写链路的正常业务 Telegram 通知

## Crypto信源与 AI信源发布约定

- 所有进入编写链路的 Crypto信源和 AI信源稿件都保留 `sourceUrl`
- 来源名优先使用 `metadata.site_display_name`
- Crypto信源路线名显示为 `Crypto信源`
- AI信源路线名显示为 `AI信源`

这同时适用于：

- `tasks.source = 'non_mainstream_media'`
- `tasks.source = 'ai_source'`
- 历史兼容 `tasks.source = 'mainstream_media'`

## 来源链接边界

编写者2只负责定稿标题和正文，不决定 Odaily 发布 API 的 `sourceUrl` 参数。

来源链接是否透出由发布者在调用发布 API 时判断；具体规则见 `发布者.md`。

## 与发布者的边界

- `format_publish` 成功后，后续是否自动发布由 `publish` 阶段单独判断。
- `publish` 阶段才会调用推送接口，并保证每条稿件只调用一次。
- 统一编写链路的正常业务通知由 `publish` 阶段回写通知结果；不再发送 Telegram，值班侧通过信息流插件查看。
