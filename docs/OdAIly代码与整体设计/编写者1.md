# 编写者1

## 职责

编写者1对应 `x_processing` 的 `write` 阶段，负责根据当前任务类型选择 Prompt，并生成草稿标题与正文。

写作阶段成功结束时，`x_task_pipeline.write_completed_at` 必须写入完成时间，用于区分搜索后判断/写作耗时与后续格式化、发布耗时。

对于 `tasks.source = 'x'` 的输入，编写者1不直接使用抓取时的原始作者显示名，而是先使用统一作者名决策结果 `effective_author_name`。该名称按以下顺序计算：

- 优先使用控制台 `x_capture_accounts` 中按账号维护的 `写作名`
- 没有 `写作名` 时，回退到任务元数据中的原始 `author_display_name`
- 仍为空时，回退到任务元数据中的 `author_username`

处理时如果能读取到当前账号配置，应以当前 `x_capture_accounts` 配置重新计算 `effective_author_name`；任务入库时缓存到 `metadata` 的值只作为兜底。

常规模板中“{发言人}在X平台发文表示”的 `{发言人}` 应使用 `effective_author_name`。

常规、链上、Crypto信源和 AI信源写作模板现在统一限制标题中的 `[主体]：` 格式：

- 只有当主体本身显著增加新闻价值时才使用冒号标题。
- 读者高度熟知的人物直接用名字，例如“特朗普：”“CZ：”“Vitalik Buterin：”。
- 读者不一定熟悉姓名、但其职位本身构成新闻价值的人物，用关键职位而不是陌生姓名，例如“美联储主席：”“SEC主席：”“财政部长：”。
- 项目或公司的正式公告主体可用项目名或公司名。
- 普通 KOL、研究员、分析师、CFO/COO、业务负责人、投资人、项目员工等，不得仅因其发言而使用冒号标题。
- 普通分析师、研究员、KOL、专栏作者等陌生人物观点，不把姓名作为标题主体；观点类标题优先写 `分析：{直接观点}`。
- 只有当前可配置知名人物名单命中的人物，才可使用 `人名：{直接观点}` 标题。

分析、专栏和评论类材料的标题应使用“人或分析 + 直接观点”的主动视角，避免把第三方事物写成“XX被指/被称/被认为……”这类被动客观视角。安全事件标题优先写主体、攻击者或身份、进入方式、涉及范围；“未造成数据和资金损失”“未导致损失”“未发现异常”等风险结果只能作为正文信息或标题后半句，不作为标题前置亮点。预测市场内容禁用“押注”“投注”“赔率”等博彩词，统一改用“购入/买入/持有 XX 万美元某结果”“某结果概率暂报 XX%”等交易和概率表达。

翻译口径保留“外国人名、机构、项目名默认保留英文原名”的原则，但补充语言描述：中文语境中已有高度通行中文名称的外国大型企业或股票代码，必须优先使用通行中文名称；不存在稳定通行中文名称的机构、项目、人名仍保留英文原名。当前模板明确要求 Apple/AAPL 写“苹果”、NVIDIA/NVDA 写“英伟达”、Samsung/Samsung Electronics 写“三星/三星电子”。

定稿阶段会对上述高频大型企业名称做兜底规范化，避免模型输出 `AAPL`、`NVDA`、`Samsung` 等英文名或股票代码后直接进入发布。

默认模型配置：

- `X_PROCESS_WRITER_MODEL`：生产默认 `odaily-gpt-writer`，由本机 LiteLLM 路由到 OpenAI 中转站。
- `X_PROCESS_WRITER_REASONING_EFFORT`：默认 `low`

当前共享 AI 客户端兼容两类 `chat_completions` 返回：

- 常规 `application/json`
- 兼容网关返回的 `text/event-stream` 分块文本

写作阶段会先在客户端侧把 `text/event-stream` 聚合回完整正文，再进入标题/正文解析，避免兼容网关使用 SSE 返回时直接落成 `write_failed`。

生产写作阶段使用严格 JSON Schema，一次模型调用同时返回可发布标题、正文与标题溯源。结构化链路不会再向模型追加“一行标题、空一行、正文”的旧文本协议；JSON 的 `title` 只允许标题文本，`content` 只允许正文文本，不得携带“标题为：”“正文为：”等字段标签、重复标题或 `Odaily` 刊头。Chrome 插件 `快速生成` / `生成快讯` 继续使用原文本契约，不受此变更影响。

生产字段固定为：`title`、`content`、`title_strategy`、`title_strategy_reason`、`matched_title_rules`、`feature_mode_applied`、`feature_mode_reason`。`title_strategy` 只允许 `plain / speaker_anchor / entity_front / action_first / result_front / amount_front / time_window_front`。服务端把确定性识别的 `matched_known_subjects` 补入 `writer_output.trace`，并继续保留 `raw_output`；模型不能自行声明知名主体。

结构化输出无效、缺字段、枚举越界，或标题正文包含 Markdown 链接、`http(s)` 链接、来源提示、字段标签、重复标题与模型过程说明时，写作直接失败并进入现有重试机制，不回退旧文本格式。特色模式未启用时，服务端强制 `feature_mode_applied = false`。

知名人物名单由控制台 `Prompt及规则管理 / 知名人物` 维护，运行时读取 Linux 本地 `data/config/known_title_subjects.json`，默认种子为 `Rune、Cobie、Vitalik、CZ`。后台只保存姓名名单，不保存别名、身份或逐人标题情景；需要匹配别名时，把别名作为名单中的另一个名称加入。每条任务只按当前名单匹配材料，命中时向 Prompt 注入简洁提示：`当前材料命中知名人物：CZ、Vitalik。遇到这些人物观点时可使用“人名：观点”标题。` 未命中任务不会携带整张名单或复杂 JSON。

所有写作 Prompt 都要求利润、亏损、盈亏、损益按中文语义落稿：负数盈亏必须写为“亏损490万美元”，正数盈亏可写为“盈利490万美元”，不得输出“盈亏为-490万美元”“累计盈亏-490万美元”“损益为-490万美元”等负号式表达。

Chrome 插件 `快速生成` 会复用编写者1 Prompt，但单独把文案生成模型固定为 `EDITOR_PLUGIN_QUICK_GENERATE_WRITER_MODEL`，生产默认 `odaily-deepseek-fast`，reasoning effort 固定为 `low`。Chrome 插件 `生成快讯` 会复用默认 writer 模型，生产默认 `odaily-gpt-writer`，但插件轻服务内的 reasoning effort 固定为 `low`；这些插件专用设置不会改变自动化主链路的默认 writer 配置。插件 `AI查重` 复用搜索者 AI 复核并固定走 DeepSeek 别名，快速生成走 DeepSeek fast 别名，生成快讯走 GPT writer 别名。

## Prompt 选择

- `regular` -> `x_regular_writer`
- `onchain` -> `x_onchain_writer`
- `funding` -> `x_funding_writer`
- `non_mainstream_media` -> `mainstream_media_writer`
- `ai_source` -> `mainstream_media_writer`
- `jin10` 由 `judge_jin10` 保留后写入 `regular`，因此使用 `x_regular_writer`
- 历史兼容 `tasks.source = 'mainstream_media'` -> `mainstream_media_writer`

`non_mainstream_media_writer` 和 `ai_source_writer` 不再被新任务使用，只保留历史模板记录。

标记为 `AI信源` 的 X 账号会先进入判断者-AI；如果判断者-AI 保留，`news_type` 固定为 `regular`，因此仍使用 `x_regular_writer`，不改用 Crypto信源 / AI信源全文模板。发布阶段另行按 `AI信源` 发布者规则块判断，当前该规则块关闭时直接挂后台。

2026-07-15 检查生产 active `x_regular_writer` 时，发现 `version_number = 6`、`version_id = 20` 的内容不包含“冒号标题克制使用”规则；如果生产仍出现普通 KOL / 分析师 `发言人：内容` 式标题，应优先检查并发布前端 Prompt 控制台中的 `x_regular_writer` 新版本。

## X 写作输入

X 来源写作输入仍然保留原始来源链接和正文，但作者名使用标准化后的 `effective_author_name`。

- 自动化 X 主链路使用 `x_capture_accounts` 里的 `写作名` 规则
- 插件 `快速生成` / `生成快讯` 在处理 X 帖文时复用同一套 `effective_author_name` 规则
- 该规则只影响编写口径，不改动原始抓取作者字段

## Crypto信源写作输入

统一 Crypto信源写作输入现在分两类：

- Crypto信源全文：媒体名、作者、原标题、来源链接、正文
- AI信源全文：媒体名、作者、原标题、来源链接、正文，输入块标题显示为“待处理AI信源原文”
- 历史 Crypto信源快讯：媒体名、原标题、来源链接、原始快讯

CoinDesk、Cointelegraph、Decrypt 的新任务属于 Crypto信源全文；TheElec CHINA、ETNews Electronics、ETNews SW 属于 AI信源全文。

## 金十写作输入

金十来源在写作阶段使用“快讯材料”输入块，只提供标题、正文、发布时间和必要 metadata，不向 Prompt 强调“金十”来源名。

金十任务不向后续发布接口透出原始来源链接；发布阶段默认通过 `jin10` 渠道挂后台。
