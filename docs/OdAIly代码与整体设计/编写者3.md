# 编写者3

## 职责

编写者3负责在快讯主发布链路之外，判断 Odaily 已发布快讯是否需要追加“此前消息”，并在需要时生成一条历史背景内容供信息流插件展示。

编写者3的目标是补充读者理解当前事件所需的前序事实，例如某个法案当前进入新阶段前，数月内曾有关键进展；某个项目发生重大进展前，近期曾有融资、主网上线、空投或代币经济学等关键节点。

编写者3不负责主快讯生成、不负责确定性格式化、不负责后台推送，也不替代搜索者的查重职责。

## 链路位置

由于快讯发布速度要求，编写者3不插入判断者、搜索者、编写者1、编写者2组成的主发布链路。

当前主链路仍保持：

```text
判断者 / 搜索者 -> 编写者1 -> 编写者2 -> 后台审核 / 发布
```

生产环境以 Odaily 已发布快讯镜像作为统一入口异步调用编写者3：

```text
Odaily 已发布快讯同步到 odaily_reference_items
  -> 编写者3判断是否需要此前消息
  -> 编写者3检索历史内容
  -> 编写者3生成此前消息
  -> 写入此前消息结果，进入信息流插件
```

路径化排除规则只在原始来源创建 `tasks` 前执行。编写者3消费的是 Odaily 已发布快讯镜像，不属于 `x / competitor / crypto_source / ai_source / mixed_source / jin10` 任一路径，因此不重复读取排除规则，也不再读取旧 `competitor_filter_keywords`。

编写者3失败、超时、命中排除词或判断不需要补充时，不影响原快讯上线。

## 生产实现

生产代码位于 `backend/packages/writer3/`，作为独立异步 worker 运行。历史实验目录 `writer3_lab/` 已从仓库移除，不进入生产调用链，也不作为当前部署资产保留。

生产输入来自 Supabase：

- `odaily_reference_items` 中的 Odaily 已发布快讯镜像。
- 该镜像中的正文应保留 Odaily 原始段落换行；实时抓取与 `writer3-backfill-odaily` 回填都必须使用同一规则，避免把分段公告粘连成单行正文。
- 不区分快讯最初来自 AI 采集、人工编辑后发布，还是人工直接在后台新增发布；只要最终出现在 Odaily 已发布快讯镜像中，就使用同一套编写者3流程。
- 只处理 `WRITER3_START_AFTER` 之后发布的新快讯。
- 只处理发布时间距离当前时间不超过 `WRITER3_CURRENT_FRESHNESS_WINDOW_SECONDS` 的新快讯，默认 600 秒。超过窗口的快讯直接跳过，不调用 AI、不检索、不发送 Telegram，用于避免停机重启后旧快讯集中进入编写者3。
- 当前快讯必须有发布时间；历史检索只使用发布时间明确且早于当前快讯的 Odaily reference。

生产状态写入 `writer3_contexts`，按 `current_source + current_source_item_id` 防重复，记录处理状态、通知结果和失败原因。状态固定为 `pending / processing / skipped / sent / failed`。旧的 `task_id` 仅作为兼容字段保留；编写者3不再依赖 `x_task_pipeline.final_content` 作为唯一输入。

编写者3只把拼接后的内容写入自身结果表供信息流插件展示，不调用 Push API，不写回网站，不修改主快讯正文。`telegram_result` 固定为 `skipped=true`，表示正常业务消息不发 Telegram。

## 触发类型

第一版采用正向触发。只有当前快讯命中明确需要历史上下文的关键事实，才进入编写者3；不做泛化的“同一主体历史追踪”。

可触发类型：

- 监管、法案、诉讼：包括政策进展、法案阶段变化、监管机构动作、法院裁决、诉讼进展等。
- 项目重大进展：包括融资、主网上线、测试网、空投、代币经济学、黑客事件等。

不触发类型：

- 所有报价、价格结果、涨跌幅、突破、跌破等价格交易类快讯。
- 交易所上币、上线交易对、Launchpool、Alpha、现货或合约上线等交易所 listing 类快讯。
- 单纯因为“同一主体近期出现过其他新闻”而触发。
- 没有明确前序事实需求的普通公告、合作、活动和社区动态。

## 搜索方案

### 第一版方案：焦点主体检索

第一版不再采用向量相似度作为主召回方式。向量相似度容易把句式、平台、动作相似但主体不同的快讯召回为高相关候选，例如不同代币的交易所上线快讯。

第一版采用“焦点主体/事项锚点 + 固定前序类型 + 本地打分截断 + AI 复判”：

1. 先做非触发类型过滤。
2. 由 AI 判断当前快讯是否命中可触发类型，并抽取焦点主体、主体别名、当前事件类型，以及法案/诉讼/监管类的事项锚点。
3. 代码根据当前事件类型映射固定的可接受前序类型。
4. 项目类按焦点主体及别名检索候选内容；法案、诉讼、监管类按事项锚点及别名检索候选内容。
5. 代码用主体命中位置、前序类型关键词、发布时间和非触发词做本地打分。
6. 只保留少量高分候选交给 AI 复判。
7. AI 从候选中选择一条真正可用的证据；如果没有可用候选，则不生成此前消息。
8. 只有 AI 明确判断存在可写入的前序关系时，才生成此前消息并与原快讯拼接后写入结果表。

### 焦点主体

焦点主体是当前事件真正发生在其身上的对象，不是所有被提到的实体。

示例：

- “某项目完成融资”中，焦点主体是该项目。
- “某法案进入参议院审议”中，焦点主体是该法案或对应监管事项。
- “某协议主网上线”中，焦点主体是该协议或项目。

交易所、监管机构、法院、媒体、投资机构等实体只有在其本身是当前事件对象时才可作为焦点主体；否则只作为上下文实体，不用于主检索。

交易所上币类快讯不进入编写者3，因此不需要围绕交易所或被上线代币检索此前消息。

### 事项锚点

法案、诉讼、监管类除了焦点主体，还需要额外抽取“事项锚点”。

事项锚点用于表达“同一法案、同一案件、同一监管事项”，避免只因命中同一机构名就召回大量弱相关历史内容。

示例：

- “CLARITY法案最新草案新增内幕交易条款”中，事项锚点应为 `CLARITY Act` 或 `CLARITY法案`。
- “CFTC在上诉案中支持Kalshi，主张预测市场监管权归联邦所有”中，事项锚点应为 `Kalshi预测市场管辖权争议`。
- “Bitcoin Fog上诉案聚焦管辖权争议”中，事项锚点应为 `Bitcoin Fog上诉案`。

SEC、CFTC、司法部、法院、参议院、众议院、委员会、美联储等高频机构名不能单独作为事项锚点；它们只能作为上下文实体或事项锚点的一部分。

### 当前事件类型与前序类型

AI 只负责判断当前快讯属于哪种可触发事件类型，以及焦点主体是谁；不让 AI 每次自由生成检索关键词。

对于法案、诉讼、监管类，AI 还需要输出事项锚点及其别名。

代码维护固定映射：

```json
{
  "financing": ["prior_financing", "mainnet_or_testnet", "tokenomics", "roadmap"],
  "mainnet_launch": ["prior_financing", "testnet", "tokenomics", "airdrop", "roadmap"],
  "testnet_launch": ["prior_financing", "roadmap", "ecosystem_progress"],
  "airdrop": ["prior_financing", "tokenomics", "mainnet_or_testnet", "roadmap"],
  "tokenomics": ["prior_financing", "mainnet_or_testnet", "airdrop", "roadmap"],
  "regulation": ["same_policy_prior_stage", "same_agency_prior_action"],
  "bill": ["same_bill_prior_stage", "same_policy_prior_stage"],
  "lawsuit": ["same_case_prior_stage", "same_party_prior_action"],
  "security": ["same_incident_prior_stage", "attack_loss_tracking", "compensation_or_recovery"]
}
```

每个前序类型对应一组本地维护的中英文关键词。关键词用于候选打分和截断，不作为最终是否补充的唯一依据。

示例：

```json
{
  "prior_financing": ["融资", "完成融资", "投资", "领投", "参投", "种子轮", "战略轮", "funding", "raised"],
  "testnet": ["测试网", "激励测试网", "testnet"],
  "tokenomics": ["代币经济学", "代币分配", "TGE", "空投", "tokenomics"],
  "roadmap": ["路线图", "计划", "预计", "roadmap"],
  "same_bill_prior_stage": ["法案", "投票", "通过", "提交", "审议", "bill", "vote"],
  "same_case_prior_stage": ["诉讼", "起诉", "裁决", "和解", "法院", "lawsuit", "court"],
  "same_incident_prior_stage": ["攻击", "漏洞", "损失", "追踪", "赔付", "恢复", "exploit", "hack"]
}
```

### 候选召回与截断

项目类检索必须先命中焦点主体或其别名。法案、诉讼、监管类检索必须先命中事项锚点或事项别名。

纯英文或数字别名必须按完整词命中，不能从其他英文单词中截取子串。例如 `TAC` 只能命中独立的 `TAC`，不能命中 `attacks`、`Stack`、`TACO` 或 `Coinstack`。中文或中英混合别名可继续按包含关系匹配。

只命中交易所、监管机构、法院、媒体、投资机构、赛道词或动作词的内容，不作为有效候选。

候选打分建议：

- 焦点主体命中标题：高权重。
- 焦点主体命中正文：中权重。
- 法案、诉讼、监管类的事项锚点命中标题：最高权重。
- 法案、诉讼、监管类的事项锚点命中正文：高权重。
- 同时命中当前事件类型对应的前序类型关键词：加权。
- 命中报价、价格交易或交易所 listing 词：降权或丢弃。
- 只命中上下文实体、平台名、机构名、动作词：丢弃。
- 发布时间越近可适度加权，但不能高于主体准确性和前序类型匹配。

每条当前快讯最多保留少量候选进入 AI 复判，默认上限为 20 条。超过上限时按本地分数、发布时间和来源质量排序截断。

向量相似度只可作为辅助排序信号，不作为主召回条件，也不作为是否补充此前消息的直接依据。

## 时间窗口

按触发类型使用固定历史窗口：

- 所有可触发类型：向前检索 90 天。

如果候选历史内容超过对应窗口，即使语义相似，也默认不作为此前消息使用。

发布时间为空的历史内容不进入生产检索，避免无法判断其是否早于当前快讯。

## AI 判断标准

AI 分两次使用。

第一次用于当前快讯解析：

- 判断当前快讯是否属于可触发类型。
- 抽取焦点主体和主体别名。
- 输出当前事件类型。
- 识别上下文实体，但上下文实体不用于主检索。
- 法案、诉讼、监管类输出事项锚点和事项别名。

第一次解析建议输出：

```json
{
  "should_run_writer3": true,
  "current_event_type": "mainnet_launch",
  "focus_subject": {
    "name": "string",
    "type": "project | protocol | token | bill | lawsuit | policy | security_incident",
    "aliases": ["string"]
  },
  "context_entities": ["string"],
  "matter_key": "string",
  "matter_aliases": ["string"]
}
```

第二次用于候选复判和此前消息编写。AI 需要判断历史候选是否满足以下条件：

- 与当前快讯存在明确前序关系。
- 能帮助读者理解当前事件为什么重要、发展到哪一步或此前发生过什么。
- 不只是同一主体的无关历史新闻。
- 不只是同一赛道、同一人物、同一机构或同一代币的弱相关内容。
- 不把时间先后关系强行写成因果关系。
- 可以使用一条或多条候选，但所有写入正文的事实都必须被候选明确支持。

允许补充的关系包括：

- 当前项目重大进展前，项目此前有融资、主网上线、测试网、空投、代币经济学等关键节点。
- 当前监管、法案或诉讼进展前，90 天内有同一事项的前序阶段。
- 当前黑客事件后续前，90 天内有攻击、损失、追踪、赔付或恢复进展。

## 写作格式

此前消息必须以“此前消息，”开头，并只陈述候选中的前序事实。正文不得写 AI 的判断过程、选择理由或关联分析。

不要硬套“主体 + 于 + 时间 + 发生/宣布/完成/进入”的句式，不要输出 ISO 时间戳。除非候选原文中的自然日期对理解前因后果必要，通常不写时间。

不要把“为什么相关”“如何衔接当前事件”写进正文；这些是判断过程，不是可发布事实。不要写虚性描述、抽象评价或候选没有明确说出的延伸判断，例如“引发持续讨论”“受到关注”“延续布局”“形成前后衔接”“提供前情”“可视为下一步”。如果候选只提供了具体动作、文件、金额、条款或进展，就只写这些具体事实。

## 通知输出

生产环境只保存最终可读内容供信息流插件展示，不输出候选列表、选择理由或模型思路。Telegram 不再发送正常“此前消息”；只有监督者发现 worker 异常或积压时才发 Telegram 错误告警。

只有 `should_write=true`、`context_text` 非空且模型返回的 `evidence_source_item_ids` 能对应到实际候选时，才写入 `sent` 状态。

保存正文格式固定为四行：

```text
{Odaily 已发布快讯正文}
{context_text}
当前快讯链接：{current_news_url}
此前消息来源：{evidence_source_url}
```

原快讯正文和此前消息之间只换一行，不插入空行。`context_text` 必须以“此前消息，”开头。

`当前快讯链接` 指本次进入编写者3的已发布 Odaily 快讯。若输入来自 Odaily reference 镜像，优先用 `current_source_item_id` 生成 `https://www.odaily.news/zh-CN/newsflash/{id}`，避免使用镜像中保留的外部来源链接造成误解。

`此前消息来源` 指模型写作“此前消息”时选中的 Odaily 证据快讯链接。即使 AI 使用多条候选写作，也只附一条用于追溯的 Odaily 证据链接。

历史 Telegram inline 确认链路仅用于兼容旧消息；正常“此前消息”不再创建新的 Telegram 确认目标。

## 信息流插件反馈

当前独立的 `信息流插件` 会把“此前消息”纳入低频区，并提供：

- `接受`
- `拒绝`

插件反馈用于记录值班编辑对“此前消息”质量的判断，并把反馈人和反馈时间写入数据库，便于后续复盘。

当前生产以信息流插件反馈为准；本地 SQLite 确认记录只保留历史 Telegram 消息兼容。

## 运行命令

```bash
python backend/src/main.py writer3-init-db
python backend/src/main.py writer3-backfill-odaily --days 90
python backend/src/main.py writer3-sync-index --days 90
python backend/src/main.py writer3-worker --once
python backend/src/main.py writer3-worker
python backend/src/main.py writer3-confirm-worker --once
python backend/src/main.py writer3-confirm-worker
python backend/src/main.py writer3-reset-task --task-id 123
```

关键环境变量：

```text
WRITER3_START_AFTER=
WRITER3_HISTORY_DAYS=90
WRITER3_ANALYSIS_MODEL=odaily-deepseek-fast
WRITER3_WRITER_MODEL=odaily-gpt-writer
WRITER3_WRITER_REASONING_EFFORT=medium
WRITER3_CANDIDATE_LIMIT=20
WRITER3_CONTEXT_CANDIDATES=5
WRITER3_CURRENT_FRESHNESS_WINDOW_SECONDS=600
WRITER3_TELEGRAM_MESSAGE_THREAD_ID=
```

`WRITER3_TELEGRAM_MESSAGE_THREAD_ID` 仅保留历史兼容；正常“此前消息”不再发送 Telegram，缺失时不会阻止 worker 启动。

## 失败处理

- 检索失败：记录失败原因，不补充此前消息。
- AI 判断失败或超时：记录失败原因，不补充此前消息。
- 当前快讯发布时间超过新鲜度窗口：记录 `expired_current_news`，不调用 AI、不检索、不写入通知结果。
- 候选内容不足：输出 `should_write=false`。
- 补充内容无法被历史证据支持：输出 `should_write=false`。
- 确认按钮回调失败：不影响此前消息已发送状态；确认 worker 记录本轮失败并继续轮询。
- `failed` 任务不会被 worker 自动无限重试；排查配置或模型错误后，用 `writer3-reset-task --task-id` 手动重置。
- 常驻 worker 会捕获单轮外围异常并继续下一轮，避免一次领取任务、数据库连接或外部服务波动导致编写者3进程退出。

编写者3是异步补充模块，不应阻断主快讯发布链路。
