# 竞品监控

## 背景

OdAIly 是本项目的业务主体。BlockBeats、PANews、金色财经等专注 Crypto 领域的新闻媒体属于竞品信源。

竞品监控不是直接搬运竞品内容，而是作为信息采集程序 A 的一个组件，持续发现竞品已经发布但 Odaily 可能尚未发布的快讯线索，并将可处理信息写入统一任务结构。

Odaily 自身快讯不是竞品信源。它是本方已发布语料，供搜索者判断竞品线索是否已经被 Odaily 报道过。

## 目标

- 持续采集竞品最新快讯。
- 清洗竞品固定前缀、页面噪声和来源侧冗余内容。
- 将竞品快讯写入数据库中的统一任务结构。
- 记录来源、来源侧 ID、标题、正文、发布时间、原始链接和原始 payload，便于追溯。
- 将竞品任务交给搜索者和判断者处理，不在竞品监控组件内实现去重、品类判断和写作。

## 竞品范围

首批：

- BlockBeats。
- PANews。
- 金色财经。

后续可增加其他 Crypto 新闻媒体。

## 采集职责

竞品监控组件只负责采集和入库前的基础规范化。

它负责：

- 调用竞品 API 或页面抓取逻辑。
- BlockBeats 使用官方 API；PANews 和金色财经使用当前可用快讯接口。
- Odaily 使用官方 REST API 的快讯接口作为本方参考语料来源。
- 提取标题、正文、发布时间、来源链接和来源侧 ID。
- 删除媒体固定前缀，例如 `BlockBeats 消息，5 月 9 日，`、`PANews 5月9日消息，` 等。
- 删除竞品品牌词，清洗后正文不得残留 `律动`、`BlockBeats`、`PANews`、`金色财经`。
- 写入统一任务结构。
- 确保同一竞品来源侧 ID 不重复写入。

它不负责：

- 判断是否已经被 Odaily 报道。
- 判断是否为 Crypto、AI 或其他品类。
- 判断是否值得发布。
- 生成快讯文本。
- 推送后台。

这些职责分别由搜索者、判断者、编写者和发布层承担。

## 排除词表

竞品监控在写入正式编写流程和事件复盘流程前，读取控制台维护的 `competitor_filter_keywords` 启用词表。数据库表名沿用旧名，控制台和文档统一称为“排除词”。

默认启用词：

- `跌破`
- `突破`
- `爆仓`
- `Bitget`

匹配规则：

- 匹配全部快讯来源：`odaily`、`blockbeats`、`panews`、`jinse`。
- 不匹配 X 来源。
- 匹配范围是清洗后的标题 + 正文。
- 中文按原文子串匹配；英文大小写不敏感，例如 `Bitget` 可命中 `bitget`。

命中后的行为：

- 不写入 `tasks`。
- 不写入 `odaily_reference_items`。
- 不进入 `newsflash_items/newsflash_events/newsflash_event_sources` 事件分析流程。
- 不进入搜索者、判断者、编写者和推送流程。
- 只在 Linux journal 中记录 `source/source_item_id/matched_terms/title`，便于排查。

## 流程位置

```text
Odaily/竞品快讯采集
  -> 排除词表命中则跳过并写服务日志
  -> 未命中排除词时写入事件复盘表
  -> 未命中排除词的竞品写入 tasks
  -> 搜索者：判断是否与 Odaily 已发布快讯或流水线候选事件重复
  -> 未重复时进入判断者
  -> 判断者决定是否进入后续快讯流水线
```

搜索者的相似度阈值、AI 复判、embedding 缓存和 in-flight 去重方案详见 `搜索者.md`。

判断者的品类判断、AI 泛科技内容处理和丢弃规则详见 `完整程序架构.md` 的“判断者”章节。

## 事件聚合与控制台复盘方案

### 目标

竞品监控后续需要支持按“新闻事件”复盘，而不是只按单条快讯查看。

核心关注点：

- 哪些事件 Odaily 已发布，且多家竞品也发布了，用于人工对比标题、正文和信息取舍。
- 哪些事件只有 Odaily 发布，用于复盘本方选题边界。
- 哪些事件只有竞品发布，尤其是多家竞品共同发布但 Odaily 未发布，用于复盘值班判断和漏发风险。

系统只提供事件聚合、来源归属、筛选、展示、收藏和备注能力。内容质量判断由用户人工完成，不做 AI 辅助点评。

### 数据边界

原始快讯不合并、不覆盖。每家媒体的每条快讯都保留原始记录：

- 来源媒体。
- 来源侧 ID。
- 标题。
- 正文。
- 发布时间。
- 来源链接。
- 抓取时间。
- 原始 payload。

线上需要为同一新闻事件生成稳定的 `event_id`。当前实现使用 `evt_` 前缀的 ULID 风格文本 ID，由后端生成。

`event_id` 只表达“这些快讯被归为同一事件”。标题 hash、正文 hash 或 embedding hash 只能作为去重和缓存辅助，不作为业务事件号。

### Embedding 存储

所有参与事件归属判断的快讯都需要向量化，包括 Odaily、BlockBeats、PANews、金色财经等来源。

embedding 只保存在服务器本地 SQLite 缓存，不写入 Supabase。Supabase 只保存 `newsflash_items` 原始快讯、`newsflash_events` 事件、`newsflash_event_sources` 事件来源关系、收藏状态和备注等业务数据。

服务器本地 embedding 缓存至少需要支持：

- 按 `source + source_item_id + content_hash + embedding_model` 复用向量。
- 保存每条来源快讯的向量，而不是只保存事件主记录的向量。
- 支持按时间窗口清理过期向量。
- worker 重启后可复用近期缓存，避免重复请求 embedding 服务。

### 事件归属流程

新快讯进入后，使用该快讯发布时间前 6 小时内的内容作为主要候选窗口。优先使用 `published_at`，缺失时使用入库 `created_at` 兜底。

事件归属采用相似度阶梯：

- 高相似：自动归入同一事件。
- 中相似：调用 AI 只判断是否同一事件。
- 低相似：新建事件。

AI 只做事件归属分析，输出是否同一事件、目标事件和简短原因。AI 不评价标题好坏、不评价正文质量、不生成内容点评。

同一事件下必须保留所有来源快讯的 embedding。新快讯不只和事件主记录比较，也要和事件下已有的每条来源快讯比较。只要新快讯与事件内任一来源满足同事件条件，就可以挂到同一个 `event_id`。

如果同一轮抓取中有多条新快讯同时进入，建议采用 micro-batch 处理：

- 先对本批新快讯统一向量化。
- 同时比较“本批新快讯之间”和“最近 6 小时已有事件来源”。
- 用相似关系建立候选连通分量，再统一分配或合并 `event_id`。

这可以避免 A 先进入后，B、C 同时进入时只识别出 A-B 或 A-C，却没有把 A、B、C 作为同一事件整体处理。

链式相似需要保守处理。如果出现 A-B 相似、B-C 相似，但 A-C 明显不相似，应标记为待人工确认，而不是自动无限扩张事件范围。

### 控制台主表

控制台主表按事件展示，一行代表一个 `event_id`，不按单条快讯直接铺开。

主表建议字段：

- 事件时间。
- 事件标题，默认取首发标题或事件代表标题。
- Odaily 发布情况。
- BlockBeats 发布情况。
- PANews 发布情况。
- 金色财经发布情况。
- 共同发布数量。
- 首发媒体和首发时间。
- 事件状态。
- 收藏按钮。

“共同发布数量”显示数字即可，鼠标 hover 时展示具体发布媒体列表。这个信息属于事件摘要，主表加载时即可返回，不需要额外请求详情。

控制台顶部需要提供一键筛选，而不是要求用户手动组合条件：

- 多家共同发布：来源数大于等于 2。
- 我们也发布：来源包含 Odaily。
- 高价值对比：来源包含 Odaily，且总来源数大于等于 2。
- 竞品有我方无：来源不包含 Odaily，且至少一家竞品发布。
- 竞品共识但我方无：来源不包含 Odaily，且竞品来源数大于等于 2。
- 仅我方发布：只有 Odaily 发布。
- 我方晚发：来源包含 Odaily，但首发媒体不是 Odaily。
- 我方首发：来源包含 Odaily，且首发媒体是 Odaily。
- 已收藏：只看用户收藏的事件。

### 事件详情

事件详情采用懒加载。主表平时只请求事件摘要；用户点击某个事件行后，再请求该事件下的完整快讯明细。

桌面端优先使用右侧抽屉或行内展开面板；移动端可使用底部抽屉。详情区展示：

- `event_id`。
- 事件代表标题。
- 首发媒体和首发时间。
- 参与媒体数量和媒体列表。
- 按发布时间排序的来源快讯列表。
- 每条快讯的标题、正文、发布时间和来源链接。
- 每条快讯的用户备注。

详情页不展示 AI 内容质量点评。用户自行对比标题、正文和信息取舍。

### 收藏与备注

每个事件在主表和详情页都需要有收藏按钮。用户收藏后，可以在专门的收藏页面查看这些事件。

收藏页按事件展示，保留与主表一致的事件摘要和一键筛选能力，并支持进入事件详情。

完整事件页和收藏页都需要支持给每个条目写备注。这里的“条目”指事件下的具体来源快讯，而不只是事件本身。备注用于记录人工复盘结论，例如标题问题、正文信息缺失、值班判断、漏发原因或后续跟进。

备注和收藏属于用户复盘数据，应写入 Supabase，和服务器本地 embedding 缓存分离。

## 竞品任务字段

竞品任务写入 `tasks` 时，至少应保留：

- `source`：竞品来源，例如 `blockbeats`、`panews`、`jinse`。
- `source_item_id`：竞品来源侧唯一 ID。
- `source_url`：原始链接。
- `title`：清洗后的标题。
- `content`：清洗后的正文。
- `published_at`：竞品发布时间。
- `raw_payload`：原始响应中的关键字段。
- `status`：初始状态。

具体表结构以实际 `tasks` 表和后续迁移为准。

Odaily 自身快讯写入 `odaily_reference_items`，并同步到本地搜索者 SQLite 缓存，不写入 `tasks`，也不进入编写或推送流程。

## 当前与未来边界

当前阶段，竞品监控只采集竞品快讯，不直接筛选 AI 泛科技内容。AI 泛科技内容是否丢弃，由判断者统一处理。

未来如果 Odaily 需要发布部分 AI 相关新闻，竞品监控仍只负责采集；品类配置、判断规则和模板扩展应放在判断者、编写者和搜索者对应文档中。
