# 信息流插件

> 当前仓库已实现 V1 独立 Chrome 插件，代码位于 `extension/`。现阶段生产正常业务消息以插件信息流为主，Telegram 只承接系统级错误、宕机、积压和程序异常。当前插件规划包含两个一级模块：`信息流` 和 `快讯生成`；本文聚焦已经落地的 `信息流` 模块。

## 定位

信息流插件是给值班编辑使用的独立 Chrome 插件，不属于控制台，也不是控制台的一个页面。

它的目标是承接新快讯、标题提醒、审核者、此前消息和巨鲸等业务信息，并在浏览器侧边栏里提供：

- 更强的视觉区分。
- 更顺手的常驻查看。
- 更明确的反馈动作。
- 可统计的查看和反馈留痕。

第一版不是新业务链路，而是现有业务结果的增强工作面。

## 非目标

第一版不追求：

- 替代控制台。
- 承担控制台配置能力。
- 直接改稿。
- 自动修改审核结果。
- 承担系统错误、积压和心跳告警。

## 当前关系

插件和控制台无关，是正常业务消息的主要查看入口。

目标上，插件承接的是值班编辑需要连续查看的这几类信息：

- `新快讯`
- `Crypto信源标题提醒`
- `审核者`
- `此前消息`
- `巨鲸`

也就是说，插件不是再造一套来源，而是消费现有上游结果，并在界面层补上查看、声音提醒和反馈能力。Telegram 只保留系统级错误告警。

## 独立代码目录

插件代码必须独立放在仓库根目录单独文件夹，不与 `backend/` 混放，也不依赖把 `backend/` 一起打包给编辑。

当前实现目录：

```text
extension/
  manifest.json
  src/
    background.js
    sidepanel.html
    sidepanel.js
    options.html
    options.js
    styles.css
    lib/
  README-install.md
```

约束：

- 插件可直接以 unpacked extension 方式加载 `extension/` 文件夹。
- 插件运行时代码不要和 `backend/` 目录混在一起。
- 插件工程应尽量自包含，不依赖跨目录引用 `frontend/` 或 `backend/` 的运行时代码。
- 后端只提供数据库、RLS、RPC 或其他远端接口，不承载插件前端源码。

## 形态选择

推荐形态是 `Chrome Extension Manifest V3 + Side Panel`：

- 编辑浏览网页时可以常驻在侧边栏。
- 比 Telegram 和独立网页少一次切换。
- 更适合连续盯消息。
- 可以本地保存分栏高度、筛选和已读状态缓存。

当前不建议第一版做成：

- 独立网页。
- 浏览器 popup。
- 注入网页的悬浮层。

## 一级信息架构

当前插件顶部规划两个一级模块：

- `信息流`
- `快讯生成`

其中：

- `信息流` 负责通知查看、反馈和留痕。
- `快讯生成` 负责从 X 页面抽取正文并调用搜索者 / 编写者能力。

插件其他能力如果未来需要，再继续挂在这两个一级模块之下，不再额外扩成重导航。

当前实现里，这两个一级模块已经共用同一个 side panel 外壳、同一套登录态和同一条顶部导航。

登录态由插件轻服务统一承接。侧边栏提交邮箱密码到 `POST /plugin/auth/login`，轻服务通过 Postgres 读取 Supabase Auth 用户的 bcrypt 密码 hash 并校验 `editor_plugin_users` 白名单，然后签发插件专用短 session，并保存在 Chrome 本地 storage。登录提交期间侧边栏会禁用表单并显示 `正在登录...`；网络请求设置本地超时，超时后提示 `请求超时，请稍后重试`，避免接口不可达时表现为点击登录无反馈。

生产环境中，插件轻服务登录地址必须使用 HTTPS；浏览器侧不会向非 localhost 的 HTTP 插件服务提交密码。当前插件内置地址为 `https://47.76.243.147.sslip.io`，由 Caddy 反向代理到服务器本机 `127.0.0.1:8765`。

顶部导航左侧把 `OdAIly` 品牌字样本身做成了一个快捷入口，用于浏览器内快速定位到 `https://odaily.info/content/newsflash/newsflash-list`：

- 如果目标标签页已经存在，就直接激活并切过去。
- 如果不存在，就新开目标页面。
- 页面就绪后先等待 `0.5` 秒，再做几次短轮询，找到 `重置` 按钮后自动点击。

这个动作运行在插件 `background.js`，通过 `tabs + scripting` 权限完成，不改 Odaily 站点代码。

## 主界面布局

`信息流` 页面按固定高度纵向布局展示三个分区：

- 上方：`高频区`
- 中间：`AI区`
- 下方：`低频区`

默认高度比例为 `60:20:20`。每个区只在自己的列表内滚动，区内新消息不会撑高该区，也不会挤压其他区。编辑可以在插件本地拖拽分隔条调整三段高度，调整后的比例保存在 Chrome 本地 storage。

当某条卡片展开全文时，只会增加该分区内部列表的滚动内容，不改变其他分区的展示高度。

当前实现优先保证同屏可见条数，卡片、按钮和字体都按侧边栏高密度浏览做了收缩，不走 Telegram 那种大块通知样式。

### 高频区

高频区放最需要值班编辑即时扫视的内容：

- `新快讯`
- `Crypto信源标题提醒`
- `审核者`

### 低频区

低频区放需要持续关注但不一定秒级处理的内容：

- `此前消息`
- `巨鲸`

### AI区

AI区专门放 AI 信源相关的新快讯：

- `tasks.source = 'ai_source'` 的新快讯。
- X 账号在控制台标记为 `AI信源` 后，该账号的新快讯在插件里进入 AI区。

X 账号的 `AI信源` 标记不改变 `tasks.source`，仍写入 `x`；它会让自动化主链路进入判断者-AI，保留后仍按 X 常规快讯写作，发布阶段按 AI信源发布者规则块处理，同时在插件里进入 AI区。

## 信息类型定义

### 1. 新快讯

来源仍是现有快讯通知链路。

插件里要重点区分两种状态：

- `已直发`
- `挂后台`

这里的 `挂后台` 只是快讯发布状态，不是“审核者发现错误”。

### 2. Crypto信源标题提醒

这里专指 `tasks.source = 'external_media_alert'` 的标题提醒结果。

当前已覆盖的典型来源包括：

- `The Block`
- `FT Crypto`
- `Fortune Crypto`
- `WSJ`

这些卡片的作用是让编辑在插件里直接看到 Crypto信源标题提醒，不必只盯 Telegram。

### 3. 审核者

这里专指现有 `审核者` 模块发现疑似文本错误后发出的提醒。

值班编辑需要在插件里对每条审核提醒点击：

- `接受`
- `拒绝`

这两个动作的意义是：

- `接受`：认可这条审核提醒指出的问题成立。
- `拒绝`：认为这条审核提醒不成立，属于误报或不值得处理。

它的用途有两层：

- 统计值班编辑响应速度。
- 方便后续复盘审核者误报。

### 4. 此前消息

这里专指现有 `编写者3` 输出的“此前消息”。

值班编辑同样可以点击：

- `接受`
- `拒绝`

用途是给“此前消息”质量留反馈，而不是改动主快讯发布状态。

### 5. 巨鲸

这里沿用现有巨鲸监控内容，只做查看和定位，不需要第一版就加复杂反馈动作。

## 视觉区分

当前 Telegram 里用“金色财经有新快讯-链上-特色模式”这类连续字符串表达来源和类型，扫描效率不够。

插件里建议把一条信息拆成固定视觉槽位：

1. 左侧色条：区分大类。
2. 第一行：标题或主句。
3. 同行轻量标签：状态、来源、模式等必要信息。
4. 摘要。
5. 底部：时间、动作。

建议映射：

| 类型 | 区域 | 主色 | 状态示例 |
| --- | --- | --- | --- |
| 新快讯 | 高频 | 红橙 | `已直发` / `挂后台` |
| Crypto信源标题提醒 | 高频 | 玫红 | `标题提醒` |
| 审核者 | 高频 | 琥珀 | `审核者` / `已接受` / `已拒绝` |
| 此前消息 | 低频 | 蓝色 | `此前消息` / `已接受` / `已拒绝` |
| 巨鲸 | 低频 | 青绿 | `新信号` |

标签不要继续拼成长串，而要拆成单独 badge，例如：

- `金色财经`
- `链上`
- `特色`
- `挂后台`

为了节约侧边栏空间，当前实现不再展示 `问题数`、`严重度` 这类二级统计标签；`此前消息` 和 `巨鲸` 也不展示 `来源`、`币种`、`方向` 等正文或摘要里已经能看到的信息。

## `已直发` 和 `挂后台` 的区分设计

这是新快讯卡片里必须明显区分的地方。

推荐同时使用：

- 状态 badge
- 边框/色条
- 分组文案

建议表现：

- `已直发`
  - 绿色 badge
  - 卡片整体对比度略低
  - 强调“快速浏览”
- `挂后台`
  - 灰蓝或琥珀 badge
  - 卡片边框更明显
  - 强调“这是未直接发布的快讯”

不要再依赖正文里隐含的 `isPublish=false` 语义。

## 超链接呈现

插件里不应该把链接单独起一行当附件。

推荐：

- 标题可点击。
- 正文中的 URL 自动转为文内链接。
- `此前消息` 等低频卡片的标题优先跳转到 Odaily 对应快讯页；只有拿不到 Odaily item id 时才回退到原始链接。
- 高频区标题不作为跳转入口，点击标题或正文只用于展开/收起卡片全文。
- 如果后续能提供结构化链接片段，则优先渲染为正文内链。
- 卡片底部不再展示弱化的 `详情` 入口，避免挤占侧边栏空间。

## 鉴权方案

鉴权要简单，但必须能区分具体编辑，因为要做响应时效和反馈复盘。

推荐方案：

- 使用 `Supabase Auth`
- 采用 `邮箱 + 密码`
- 额外维护一张插件专用白名单表，例如 `editor_plugin_users`

当前实现补充约束：

- 插件包内直接内置 HTTPS 插件轻服务地址，编辑不需要手填连接参数。
- 浏览器不再直接调用 Supabase Auth token 接口，也不直接调用 Supabase RPC。
- 侧边栏每次轮询和提交动作前都会通过插件轻服务检查登录会话；检查失败则回到登录页。

不建议直接复用 `console_admins`，因为插件不是控制台，值班编辑也不应默认获得控制台权限。

推荐字段：

```text
user_id
email
enabled
display_name
created_at
updated_at
```

不建议第一版用：

- 共享账号
- 共享密码
- 无身份访问

因为这样无法区分是谁看的、是谁反馈的。

## 读模型

### 当前实现

当前实现已经完成第一阶段本地化：浏览器侧仍调用插件轻服务，但 `/plugin/feed/items`、`/plugin/feed/state`、`/plugin/feed/mark-seen` 和 `/plugin/feed/feedback` 的请求路径优先读写 Linux 本地 SQLite store，不再同步调用 Supabase RPC。

本地 store 文件：

```text
data/runtime/editor_plugin_local.sqlite
```

当前本地 store 包含：

- 本地信息流卡片缓存。
- 本地 `接受 / 拒绝` 反馈队列。
- 本地插件 session 缓存，用于减少刷新时反复查询远端 `editor_plugin_sessions`。

后台 syncer 仍会低频使用数据库函数 `editor_plugin_feed` 从 Supabase 回填本地 feed，并把本地反馈异步写入 `editor_plugin_submit_feedback`。因此当前实现已经切断“每次插件刷新都触发 Supabase 聚合查询”的压力源，但还没有完成“所有 worker 直接写本地 feed store”的最终目标。

相关环境变量：

```text
EDITOR_PLUGIN_LOCAL_FEED_SYNC_ENABLED=true
EDITOR_PLUGIN_LOCAL_FEED_SYNC_INTERVAL_SECONDS=30
EDITOR_PLUGIN_LOCAL_FEED_MAX_AGE_HOURS=2
EDITOR_PLUGIN_LOCAL_FEED_SYNC_EMAIL=
```

`EDITOR_PLUGIN_LOCAL_FEED_SYNC_EMAIL` 可选。为空时，轻服务会使用最近一次成功登录或刷新请求的编辑邮箱作为后台 syncer 的 RPC 身份。

当前来源：

- `新快讯`
  - 来自当前快讯通知链路对应的任务、发布结果和通知记录
  - `ai_source` 和标记为 `AI信源` 的 X 账号新快讯进入 AI区
- `Crypto信源标题提醒`
  - 来自 `external_media_alert_pipeline`
  - 当前主要承接 `alert_only` Crypto信源站点，例如 `The Block`
- `审核者`
  - 来自审核者结果和提醒记录
- `此前消息`
  - 来自 `writer3_contexts` 和对应通知记录
- `巨鲸`
  - 来自 `whale_watch_activities`
  - 来自 `whale_watch_hyperliquid_activities`

当前信息流只展示最近 `2 小时` 内的内容，避免旧消息长期挤占侧边栏空间，同时降低侧边栏打开稍晚就错过直发或挂后台新快讯的概率；窗口内多条消息通过各自分区内部滚动承载，不改变高频区、AI区、低频区的展示高度。

RPC 当前返回统一字段：

```text
feed_item_id
feed_kind
lane
priority
title
summary
badges[]
status_label
status_tone
occurred_at
source_url
detail_url
action_schema
meta_json
```

当前 `feed_kind`：

```text
newsflash
external_media_alert
auditor_alert
writer3_context
whale_onchain
whale_hyperliquid
```

当前 `lane`：

```text
high
ai
low
```

### 目标实现

长期目标是上游 worker 直接把可展示事件写入 Linux 本地 feed store，插件刷新不实时查询 Supabase，也不依赖后台 syncer 调用 `editor_plugin_feed` 跨多张业务表聚合。

目标边界：

- worker 先把可展示事件写入 Linux 本地 feed store，而不是只写 Supabase 后等待 syncer 回填。
- `/plugin/feed/items` 读取本地 feed store，并继续返回上面的统一字段。
- `/plugin/feed/state` 读取本地反馈状态和本地处理状态。
- `/plugin/feed/feedback` 先写 Linux 本地 feedback queue，再由异步 syncer 写入 Supabase。
- Supabase 作为异步档案库、控制台查询源和历史复盘源，不再作为插件常规刷新链路的实时依赖。

在 worker 直接写本地 feed store 完成前，排障仍要关注后台 syncer 对 `editor_plugin_feed` 和反馈 RPC 的连接占用。但插件刷新请求本身的主要故障面已经变为本地轻服务、本地 SQLite store 和 async syncer。

## 动作语义

### 高频区卡片

高频区卡片默认折叠摘要，点击标题或正文摘要切换展开/收起全文。

展开全文时，高频区自身列表出现或继续使用纵向滚动，不借用 AI区和低频区高度。

为了避免卡片底部被折叠内容挤出可视区域，`审核者` 的 `接受 / 拒绝` 按钮放在卡片第一行右侧，与时间同排展示。

### 审核者卡片

`接受 / 拒绝` 只表示编辑对审核者提醒是否认可。

界面表现：

- 未反馈时不显示 `待反馈` 占位词。
- 有 `接受 / 拒绝` 按钮就表示仍待编辑判断。
- 一旦反馈成功，按钮立即消失，卡片改为显示 `已接受` 或 `已拒绝`，并保留反馈时间。

第一版不要让这个动作直接：

- 改快讯正文
- 修改已发布状态
- 自动撤稿

它首先是反馈留痕动作。

### 此前消息卡片

`接受 / 拒绝` 只表示编辑对“此前消息”质量是否认可。

第一版同样先不直接改主链路状态。

补充交互：

- 默认摘要折叠显示，避免低频区卡片过高。
- 支持 `展开全文 / 收起`；展开只影响低频区自己的纵向滚动，不改变其他分区高度。
- 支持 `复制全文`，便于值班编辑把补充信息带走使用。
- 反馈后同样隐藏 `接受 / 拒绝` 按钮，只保留结果状态和时间。
- 卡片正文按 Telegram 同款结构同时展示 `原文` 和 `此前消息`，方便编辑对照查看和复制。

### 新快讯卡片

新快讯第一版默认只做：

- 查看
- 已读留痕
- 链接跳转

不强制加 `接受 / 拒绝`。

## 声音提醒

可以做。

当前实现支持：

- `Side Panel` 打开状态下
- 有新消息进入当前信息流时
- 按消息业务类型播放不同提示音

当前声音分类固定为五类：

- `挂后台新快讯`
- `直发新快讯`
- `审核者`
- `此前消息`
- `巨鲸`

前端按如下规则把每条新消息归到唯一声音类型：

- `auditor_alert` → `审核者`
- `newsflash + 已直发` → `直发新快讯`
- `newsflash + 挂后台` → `挂后台新快讯`
- `writer3_context` → `此前消息`
- `whale_onchain / whale_hyperliquid` → `巨鲸`

这里的声音类型只决定提醒方式，不影响卡片展示分区。

设置页当前支持：

- `总开关`
- 每类消息单独的 `启用 / 关闭`
- 每类消息单独的 `声音样式`
- 每类消息单独的 `音量`
- 每类消息单独的 `冷却时间（毫秒）`
- 每类消息单独的 `测试这个声音`

当前内置声音样式预设：

- `单短音`
- `双短音`
- `三连音`
- `尖锐短音`
- `尖锐长连音`
- `柔和脉冲`

默认策略：

- `挂后台新快讯`：`单短音`
- `直发新快讯`：`双短音`
- `审核者`：`尖锐长连音`
- `此前消息`：`单短音`
- `巨鲸`：`单短音`
- 侧边栏打开并经过用户交互后，插件会解锁浏览器音频上下文；设置页支持按类别测试实际播放效果

为了避免连续刷屏，当前采用批次合并和分类冷却：

- 同一轮刷新如果有多条新消息，只播放一次
- 播放这一轮新消息中优先级最高的那类声音
- 当前优先级：`审核者 > 直发新快讯 > 挂后台新快讯 > 此前消息 > 巨鲸`
- 每个声音类型各自维护冷却时间；冷却期内同类新消息不重复播放

如果未来希望在 `Side Panel` 没打开时也能发声，需要单独评估更重的后台音频方案；这不放进第一版。

新消息出现后前 `2 分钟`，卡片标题使用红色显示；超过 2 分钟后恢复普通标题颜色。

## 当前接口

浏览器侧当前调用插件轻服务 HTTP 接口：

```text
POST /plugin/auth/login
POST /plugin/auth/logout
POST /plugin/auth/profile
POST /plugin/feed/items
POST /plugin/feed/state
POST /plugin/feed/mark-seen
POST /plugin/feed/feedback
```

服务端后台 syncer 继续复用数据库函数：

```text
editor_plugin_profile()
editor_plugin_feed(p_limit)
editor_plugin_state(p_feed_item_ids)
editor_plugin_mark_seen(p_feed_item_id, p_feed_kind, p_session_id, p_extra_json)
editor_plugin_submit_feedback(p_feed_item_id, p_feed_kind, p_feedback, p_session_id, p_extra_json)
```

这些函数不再位于插件刷新请求的同步路径上。`/plugin/feed/items`、`/plugin/feed/state` 和 `/plugin/feed/feedback` 已优先读写 Linux 本地 store / queue，Supabase RPC 只作为后台回填和异步归档能力，不再作为插件刷新成功的必要条件。

其中：

- `editor_plugin_profile`：检查当前登录账号是否在插件白名单；轻服务 `/plugin/auth/profile` 返回单个 profile 对象，插件前端按对象读取并兼容旧数组返回。
- `editor_plugin_feed`：返回统一信息流卡片。
- `editor_plugin_state`：返回当前用户自己的最近反馈状态；已看字段保留为空值兼容旧返回结构。
- `editor_plugin_mark_seen`：兼容旧插件调用，不再写入已看记录。
- `editor_plugin_submit_feedback`：写入 `接受 / 拒绝` 反馈。

注意：这两个函数都通过插件轻服务代理调用，`editor_plugin_mark_seen` 期望 4 个参数，`editor_plugin_submit_feedback` 期望 5 个参数；服务端包装器会按函数名动态拼接 SQL 占位符，不再使用固定“四参数”模板。

当前没有实现 `editor_plugin_heartbeat`。

## 排序建议

默认优先级建议：

```text
auditor_alert > newsflash > writer3_context > whale_onchain > whale_hyperliquid
```

补充规则：

1. 展示顺序为高频区、AI区、低频区。
2. 高频区必须同时给 `审核者` 和 `新快讯` 留可见席位，不能让单一类型长期挤满整栏。
3. AI区展示 AI 信源新快讯，按优先级和时间排序。
4. 低频区必须同时给 `此前消息` 和 `巨鲸` 留可见席位，不能让单一类型长期挤满整栏。
4. 每个类型内部再按 `occurred_at DESC` 展示。
5. 已反馈卡片保留展示，但动作按钮消失，不再占用“待处理”心智。

## 值班考核口径

至少记录三个时间点：

- `delivered_at`
- `first_seen_at`
- `acted_at`

核心指标：

- `delivered_at -> first_seen_at`
- `first_seen_at -> acted_at`

这样才能区分：

- 看到了但处理慢
- 根本没及时看到

## 与现有模块的关系

### 审核者

- 插件消费审核者问题提醒。
- 插件补充 `接受 / 拒绝` 反馈。
- 审核者不再发送 Telegram，问题只进入信息流插件和数据库留痕。

### 编写者3

- 插件消费“此前消息”结果。
- 插件补充 `接受 / 拒绝` 反馈。
- 正常“此前消息”不再发送 Telegram inline 确认按钮。

### 发布者 / 新快讯

- 插件展示新快讯。
- 对新快讯重点区分 `已直发` 和 `挂后台`。
- 这里的 `挂后台` 只是发布状态，不等于审核者提醒。

### 巨鲸

- 插件只消费现有巨鲸活动结果。
- 不改变巨鲸识别逻辑；正常巨鲸活动不再发送 Telegram。

## 安装与初始化

数据库初始化：

```bash
python backend/src/main.py editor-plugin-init-db
```

`editor-plugin-api-server` 是常驻轻服务，启动时不自动执行插件表、RPC 或 X 相关 schema 初始化。插件 schema、RPC 和依赖表变更后，先显式执行初始化命令，再重启轻服务。

白名单管理：

```bash
python backend/src/main.py editor-plugin-grant-user --email editor@example.com --display-name "夜班编辑"
python backend/src/main.py editor-plugin-list-users
python backend/src/main.py editor-plugin-revoke-user --email editor@example.com
```

Chrome 加载方式：

1. 打开 `chrome://extensions/`
2. 开启开发者模式
3. 选择“加载已解压的扩展程序”
4. 选中仓库中的 `extension/`
5. 打开侧边栏直接登录；当前包内已内置 HTTPS 插件轻服务地址

补充说明：

- 插件白名单是 `editor_plugin_users`
- 控制台白名单是 `console_admins`
- 两者职责不同，不自动互通
