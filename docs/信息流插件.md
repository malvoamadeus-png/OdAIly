# 信息流插件

> 当前仓库已实现 V1 独立 Chrome 插件，代码位于 `extension/`。现阶段生产仍以 Telegram 通知为主，插件是在浏览器侧边栏补充查看、反馈和留痕能力。当前插件规划包含两个一级模块：`信息流` 和 `快讯生成`；本文聚焦已经落地的 `信息流` 模块。

## 定位

信息流插件是给值班编辑使用的独立 Chrome 插件，不属于控制台，也不是控制台的一个页面。

它的目标是承接当前 Telegram 通知渠道里的同一批信息，并在浏览器侧边栏里提供：

- 更强的视觉区分。
- 更顺手的常驻查看。
- 更明确的反馈动作。
- 可统计的查看和反馈留痕。

第一版不是新业务链路，而是现有通知渠道的增强工作面。

## 非目标

第一版不追求：

- 替代控制台。
- 承担控制台配置能力。
- 直接改稿。
- 自动修改审核结果。
- 自动替代 Telegram 全量下线。

## 当前关系

插件和控制台无关，但和当前 Telegram 通知渠道直接相关。

目标上，插件承接的就是现在值班编辑在 Telegram 里看到的这几类信息：

- `新快讯`
- `审核者`
- `此前消息`
- `巨鲸`

也就是说，插件不是再造一套来源，而是消费和当前 Telegram 通知相同或等价的上游结果，并在界面层补上反馈能力。

## 独立代码目录

插件代码必须独立放在仓库根目录单独文件夹，不与 `backend/` 混放，也不依赖把 `backend/` 一起打包给编辑。

当前实现目录：

```text
extension/
  manifest.json
  src/
    background.js
    sidepanel.html
    sidepanel.js
    options.html
    options.js
    styles.css
    lib/
  README-install.md
```

约束：

- 插件可直接以 unpacked extension 方式加载 `extension/` 文件夹。
- 插件运行时代码不要和 `backend/` 目录混在一起。
- 插件工程应尽量自包含，不依赖跨目录引用 `frontend/` 或 `backend/` 的运行时代码。
- 后端只提供数据库、RLS、RPC 或其他远端接口，不承载插件前端源码。

## 形态选择

推荐形态是 `Chrome Extension Manifest V3 + Side Panel`：

- 编辑浏览网页时可以常驻在侧边栏。
- 比 Telegram 和独立网页少一次切换。
- 更适合连续盯消息。
- 可以本地保存分栏高度、筛选和已读状态缓存。

当前不建议第一版做成：

- 独立网页。
- 浏览器 popup。
- 注入网页的悬浮层。

## 一级信息架构

当前插件顶部规划两个一级模块：

- `信息流`
- `快讯生成`

其中：

- `信息流` 负责通知查看、反馈和留痕。
- `快讯生成` 负责从 X 页面抽取正文并调用搜索者 / 编写者能力。

插件其他能力如果未来需要，再继续挂在这两个一级模块之下，不再额外扩成重导航。

当前实现里，这两个一级模块已经共用同一个 side panel 外壳、同一套登录态和同一条顶部导航。

## 主界面布局

`信息流` 页面分为上下两栏：

- 上栏：`高频区`
- 下栏：`低频区`

两栏之间可拖动调整高度，默认建议：

- 高频区：`65%`
- 低频区：`35%`

拖拽结果保存在插件本地存储，按用户恢复。当前实现使用本地设置保存高频区比例。

当前实现优先保证同屏可见条数，卡片、按钮和字体都按侧边栏高密度浏览做了收缩，不走 Telegram 那种大块通知样式。

### 高频区

高频区放最需要值班编辑即时扫视的内容：

- `新快讯`
- `审核者`

### 低频区

低频区放需要持续关注但不一定秒级处理的内容：

- `此前消息`
- `巨鲸`

## 信息类型定义

### 1. 新快讯

来源仍是现有快讯通知链路。

插件里要重点区分两种状态：

- `已直发`
- `挂后台`

这里的 `挂后台` 只是快讯发布状态，不是“审核者发现错误”。

### 2. 审核者

这里专指现有 `审核者` 模块发现疑似文本错误后发出的提醒。

值班编辑需要在插件里对每条审核提醒点击：

- `接受`
- `拒绝`

这两个动作的意义是：

- `接受`：认可这条审核提醒指出的问题成立。
- `拒绝`：认为这条审核提醒不成立，属于误报或不值得处理。

它的用途有两层：

- 统计值班编辑响应速度。
- 方便后续复盘审核者误报。

### 3. 此前消息

这里专指现有 `编写者3` 输出的“此前消息”。

值班编辑同样可以点击：

- `接受`
- `拒绝`

用途是给“此前消息”质量留反馈，而不是改动主快讯发布状态。

### 4. 巨鲸

这里沿用现有巨鲸监控内容，只做查看和定位，不需要第一版就加复杂反馈动作。

## 视觉区分

当前 Telegram 里用“金色财经有新快讯-链上-特色模式”这类连续字符串表达来源和类型，扫描效率不够。

插件里建议把一条信息拆成固定视觉槽位：

1. 左侧色条：区分大类。
2. 第一行：标题或主句。
3. 同行轻量标签：状态、来源、模式等必要信息。
4. 摘要。
5. 底部：时间、动作。

建议映射：

| 类型 | 区域 | 主色 | 状态示例 |
| --- | --- | --- | --- |
| 新快讯 | 高频 | 红橙 | `已直发` / `挂后台` |
| 审核者 | 高频 | 琥珀 | `审核者` / `已接受` / `已拒绝` |
| 此前消息 | 低频 | 蓝色 | `此前消息` / `已接受` / `已拒绝` |
| 巨鲸 | 低频 | 青绿 | `新信号` |

标签不要继续拼成长串，而要拆成单独 badge，例如：

- `金色财经`
- `链上`
- `特色`
- `挂后台`

为了节约侧边栏空间，当前实现不再展示 `问题数`、`严重度` 这类二级统计标签；`此前消息` 和 `巨鲸` 也不展示 `来源`、`币种`、`方向` 等正文或摘要里已经能看到的信息。

## `已直发` 和 `挂后台` 的区分设计

这是新快讯卡片里必须明显区分的地方。

推荐同时使用：

- 状态 badge
- 边框/色条
- 分组文案

建议表现：

- `已直发`
  - 绿色 badge
  - 卡片整体对比度略低
  - 强调“快速浏览”
- `挂后台`
  - 灰蓝或琥珀 badge
  - 卡片边框更明显
  - 强调“这是未直接发布的快讯”

不要再依赖正文里隐含的 `isPublish=false` 语义。

## 超链接呈现

插件里不应该把链接单独起一行当附件。

推荐：

- 标题可点击。
- 正文中的 URL 自动转为文内链接。
- `此前消息` 等低频卡片的标题优先跳转到 Odaily 对应快讯页；只有拿不到 Odaily item id 时才回退到原始链接。
- 高频区标题不作为跳转入口，点击标题或正文只用于展开/收起卡片全文。
- 如果后续能提供结构化链接片段，则优先渲染为正文内链。
- 卡片底部不再展示弱化的 `详情` 入口，避免挤占侧边栏空间。

## 鉴权方案

鉴权要简单，但必须能区分具体编辑，因为要做响应时效和反馈复盘。

推荐方案：

- 使用 `Supabase Auth`
- 采用 `邮箱 + 密码`
- 额外维护一张插件专用白名单表，例如 `editor_plugin_users`

当前实现补充约束：

- 插件包内直接内置 `Supabase URL` 与公开可分发的 `publishable key`，编辑不需要手填连接参数。
- 侧边栏每次轮询和提交动作前都会检查登录会话；接近过期时自动 refresh，refresh 失败则回到登录页。

不建议直接复用 `console_admins`，因为插件不是控制台，值班编辑也不应默认获得控制台权限。

推荐字段：

```text
user_id
email
enabled
display_name
created_at
updated_at
```

不建议第一版用：

- 共享账号
- 共享密码
- 无身份访问

因为这样无法区分是谁看的、是谁反馈的。

## 读模型

当前实现没有单独建统一视图，而是通过 Supabase RPC `editor_plugin_feed` 在数据库侧聚合四类信息。

当前来源：

- `新快讯`
  - 来自当前快讯通知链路对应的任务、发布结果和通知记录
- `审核者`
  - 来自审核者结果和提醒记录
- `此前消息`
  - 来自 `writer3_contexts` 和对应通知记录
- `巨鲸`
  - 来自 `whale_watch_activities`
  - 来自 `whale_watch_hyperliquid_activities`

当前信息流只展示最近 `30 分钟` 内的内容，避免旧消息长期挤占侧边栏空间。

RPC 当前返回统一字段：

```text
feed_item_id
feed_kind
lane
priority
title
summary
badges[]
status_label
status_tone
occurred_at
source_url
detail_url
action_schema
meta_json
```

当前 `feed_kind`：

```text
newsflash
auditor_alert
writer3_context
whale_onchain
whale_hyperliquid
```

当前已实现：


建议字段：

- 必须能追到具体编辑
- 建议 append-only，不覆盖历史


### 3. 在线心跳

## 动作语义

### 高频区卡片

高频区卡片默认折叠摘要，点击标题或正文摘要切换展开/收起全文。

为了避免卡片底部被折叠内容挤出可视区域，`审核者` 的 `接受 / 拒绝` 按钮放在卡片第一行右侧，与时间同排展示。

### 审核者卡片

`接受 / 拒绝` 只表示编辑对审核者提醒是否认可。

界面表现：

- 未反馈时不显示 `待反馈` 占位词。
- 有 `接受 / 拒绝` 按钮就表示仍待编辑判断。
- 一旦反馈成功，按钮立即消失，卡片改为显示 `已接受` 或 `已拒绝`，并保留反馈时间。

第一版不要让这个动作直接：

- 改快讯正文
- 修改已发布状态
- 自动撤稿

它首先是反馈留痕动作。

### 此前消息卡片

`接受 / 拒绝` 只表示编辑对“此前消息”质量是否认可。

第一版同样先不直接改主链路状态。

补充交互：

- 默认摘要折叠显示，避免低频区卡片过高。
- 支持 `展开全文 / 收起`。
- 支持 `复制全文`，便于值班编辑把补充信息带走使用。
- 反馈后同样隐藏 `接受 / 拒绝` 按钮，只保留结果状态和时间。
- 卡片正文按 Telegram 同款结构同时展示 `原文` 和 `此前消息`，方便编辑对照查看和复制。

### 新快讯卡片

新快讯第一版默认只做：

- 查看
- 已读留痕
- 链接跳转

不强制加 `接受 / 拒绝`。

## 声音提醒

可以做。

建议第一版只支持：

- `Side Panel` 打开状态下
- 有新消息进入当前信息流时
- 播放一声短提示音

推荐做成插件设置项：

- `声音提醒`：开 / 关
- `提醒范围`：全部 / 仅高频区
- `音量`：低 / 中 / 高

第一版建议默认策略：

- 默认只对 `高频区` 新消息发声
- `审核者` 与 `新快讯` 优先发声
- `此前消息` 与 `巨鲸` 默认静音，避免过度打扰

为了避免连续刷屏，建议加最小去重窗口，例如：

- 同一批连续到达的新消息，`3-5` 秒内最多响一次

如果未来希望在 `Side Panel` 没打开时也能发声，需要单独评估更重的后台音频方案；这不放进第一版。

## 当前 RPC

当前已实现：

```text
editor_plugin_profile()
editor_plugin_feed(p_limit)
editor_plugin_state(p_feed_item_ids)
editor_plugin_mark_seen(p_feed_item_id, p_feed_kind, p_session_id, p_extra_json)
editor_plugin_submit_feedback(p_feed_item_id, p_feed_kind, p_feedback, p_session_id, p_extra_json)
```

其中：

- `editor_plugin_profile`：检查当前登录账号是否在插件白名单。
- `editor_plugin_feed`：返回统一信息流卡片。
- `editor_plugin_state`：返回当前用户自己的已看和最近反馈状态。
- `editor_plugin_mark_seen`：记录首次查看和后续查看次数。
- `editor_plugin_submit_feedback`：写入 `接受 / 拒绝` 反馈。

当前没有实现 `editor_plugin_heartbeat`。

## 排序建议

默认优先级建议：

```text
auditor_alert > newsflash > writer3_context > whale_onchain > whale_hyperliquid
```

补充规则：

1. 高频区优先于低频区。
2. 高频区必须同时给 `审核者` 和 `新快讯` 留可见席位，不能让单一类型长期挤满整栏。
3. 低频区必须同时给 `此前消息` 和 `巨鲸` 留可见席位，不能让单一类型长期挤满整栏。
4. 每个类型内部再按 `occurred_at DESC` 展示。
5. 已反馈卡片保留展示，但动作按钮消失，不再占用“待处理”心智。

## 值班考核口径

至少记录三个时间点：

- `delivered_at`
- `first_seen_at`
- `acted_at`

核心指标：

- `delivered_at -> first_seen_at`
- `first_seen_at -> acted_at`

这样才能区分：

- 看到了但处理慢
- 根本没及时看到

## 与现有模块的关系

### 审核者

- 插件消费审核者和当前 Telegram 通知同源的提醒。
- 插件补充 `接受 / 拒绝` 反馈。
- 当前生产仍以 Telegram 提醒为准。

### 编写者3

- 插件消费“此前消息”和当前 Telegram 同源的提醒。
- 插件补充 `接受 / 拒绝` 反馈。
- 当前 Telegram inline 确认按钮和本地 SQLite 仍是现状。

### 发布者 / 新快讯

- 插件展示新快讯。
- 对新快讯重点区分 `已直发` 和 `挂后台`。
- 这里的 `挂后台` 只是发布状态，不等于审核者提醒。

### 巨鲸

- 插件只消费现有巨鲸活动结果。
- 不改变巨鲸识别和播报逻辑。

## 安装与初始化

数据库初始化：

```bash
python backend/src/main.py editor-plugin-init-db
```

白名单管理：

```bash
python backend/src/main.py editor-plugin-grant-user --email editor@example.com --display-name "夜班编辑"
python backend/src/main.py editor-plugin-list-users
python backend/src/main.py editor-plugin-revoke-user --email editor@example.com
```

Chrome 加载方式：

1. 打开 `chrome://extensions/`
2. 开启开发者模式
3. 选择“加载已解压的扩展程序”
4. 选中仓库中的 `extension/`
5. 打开侧边栏直接登录；当前包内已内置 `Supabase URL` 与公开可分发的 `publishable key`

补充说明：

- 插件白名单是 `editor_plugin_users`
- 控制台白名单是 `console_admins`
- 两者职责不同，不自动互通
