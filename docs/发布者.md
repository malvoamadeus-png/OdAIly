# 发布者

## 职责

发布者对应 `x_processing` 的 `publish` 阶段，负责：

- 读取编写者2产出的最终定稿
- 按总开关、主任务来源、渠道开关和北京时间窗决定是否进入自动发布判断
- 在需要时调用模型输出固定结构化分类
- 对每条稿件只调用一次推送接口
- 记录发布者决策、原因码、接口结果和决策时间
- 在推送调用完成后发送统一编写链路的内部 Telegram 通知

发布者不负责：

- 改写稿件内容
- 二次补发历史 `ready_review` 积压稿
- 前端可编辑 Prompt
- 把 `isPush` 打开

## 链路位置

X 主链路为：

```text
judge_crypto/judge_ai -> search -> write -> format_publish -> publish
```

竞品、外媒全文和 AI信源全文是 search-first 链路：

```text
search -> judge_crypto/judge_ai -> write -> format_publish -> publish
```

其中：

- `format_publish` 只做定稿，成功后把任务推进到 `publisher_pending`
- `publish` 在本地流水线同一进程内接手 `publisher_pending`，进入 `publishing`
- 结束后进入 `auto_published`、`ready_review` 或 `publisher_failed`

## V1 决策顺序

发布者按以下顺序判断：

1. 是否存在最终定稿；没有则直接失败
2. 发布者总开关是否开启
3. 主任务来源是否映射到发布渠道
4. 该渠道是否启用
5. 当前北京时间是否在允许时间窗内
6. 只有前面都通过时，才调用模型输出固定分类
7. 分类命中允许名单则自动发布，否则挂后台

发布者只判断“这条刚走完链路的稿件现在发不发”，不会等待未来窗口，也不会在窗口打开时回头补发旧稿。

## 渠道口径

### 业务渠道

- `external_media`：外媒全文；AI信源暂时也复用该发布渠道
- `x`：X
- `competitor`：竞品
- `jin10`：金十

### 主任务来源映射

- `tasks.source = 'non_mainstream_media'` -> `external_media`
- `tasks.source = 'ai_source'` -> `external_media`
- `tasks.source = 'x'` -> `x`
- `tasks.source in ('blockbeats', 'panews', 'jinse')` -> `competitor`
- `tasks.source = 'jin10'` -> `jin10`

`mainstream_media` 不映射到发布者渠道，只保留历史兼容；遇到这类历史任务时会按 `source_not_eligible` 挂后台。`ai_source` 虽然复用 `external_media` 发布渠道，但通知和展示路线名显示为“AI信源”。

### 第一版实验口径

- 默认只启用 `external_media`
- `x`、`competitor` 与 `jin10` 渠道存在配置项，但默认关闭
- 用户确认的实验样本按“主任务来源是 `non_mainstream_media`”计算

## 时间窗

- 唯一时区基准：`Asia/Shanghai`
- 第一版默认窗口：北京时间 `00:01` 到 `07:30`
- 时间窗只判断“发布者执行当下”的北京时间
- 不在窗口内时，不调用模型，直接走挂后台分支
- 支持跨日窗口表达

## 分类与模型

### 固定分类 key

- `policy_regulation`
- `people_view`
- `major_project_progress`
- `funding`
- `other`

自动发布允许前四类；`other` 一律挂后台。

### 输出契约

模型必须输出固定 JSON：

```json
{
  "category": "policy_regulation|people_view|major_project_progress|funding|other",
  "reason": "简短中文说明"
}
```

- `category` 决定是否自动发布
- `reason` 只用于记录和排障
- 第一版不开放前端 Prompt 编辑
- 如果未来要新增、删除或改名分类，必须改代码并同步更新前端只读展示和文档

### 运行配置

- 默认模型：`gpt-5.5`
- 默认 reasoning effort：`low`
- 环境变量：
  - `X_PROCESS_PUBLISHER_MODEL`
  - `X_PROCESS_PUBLISHER_REASONING_EFFORT`

## 推送接口调用规则

发布者对每条稿件只调用一次推送接口。

### 自动发布通过

条件同时满足：

- 总开关开启
- 来源映射到已启用渠道
- 当前北京时间在时间窗内
- 分类命中允许名单

调用参数：

```json
{
  "isPublish": true,
  "isPush": false
}
```

### 挂后台

出现以下任一情况时：

- 总开关关闭
- 来源不在实验口径或未映射渠道
- 渠道关闭
- 不在时间窗
- 分类结果为 `other`

调用参数：

```json
{
  "isPublish": false,
  "isPush": false
}
```

`isPush` 永远固定为 `false`。

## 记录字段

发布者在 `x_task_pipeline` 记录：

- `publisher_channel`
- `publisher_model`
- `publisher_category`
- `publisher_decision`
- `publisher_reason_code`
- `publisher_output`
- `publisher_decided_at`
- 复用已有 `push_result`
- 复用已有 `telegram_result`

当前 `publisher_decision` 取值：

- `auto_publish`
- `manual_review`
- `failed`

当前 `publisher_reason_code` 包括：

- `publisher_disabled`
- `source_not_eligible`
- `channel_disabled`
- `outside_publish_window`
- `category_allowed`
- `category_other`
- `model_failed`
- `push_failed`
- `format_missing`

## 控制台配置

发布者控制台使用两张表：

- `publisher_settings`
  - `enabled`
  - `timezone`
  - `window_start_local`
  - `window_end_local`
- `publisher_channels`
  - `channel_key`
  - `display_name`
  - `enabled`

分类不进控制台配置，只读展示当前代码契约。

## 状态流转

```text
written
-> formatting
-> publisher_pending
-> publishing
-> auto_published | ready_review | publisher_failed
```

- 自动发布成功：`auto_published`
- 挂后台成功：`ready_review`
- 模型失败、推送失败或缺少最终定稿：`publisher_failed`

## 与其他模块的边界

- 编写者2：只定稿，不再调用推送接口
- 提醒者：只负责外媒标题提醒，不负责统一编写链路内部通知
- 监督者：需监控 `local_pipeline` 心跳和 `publisher_failed` 等最终失败状态；`publisher_pending / publishing` 保留为观察状态，不再表示独立发布 worker 队列
- 审核者：只检查已发布快讯文本质量，不参与是否发布判断
