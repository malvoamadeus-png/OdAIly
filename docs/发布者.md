# 发布者

## 职责

发布者对应 `x_processing` 的 `publish` 阶段，负责：

- 读取编写者2产出的最终定稿
- 按主任务来源和发布者规则决定是否自动发布
- 在需要时调用模型输出 `pass / reject` 和面向编辑的判断理由
- 对每条稿件只调用一次推送接口
- 记录发布者决策、原因码、接口结果和决策时间
- 在推送调用完成后记录统一编写链路结果；正常新快讯不再发送 Telegram，值班侧通过信息流插件查看

发布者不负责：

- 改写稿件内容
- 二次补发历史 `ready_review` 积压稿
- 把 `isPush` 打开
- 运行时轮询 Supabase 配置

## 链路位置

X 主链路为：

```text
judge_crypto/judge_ai -> search -> write -> format_publish -> publish
```

竞品、Crypto信源全文和 AI信源全文是 search-first 链路：

```text
search -> judge_crypto/judge_ai -> write -> format_publish -> publish
```

其中：

- `format_publish` 只做定稿，成功后把任务推进到 `publisher_pending`
- `publish` 在本地流水线同一进程内接手 `publisher_pending`，进入 `publishing`
- 结束后进入 `auto_published`、`ready_review` 或 `publisher_failed`

## 决策顺序

发布者按以下顺序判断：

1. 是否存在最终定稿；没有则直接失败。
2. 主任务来源是否映射到发布渠道；未映射则挂后台。
3. 根据任务来源和账号级 AI信源标记选择发布者规则块。
4. 规则块是否启用；未启用则挂后台。
5. 规则块是否存在启用的通过规则；没有则挂后台。
6. 调用模型按规则输出 `pass / reject` 和一小段中文业务理由。
7. `pass` 自动发布，`reject` 挂后台。

发布者只判断“这条刚走完链路的稿件现在发不发”，不会等待未来窗口，也不会在窗口打开时回头补发旧稿。

## 来源与规则块

### 业务渠道

`publisher_channel` 继续作为历史字段和通知辅助字段保留：

- `external_media`：Crypto信源全文；AI信源也复用该渠道字段
- `x`：X
- `competitor`：竞品
- `jin10`：金十

### 主任务来源映射

- `tasks.source = 'non_mainstream_media'` -> `external_media`
- `tasks.source = 'ai_source'` -> `external_media`
- `tasks.source = 'x'` -> `x`，其中账号标记为 `AI信源` 的任务仍保留该渠道字段
- `tasks.source in ('blockbeats', 'panews', 'jinse')` -> `competitor`
- `tasks.source = 'jin10'` -> `jin10`

`mainstream_media` 不映射到发布者渠道，只保留历史兼容；遇到这类历史任务时会按 `source_not_eligible` 挂后台。

### 发布者规则块

- `tasks.source = 'ai_source'` 使用 `AI信源` 规则块。
- `tasks.source = 'x'` 且 `tasks.metadata.x_account_is_ai_source = true` 使用 `AI信源` 规则块。
- 其他已映射来源使用 `常规` 规则块。
- 当前 `AI信源` 规则块默认关闭，进入发布者时按 `publisher_profile_disabled` 挂后台。

## 规则与模型

规则保存在服务器本地 `data/config/publisher_rules.json`，由控制台通过服务器接口维护。

当代码内置默认规则版本升级时，服务端加载旧版 `publisher_rules.json` 或 Supabase 快照会自动合并新增默认规则，并更新已有默认规则的名称、说明和案例；自定义规则会保留，已有规则的启用 / 停用状态会保留。

规则块包含：

- 块级启用 / 停用
- `通过规则`
- `排除规则`

规则字段：

- `规则名`
- `规则说明`
- `案例列表`
- `启用 / 停用`

判定原则：

- 排除规则优先；命中任意启用排除规则时必须 `reject`。
- 未命中排除规则、且命中任意启用通过规则时才允许 `pass`。
- 无法明确归入任意启用通过规则时默认 `reject`。
- 案例只作为模型判断参考。

模型输出契约：

```json
{
  "decision": "pass|reject",
  "reason": "一小段中文业务理由"
}
```

`reason` 用于控制台“入库及背后信息”展示，必须说明这条快讯命中了哪些正文事实、对应哪类通过或排除规则、为什么应直发或挂后台；不要只复述“模型判断通过/拒绝”或内部原因码。

不再输出分类名；`publisher_category` 在新规则发布者下写入 `NULL`，仅保留历史兼容。

### 默认常规规则

默认通过规则：

- `链上大额 / 巨鲸 / 清算`
- `政策 / 监管 / 法案`，只放行 Crypto、稳定币、交易所、DeFi、数字资产等直接相关内容
- `产品发布 / 升级 / 上线`，只放行 Crypto 相关产品、功能、协议、交易服务、钱包、链上应用
- `预测市场 / 事件合约`，近期作为独立放行主题，覆盖预测市场、事件合约、金融/政治预测交易相关监管、诉讼、平台动作、机构限制、交易规则或市场结构变化
- `宏观数据 / 利率 / 央行`，只放行利率决议、FOMC、CPI、PCE、非农、失业率、核心通胀等关键宏观节点，或央行官员对利率路径的明确政策信号
- `机构观点 / 人物表态`，只放行“高重要性主体 + 直接 Crypto/监管/市场结构相关 + 含具体新信息/新动作/新数据”的观点
- `Crypto 公司重大股价 / 资本市场变动`，放行 Crypto 相关上市公司或准上市公司在明确公司动作、业绩/资产/产能/融资事实支撑下的显著股价或资本市场变化
- `融资`
- `安全 / 事故 / 停运`
- `项目 / 组织重大动向`

其中：

- `链上大额 / 巨鲸 / 清算` 不要求正文必须给出完整 0x 地址；团队关联实体、项目方钱包、基金会相关地址、做市商相关钱包等可识别关系主体，只要有明确卖出、转入、增减持、清算、解锁后出售等资金动作事实，可按链上资金事件判断。
- “拉高出货”“操纵”等定性判断不能单独作为直发依据；发布者应以正文中的可报道资金动作、金额、持仓变化、安全事故或其他事实为依据。
- `安全 / 事故 / 停运` 中的金额型安全损失一般需达到 10 万美元及以上；低于 10 万美元时，只有链停摆、核心协议级漏洞、持续攻击或广泛用户资产风险等重大非金额影响才可放行。
- “高重要性主体”优先按正文可直接判断的系统影响力理解，例如国家级监管、央行、立法机构、头部交易基础设施、头部稳定币/ETF/托管/核心公链基金会等。
- 发布者不应凭外部常识脑补主体重要性；如果正文没有明确证据证明主体重要，默认按不重要处理。
- “关键组织的关键人物”必须拆开判断：只有正式政策动作、监管动作、产品/市场结构变化、明确影响利率或资产负债表的政策信号，才可按重要事件理解；组建工作组、研究框架、内部审查、专家名单、方法论讨论默认挂后台。
- 高管离职不按公司名自动放行：创始人、CEO、实际控制人、主席、核心基金会负责人变动可进入重大动向判断；CFO、COO、销售/分销/业务负责人等离任默认挂后台，除非正文明确证明影响核心产品、上市进程、监管资格、托管/清算安全或广泛用户资产。
- 纯态度表达、吹捧、自我宣传、泛政治评论、宏大叙事、无新增事实的观点，即使发言人知名，也不自动发布。
- 预测市场规则服从排除规则优先；涉及中国领土、主权、安全红线时仍必须挂后台。

默认排除规则：

- `自我宣传 / 站台型表达`
- `地缘政治`
- `市场行情 / 价格异动`
- `非 Crypto 的宏大政治评论`
- `泛宏观 / 财政金融总量`
- `链上指标归纳 / 市场状态解读`
- `多主题市场综评 / 拼盘解读`
- `小额安全损失`
- `TRON / 孙宇晨相关`
- `中国领土 / 主权 / 安全红线`
- `营销活动 / 社区活动`
- `代币分发 / 空投 / 激励`
- `交易所网络升级充值提现暂停`
- `数字人民币`
- `非 Crypto 传统制造业供应链价格调整`
- `战略合作`
- `中小主体例行公司新闻`
- `小众传统金融产品异动`
- `冷门论文 / 估值模型 / 研究发布`

其中：

- `中小主体例行公司新闻` 覆盖普通牌照 / 审批获批、一般合作、常规扩区、普通产品更新、例行组织调整、常规上新等 routine 公司新闻。
- 对牌照、审批、合作、组织动向这类公司新闻，若主体不是高重要性主体，且正文没有明确写出市场地位、用户规模、资产规模、监管层级、覆盖范围或系统性影响，默认挂后台。
- `泛宏观 / 财政金融总量` 覆盖财政赤字、财政收支、政府预算、国债发行余额、货币市场基金资产规模、一般银行准备金、一般流动性规模等宏大背景数据；即使金额巨大或创历史新高，也默认挂后台。
- `链上指标归纳 / 市场状态解读` 覆盖长期持有者实现盈亏、持币年龄分布、链上估值模型、活跃地址趋势等观察性统计；除非对应具体主体资金动作、清算、安全事故或协议级风险事件，否则挂后台。
- `多主题市场综评 / 拼盘解读` 覆盖没有单一核心新闻事实、把币价涨跌、技术指标、宏观数据、法案进展、美股/个股波动、市场评论等多个主题拼在一起的综评类文本；即使其中夹带可关注元素，只要正文主线发散、不能围绕一个明确事件改写为快讯，也默认挂后台。
- `小额安全损失` 覆盖金额型安全事件中损失低于 10 万美元、且没有链停摆、核心协议级漏洞、持续攻击或广泛用户资产风险等重大非金额影响的内容，默认挂后台。
- `TRON / 孙宇晨相关` 覆盖 TRON、TRX、波场、火币、HTX、孙宇晨、Justin Sun、Sun Yuchen 等相关主体或生态新闻，默认挂后台人工看。
- `中国领土 / 主权 / 安全红线` 覆盖中国领土主权、国家安全、外交红线、制裁冲突、台海、香港、新疆、西藏、南海、中美军事/外交摩擦等敏感政治议题。
- `营销活动 / 社区活动` 覆盖交易大赛、品牌营销、社区任务、促销、积分活动等拉新或宣传型内容。
- `代币分发 / 空投 / 激励` 覆盖空投、代币领取、奖励发放、生态激励、赠款计划、积分兑换、用户补贴等代币或权益分发信息。
- `交易所网络升级充值提现暂停` 覆盖交易所因支持公链或网络升级、钱包维护、节点维护而临时暂停或恢复充值、提现的信息，例如币安暂停 Moonbeam 网络充值提现。
- `数字人民币` 覆盖数字人民币、e-CNY、数币钱包、数币试点、数币支付场景等信息，默认不自动发布。
- `非 Crypto 传统制造业供应链价格调整` 只约束常规 Crypto 发布者：与 Crypto 无直接关系的传统制造业零部件、材料、供应链、渠道价格调整默认挂后台。AI/半导体核心产业链若未来要自动发布，应由 `AI信源` 发布者或独立开关承接；当前 `AI信源` 发布者关闭时不会调用模型，直接挂后台。
- `小众传统金融产品异动` 指与 Crypto 没有直接关系的冷门 ETF、单一基金、个别衍生品、非核心交易工具的异常流入流出、规模变化或观察性点评。
- `冷门论文 / 估值模型 / 研究发布` 指小众论文发表、估值模型争论、研究发布、方法论阐释等低普适性内容；除非正文明确证明其已带来监管落地、产品上线、市场结构变化或被核心机构正式采用，否则挂后台。

- 默认 reasoning effort：`low`

  - `X_PROCESS_PUBLISHER_MODEL`

## Telegram 边界

发布者不再为正常 `新快讯`、`X`、`CoinDesk / Cointelegraph / Decrypt`、竞品或金十结果发送 Telegram。代码仍会回写 `telegram_result`，但该结果固定为 `skipped=true`，用于说明业务通知已转交信息流插件。

Telegram 只保留给监督者的系统级错误、积压、心跳、宕机和程序异常告警。

## 旧时间窗

旧表 `publisher_settings` 中的北京时间窗字段仍保留在数据库里，但发布者运行时不再读取它们。自动发布是否开启只看 `常规` / `AI信源` 两个规则块级开关。

## 推送接口调用规则

发布者对每条稿件只调用一次推送接口。

### 自动发布通过

条件同时满足：

- 来源映射到发布者渠道
- 对应发布者规则块启用
- 模型按规则输出 `pass`

调用参数：

```json
{
  "isPublish": true,
  "isPush": false
}
```

### 挂后台

出现以下任一情况时：

- 来源未映射
- 规则块关闭
- 没有启用的通过规则
- 模型输出 `reject`

调用参数：

```json
{
  "isPublish": false,
  "isPush": false
}
```

`isPush` 永远固定为 `false`。

来源链接透出约定：

- `jinse` 仅在上游收集到外部原文链接时，挂后台或自动发布才透出 `sourceUrl`
- 如果 `jinse` 上游只有站内 `jinse` 页面，则不透出 `sourceUrl`
- `blockbeats` 仅在上游收集到外部原文链接时，挂后台或自动发布才透出 `sourceUrl`
- 如果历史任务或上游只提供 `theblockbeats.info` / `blockbeats.info` 站内页，则不透出 `sourceUrl`
- `panews`、`jin10` 仍不透出 `sourceUrl`

## 记录字段

发布者在 `x_task_pipeline` 记录：

- `judge_completed_at`
- `search_completed_at`
- `write_completed_at`
- `format_completed_at`
- `publisher_channel`
- `publisher_model`
- `publisher_category`
- `publisher_decision`
- `publisher_reason_code`
- `publisher_output`
- `publisher_decided_at`
- `publish_completed_at`
- 复用已有 `push_result`
- 复用已有 `telegram_result`

其中：

- `publisher_decided_at` 表示发布者完成“发不发 / 挂后台”决策的时间。
- `publish_completed_at` 表示发布阶段整段完成、推送结果和通知结果都已落库的时间。
- `publisher_output.reason` 保存发布者给编辑看的业务判断理由；未调用模型的保守挂后台分支也会写入确定性理由。
- 复盘发布阶段耗时时，应优先使用 `format_completed_at -> publisher_decided_at -> publish_completed_at` 这组三段时间，而不是只看 `tasks.updated_at`。

当前 `publisher_decision` 取值：

- `auto_publish`
- `manual_review`
- `failed`

当前 `publisher_reason_code` 包括：

- `source_not_eligible`
- `publisher_profile_disabled`
- `publisher_no_enabled_allow_rules`
- `rule_allowed`
- `rule_rejected`
- `model_failed`
- `push_failed`
- `format_missing`

## 控制台配置

发布者控制台通过服务器接口维护规则：

- `POST /console/publisher-rules/get`
- `POST /console/publisher-rules/save`

保存链路：

```text
前端 -> 服务器保存接口 -> 服务器本地配置生效 -> Supabase 当前快照写一次
```

数据库快照表：

- `publisher_rule_config`
  - `singleton_key`
  - `config_json`
  - `prompt_text`
  - `updated_by`
  - `updated_at`

`publisher_rule_config` 只保存当前版本，不保存历史。worker 读取本地 JSON，不轮询 Supabase。

## 状态流转

```text
written
-> formatting
-> publisher_pending
-> publishing
-> auto_published | ready_review | publisher_failed
```

- 自动发布成功：`auto_published`
- 挂后台成功：`ready_review`
- 模型失败、推送失败或缺少最终定稿：`publisher_failed`

## 与其他模块的边界

- 编写者2：只定稿，不再调用推送接口
- 提醒者：只负责 Crypto信源标题提醒，不负责统一编写链路内部通知
- 监督者：需监控 `local_pipeline` 心跳和 `publisher_failed` 等最终失败状态；`publisher_pending / publishing` 保留为观察状态，不再表示独立发布 worker 队列
- 审核者：只检查已发布快讯文本质量，不参与是否发布判断
